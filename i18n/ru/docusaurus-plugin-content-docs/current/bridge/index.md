привязкой к домену и эпохе, воспроизведённые побайтово в контракте EVM, программе Solana и L1 (кросс-языковые
векторы с известным ответом). Развёртывание на Arbitrum и аудит перед мейннетом — в плане.

MetaFlux переводит **все активы — включая USDC — через MetaBridge**: кастодиальный бридж с
подписью валидаторов MetaFlux (аналог HL-Bridge2). **Сторонние бриджи и зависимость от Circle CCTP**
на критическом пути отсутствуют.

## Почему кастодия, а не CCTP

CCTP переводит USDC только между теми сетями, которые Circle зарегистрировала в качестве CCTP *доменов*. MetaFlux —
независимый L1; включение в список CCTP-доменов является бизнес-решением Circle, которое не находится в нашей власти.
Путь для депозитов, требующий одобрения третьей стороны, не может служить надёжной основой, поэтому MetaFlux
управляет собственным кастодиальным бриджем на той же **модели доверия к набору валидаторов, что и сама сеть** —
без внешнего комитета, сети хранителей или привратника.

## Модель

**Контракт бриджа в исходной сети** (первой является Base) хранит депонированные
токены. Валидаторы MetaFlux отслеживают депозиты и кредитуют L1; выводы средств высвобождаются
контрактом через набор совместных подписей ⅔ взвешенных по доле валидаторов за пределами
окна оспаривания.

### Депозит (исходная сеть → MetaFlux)

```
Base:
  1. user.approve(USDC, bridge)
  2. bridge.deposit(mtfDest, amount)        // USDC pulled into custody
  3. bridge emits Deposit{user, mtfDest, amount, nonce, …}

MetaFlux:
  4. each validator observes the Deposit event and submits an mbAttest
     (an Inbound MetaBridgeMsg partial co-signature) — validator authority,
     NEVER the public /exchange path
  5. on ⅔ stake-weighted quorum the L1 credits the user's USDC cross-collateral
     (the same system-credit primitive the faucet uses); each deposit credits
     EXACTLY ONCE (idempotent by message id)
```

Событие `Deposit` побайтово совместимо с детерминированным `message_id` L1:
`keccak256(chain ‖ direction ‖ user ‖ asset ‖ amount ‖ dst ‖ nonce)`.

### Вывод (MetaFlux → исходная сеть)

```
MetaFlux:
  1. user submits a withdraw action (Outbound MetaBridgeMsg)
  2. validators co-sign it to ⅔ quorum; the L1 retains the signature set in
     meta_bridge.mb_outbox + finalized_cosignatures

Base (two-phase: request → claim):
  3. each validator's RELAY LOOP polls the committed L1 state and submits a
     batchWithdraw(...) tx — signed with the validator's OWN key, gas paid by the
     validator's EVM address (no separate relayer key). The contract recovers
     each entry's signers, sums HOT-set stake, requires ≥⅔, and QUEUES it into
     the dispute window. A bad/raced entry in the batch is skipped (FailedWithdrawal
     event), not reverted.
  4. after BOTH the dispute window (seconds) AND a minimum block count elapse,
     claim(id) / batchClaim(ids) releases USDC to the user. Any single validator
     can dispute(id) a queued withdrawal, or the COLD ⅔-quorum can
     invalidateWithdrawal(id), as an emergency revoke during the window.
```

## Модель безопасности

- **Полномочия** — мультиподпись валидаторов MetaFlux ⅔, взвешенная по доле (secp256k1, те же
  ключи, что обеспечивают консенсус; кворум `6700` bps). Мультиподпись валидаторов и окно оспаривания
  вывода средств являются несущими элементами: компрометация ключей бриджа = потеря средств, поэтому
  контракты проходят проверку на уровне консенсуса/подписи и аудит перед мейннетом.
- **Защита от повторного воспроизведения** — каждый `message_id` применяется однократно, привязан к экономической
  идентичности chain/source-nonce, так что кредитование происходит ровно один раз даже при ротации набора
  валидаторов; контроль реализован как на L1, так и в контракте (`withdrawalSeen` / постоянный маркер трат на Solana).
  Подписи привязаны к домену и эпохе, поэтому косайнатуру нельзя воспроизвести в другом развёртывании,
  сети или эпохе набора валидаторов.
- **Управление и ротация** — нет аккаунта администратора; каждая привилегированная операция требует
  совместной подписи валидаторов. Ротация набора валидаторов двухфазная (запрос → финализация за окном
  оспаривания); в этот период любой **единственный** горячий валидатор может выполнить `pause`
  (с ограничением времени ожидания на валидатора) или полностью **cancel** (отменить) ожидающую ротацию,
  что не позволяет скомпрометированному управляющему кворуму незаметно подменить набор. На Solana тот же
  набор предоставляет `pause`/`dispute` от одного валидатора и экстренную поверхность `unpause` /
  `invalidateWithdrawal` с кворумом.
- **Вне `/exchange`** — кредиты по депозитам поступают через системный путь валидаторов и структурно
  недоступны из публичного пользовательского пути `/exchange`; учёт ведётся только по активному
  набору валидаторов.
- **Оговорка о кастодии** — USDC на MetaFlux является бриджевым требованием, обеспеченным балансом
  исходного контракта, а не Circle-каноническим на MetaFlux (аналогично модели HL).

## Методы контракта

### Base — `MetaBridgeUSDC` (EVM)

| Метод | Авторизация | Назначение |
|-------|-------------|------------|
| `deposit(mtfDest, amount)` | любой (депозитор) | Перевести USDC в кастодию, эмитировать событие `Deposit` для аттестации валидаторами |
| `withdraw(...)` / `batchWithdraw(reqs)` | любой, ретранслирующий набор косайнатур **HOT ⅔** | Проверить кворум и поставить вывод(ы) в очередь окна оспаривания |
| `claim(mid)` / `batchClaim(mids)` | любой | Высвободить созревший USDC после двойного окна по времени и блокам (не приостанавливается) |
| `dispute(mid)` | любой единственный **HOT** валидатор | Отменить поставленный в очередь вывод внутри его окна оспаривания |
| `cancelValidatorSetUpdate()` | любой единственный **HOT** валидатор | Наложить вето на ожидающую ротацию набора валидаторов внутри её окна |
| `pause()` | любой единственный **HOT** валидатор | Заморозить новые депозиты и постановку выводов в очередь (с ограничением времени ожидания на валидатора) |
| `unpause(...)` | **COLD ⅔** | Снять паузу |
| `invalidateWithdrawal(mid, ...)` | **COLD ⅔** | Отозвать поставленный в очередь, не заявленный мошеннический вывод |
| `requestValidatorSetUpdate(p, newEpoch, ...)` | **COLD ⅔** | Подать двухфазную ротацию набора горячих+холодных валидаторов |
| `finalizeValidatorSetUpdate()` | любой (без разрешения) | Применить поданную ротацию после её окна оспаривания |
| `setDisputeWindow(...)` / `setMinDisputeBlocks(...)` | **COLD ⅔** | Настроить окно оспаривания (с ограниченными min/max) |
| `computeMessageId(...)` / `computeGovDigest(...)` | view | Воспроизвести точные байты, которые подписывает валидатор |
| Геттеры `hot*/cold*` | view | Доля, участники, количество, итого, кворум bps / необходимо для валидаторов |

Все вызовы с совместной подписью принимают `(uint8[] sigV, bytes32[] sigR, bytes32[] sigS)` в порядке возрастания подписанта, с низким S, `v ∈ {27,28}`.

### Solana — `metabridge-solana`

| Инструкция | Авторизация | Назначение |
|------------|-------------|------------|
| `initialize(params)` | деплойер (однократно) | Закрепить монетный двор USDC, набор валидаторов, кворум, двойное окно оспаривания |
| `deposit(mtf_dest, amount)` | депозитор (подписант) | Перевести SPL USDC в кастодию, эмитировать `DepositEvent` |
| `withdraw(mid, user, amount, dst, nonce, cosigs)` | любой, ретранслирующий набор косайнатур **⅔** | Проверить кворум и создать PDA `PendingWithdrawal` (+ постоянный маркер трат) |
| `claim(message_id)` | любой | Высвободить кастодированный USDC после двойного окна по времени и слотам |
| `dispute(mid, cosig)` | любой единственный валидатор (1 косайнатура) | Отменить поставленный в очередь вывод внутри его окна |
| `pause(cosig)` | любой единственный валидатор (1 косайнатура) | Заморозить депозит / вывод / финализацию ротации |
| `unpause(gov_nonce, cosigs)` | косайнатура **⅔** | Снять паузу |
| `invalidate_withdrawal(mid, gov_nonce, cosigs)` | косайнатура **⅔** | Отозвать поставленный в очередь вывод |
| `request_validator_set_update(...)` | косайнатура **⅔** | Подать ротацию набора валидаторов |
| `finalize_validator_set_update()` | любой (без разрешения) | Применить поданную ротацию после её окна |

Solana использует ОДИН набор валидаторов (без разделения на горячие/холодные) и не имеет точек входа `setDisputeWindow` / batch; recovery-id — это raw secp256k1 `{0,1}` (vs EVM `{27,28}`). Обе сети отклоняют подписи с высоким S и привязывают id программы/контракта + эпоху к каждому совместно подписанному дайджесту.

## Дорожная карта

- Реальный наблюдатель за депозитами Base и ретранслятор выводов (детерминированное ядро L1 и
  контракт Base готовы; внецепочечные наблюдатели подключены и используют метку блока `finalized`
  цепи для защиты от реорганизаций).
- Развёртывание в нескольких сетях: программа кастодиального бриджа Solana работает в devnet по
  той же модели; следующим будет Arbitrum.
- Аудит безопасности перед любым развёртыванием с реальными средствами (мейннет).
- Кросс-чейн компоузабилити (вызов контрактов других сетей из MTF) — V2.
