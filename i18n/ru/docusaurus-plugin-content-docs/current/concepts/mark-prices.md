# Расчётные цены

:::tip
**Стабильно.**
:::

## Коротко о главном

**Расчётная цена** (mark price) — это авторитетная цена каждого актива в протоколе, используемая для расчёта маржи, ликвидации, финансирования и исполнения триггерных ордеров. Она вычисляется как медиана трёх компонентов — оракульного якоря (плюс EMA базиса), срединной цены внутреннего стакана и внешней медианы цен перп-рынков — и пересчитывается непрерывно. Это НЕ цена последней сделки.

## Почему расчётная цена ≠ цена последней сделки

Использование цены последней сделки для расчёта маржи уязвимо к манипуляциям: небольшая торговля по искусственно завышенной или заниженной цене может принудительно ликвидировать других пользователей. Расчётная цена — это сглаженная, многоисточниковая композиция, которую крайне сложно сдвинуть.

## Состав

> Реализованная расчётная цена — это **медиана 3 компонентов** (описана ниже, `mark_source: "MedianOfOraclesAndMid"`), а не плоская `median(mid, oracle, ema_mid)`.

### Алгоритм вычисления

```
mark = median( присутствующие компоненты из {C1, C2, C3} )
        # 3 присутствующих → median3   2 присутствующих → среднее   1 → само значение
        # одиночный C2 (без внешнего якоря) не даёт обновления — см. ниже
```

Три компонента:

| Компонент | Определение |
|-----------|------------|
| **C1** (оракульный якорь) | `C1 = oracle + EMA(quote_mid − oracle)` — оракул плюс EMA с фиксированным затуханием **базиса** перп-контракта (≈ **150 с** период полураспада при частоте пересчёта) |
| **C2** (внутренний стакан) | `mid(best_bid, best_ask)` — только лучший бид/аск. Требует обеих сторон стакана и отсутствия кросса, иначе `None`. Намеренно **исключает цену последней сделки** (включение последней сделки сделало бы пересчёт самореференциальным и могло бы заморозить тонкий односторонний стакан). |
| **C3** (внешние перпы) | `median(внешние перп-медианы)` по **5 перп-биржам** (Binance, OKX, Bybit, Gate, MEXC); требует **≥ 2** доступных площадок, иначе `None` |

Внешняя медиана устойчива к одному выбросу. **Отсутствующие компоненты просто исключаются** — при двух доступных расчётная цена равна их среднему, при одном — это само значение. Если внутренний стакан пуст и внешние перпы недоступны, `mark = C1 = oracle + EMA(basis)`, что позволяет корректно деградировать к споровому оракулу, не замораживая систему.

**Одиночный C2 отклоняется.** Если единственным доступным компонентом является срединная цена внутреннего стакана (нет ни оракула, ни внешних перпов), расчётная цена остаётся неизменной — чтобы не позволить единственному спреду в книге заявок, контролируемому злоумышленником, определять цену ликвидации/финансирования. Одиночный C1 (внешний оракул) или одиночный C3 (уже медиана ≥ 2 площадок) допустимы.

- **EMA компонента C1** — это детерминированная `DeterministicEma` (фиксированная точка `(num, denom)`, фиксированное затухание `0.9548` ≈ период полураспада 150 с при частоте пересчёта ~10 с). Она использует `quote_mid − oracle` только при наличии двустороннего стакана и оракула, поэтому пустой или односторонний стакан не вносит шума. Рынок, на котором никогда не было пригодной котировки, сохраняет `EMA = 0`, то есть `C1 = голый оракул`.
- EMA вычисляется **для каждого актива** (по одной на каждый перп-рынок).
- Вычисленная медиана записывается в **авторитетную расчётную цену** рынка (значение, которое читают все потребители), поэтому обычная сделка, фиксирующая цену последней торговли, не может её перекрыть — медиана является авторитетной для активных рынков, а не только для спокойных.

> ⚠️ **Поправка к предыдущему тексту.** В более ранней документации C2 моделировался как `median(bid, ask, last_trade)`, а C3 — как взвешенная медиана по 7 площадкам. Реальные формы таковы: C2 = **срединная цена двустороннего стакана** (только лучший бид/аск, без последней сделки), C3 = **медиана 5 внешних перп-площадок** (требуется ≥ 2), а C1 = оракул плюс EMA базиса перпа с фиксированным затуханием (`0.9548`). Отсутствующие компоненты выпадают из медианы (без замены на `unwrap_or(C1)`), а одиночный C2 не даёт обновления.

### Два ценовых плана (прочитайте, прежде чем разбирать числа)

В MTF цены представлены на **двух различных числовых планах** — это основной источник путаницы с масштабом:

| План | Тип | Масштаб | Используется в |
|-------|------|-------|---------|
| **План стакана / ордеров / расчётной цены** | `FixedPrice` (`i128`) / `price_e8` | **Фиксированная точка 1e8** (целое число = цена × 10⁸) | стакан заявок, `last_mark_px`, прекомпиляции EVM (`mark_px_e8`, `entry_px_e8`), уровень `px` в `l2_book`, `limit_px` ордера, `tick_size` |
| **План оракула / номинала / обеспечения** | `rust_decimal::Decimal` | **Целые USDC** (1 единица = 1 USDC) | агрегатор оракула + вычислитель расчётной цены (C1/C2/C3 — все в `Decimal`), `oracle_px` финансирования, движок сценариев PM, маржа/здоровье счёта, человекочитаемые поля `mark_px`/`oracle_px` в `market_info`/`markets` |

Вычислитель расчётной цены и агрегация оракула работают полностью в **плане `Decimal` (целые USDC)**. Результат конвертируется в **план `FixedPrice` с масштабом 1e8** при записи в стакан / `last_mark_px`. Однако человекочитаемые ответы `market_info` / `markets` возвращают `mark_px` и `oracle_px` уже в **плане целых USDC** (например, `"67042.335"`, а не сырое 1e8) — только поля *подачи* ордеров и стакана (`l2_book` px, `limit_px`, `tick_size`) остаются в 1e8. *Расчёты* PnL/финансирования используют третье минорное соглашение — USDC `1e6` (`accumulated_funding_e6` в `mark_settle`). Всегда проверяйте, в каком плане находится формула, прежде чем сравнивать величины.

## Оракул (якорь C1)

`oracle` — якорь C1 — это **взвешенная медиана** до 10 внешних спотовых площадок (Binance 3, OKX 2, Bybit 2, Coinbase 2, Bitget / Kraken / KuCoin / Gate / MEXC / MetaFlux-spot по 1; итого 15), публикуемая один раз за блок и подписанная валидаторами оракула. Веса для каждого символа задаются через управление (`SetOracleWeights`, `ActionId 148`); фид, устаревший более чем на 60 с или отклонившийся более чем на 5 % от кросс-биржевой медианы, отбрасывается; если доступно менее 50 % веса, слот сохраняет последнее корректное значение.

**Компонент C3** расчётной цены — это **отдельный набор фидов** — срединные цены перпов с **5 перп-площадок** (Binance, OKX, Bybit, Gate, MEXC), а не таблица спотового оракула. Полный состав, правила надёжности, переопределения для отдельных символов и чтение `oracle_sources` описаны в **[Цены оракула](./oracle-prices.md)**.

Чтение [`market_info`](../api/rest/info.md#market_info) возвращает `mark_source` (дескриптор `"MedianOfOraclesAndMid"`), а также составные `mark_px` / `oracle_px`; отдельные компоненты C1/C2/C3 и взвешенный список источников не выводятся как поля протокола.

## Расчётная цена vs оракул — почему они расходятся

Отклонение расчётной цены от оракула — **нормальное и ожидаемое явление**. Оракул отслеживает **спот**; расчётная цена отслеживает, где торгуется **перп** — а перп может иметь устойчивый **базис** (премию или дисконт) по отношению к споту:

- **C2** — это срединная цена перп-стакана MetaFlux, **C3** — внешняя медиана цен *перп-рынков*. Оба отражают перп, а не спот.
- **C1** — это `oracle + EMA(quote_mid − oracle)`: EMA базиса притягивает даже оракульный якорь к текущей премии перпа, поэтому все три компонента отслеживают перп.

Когда перп торгуется, например, с дисконтом 30 % к споровому индексу, `mark ≈ perp` и `oracle ≈ spot` обоснованно расходятся примерно на 30 %. Именно этот разрыв и должно закрывать **[финансирование](./funding-rates.md)**. Заметим, что если оракул ненадёжен, финансирование *отключается и затухает до 0* даже при большом разрыве mark/oracle (то есть большой разрыв при ~0 финансировании означает, что оракул для данного рынка не вызывает доверия, а не что финансирование сломано). См. [отключение финансирования](./funding-rates.md#gating-when-the-oracle-is-untrusted).

## Защитные полосы

> **Статус реализации:** медиана трёх компонентов (с исключением отсутствующих и отклонением одиночного C2) реализована в настоящее время. Ограничение по блокам, описанное в этом разделе (`clamp(candidate, prior ± max_step)` поверх медианы), является **проектной спецификацией, но не реализовано как отдельный зажим в вычислителе расчётной цены** — структурная медиана сейчас является основной защитой от манипуляций. Ближайший к нему работающий механизм копирует оракул в `last_mark_px` устаревшего стакана без нарастания. Числа ниже отражают задуманный дизайн полосы; перед использованием точных значений зажима проверьте их в актуальной системе.

Даже при составной медиане злоумышленник может одновременно сдвинуть два из трёх источников. Поэтому в расчётной цене предусмотрено применение **полос** на каждый блок:

```
prior_mark   = mark at block T-1
band_pct     = market parameter, default 0.5% per second
max_step     = prior_mark * band_pct * (block_time_ms / 1000)

candidate    = median(...)
mark_T       = clamp(candidate, prior_mark - max_step, prior_mark + max_step)
```

| По умолчанию | Значение |
|---------|-------|
| `band_pct` | 0,5% в секунду |
| `block_time_ms` | 100 мс |
| `max_step_per_block` | ~0,05% |

При шаге 0,05% за 100 мс движение на 5% займёт ≈10 секунд. Реальные быстрые движения нагоняются за небольшое число блоков; искусственные скачки ограничиваются коротким нарастанием.

Если кандидатная цена выходит за пределы полосы в течение `band_violation_blocks` блоков подряд (по умолчанию 50 = ~5 с), протокол считает, что произошёл настоящий режимный сдвиг, и на одно окно расширяет полосу в 2 раза. Это предотвращает постоянную заморозку при реальных рыночных дислокациях.

## Где используется расчётная цена

| Потребитель | Зачем нужна расчётная цена? |
|----------|-----------|
| Маржа (начальная и поддерживающая) | Стабильный базис; устойчивость к манипуляциям |
| Оценка уровня для ликвидации | То же |
| Финансирование (vs оракул) | Требуется формулой финансирования |
| Триггерные ордера (StopLoss / TakeProfit) | Устойчивость к скачкам от единичных сделок |
| Отображение PnL (нереализованная) | Стабильное значение для пользователя |
| Страховой учёт | Стабильное значение |

Потребители, использующие **цену последней сделки** вместо расчётной:
- Цены исполнения в стакане (сделки заключаются по реальным ценам стакана)
- Комиссии мейкера/тейкера (вычисляются от цены сделки, не от расчётной)
- Реализованный PnL (вычисляется по цене сделки закрытия)

## Запрос данных

```bash
curl -X POST https://devnet-gateway.mtf.exchange/info \
  -H 'content-type: application/json' \
  -d '{"type":"market_info","asset_id":0}'
```

Чтение [`market_info`](../api/rest/info.md#market_info) возвращает `mark_px` и
`oracle_px` в **плане целых USDC** (например, `"67042.335"`), а также
дескриптор `mark_source`:

```json
{
  "type": "market_info",
  "data": {
    "asset_id":    0,
    "mark_source": "MedianOfOraclesAndMid",
    "mark_px":     "67042.335",
    "oracle_px":   "67042.335"
  }
}
```

Три внутренних компонента `C1`/`C2`/`C3` (оракул+EMA якорь / медиана внутреннего стакана / взвешенная медиана внешних перпов) и состояние полосы (`Ok` / `Banded` /
`Frozen`) хранятся внутри вычислителя расчётной цены; они
**не** выводятся как отдельные поля протокола `market_info` — публикуется только составной
`mark_px`. `Banded` означает, что полоса ограничила кандидата в данном блоке;
`Frozen` означает, что все источники недоступны и протокол удерживает предыдущую расчётную цену.

Выделенный WS-канал `mark` включён в [план развития WS](../api/ws/subscriptions.md#roadmap--not-yet-available) (стриминг ещё не реализован); пока используйте опрос [`market_info`](../api/rest/info.md#market_info) для получения `mark_px`.

## Граничные случаи

<details>
<summary>Показать граничные случаи</summary>

- **Реальное движение на 5% за 1 с.** Полоса ограничивает первые 10 блоков; расчётная цена нагоняет ~0,05% за блок, затем обнаружение режимного сдвига расширяет полосу и нагон ускоряется. Полная задержка составляет ~1–2 секунды — значительная, но ограниченная.
- **Отказ оракула: > 50 % веса отсутствует.** Когда в тике доступно менее 50 % спотового веса, слот оракула **не обновляется** — сохраняется последнее корректное значение. C1 продолжает использовать последний корректный оракул плюс его EMA, поэтому расчётная цена остаётся привязанной.
- **Пустой / односторонний стакан MTF.** C2 выпадает. Расчётная цена = `midpoint(C1, C3)` (или только `C1`, если перпы тоже недоступны) — отслеживает оракульный якорь в сочетании с внешними перпами. Финансирование продолжает использовать доступный оракул; ликвидации проводятся по расчётной цене, привязанной к оракулу.
- **Все внешние перп-площадки недоступны.** C3 выпадает. Расчётная цена = `midpoint(C1, C2)` — срединная цена внутреннего стакана vs оракульный якорь.
- **Нет стакана и нет перпов.** Остаётся только C1 ⇒ `mark = C1 = oracle + EMA(basis)`. Чистая привязка к оракулу.
- **Только односторонний/пустой стакан без оракула.** Одиночный C2 отклоняется ⇒ нет обновления; расчётная цена удерживает последнее значение до появления внешнего якоря (оракула или перпов).
- **Устаревший EMA компонента C1.** По конструкции EMA всегда определена после того, как хотя бы один внутренний mid был учтён; она удерживает последнее значение, пока не приходит новый mid (EMA обновляется только при наличии C2).
- **Триггерные ордера при заморозке.** Исполнение триггеров использует расчётную цену; при заморозке ни один триггер не срабатывает. Ордера в ожидании остаются до возобновления реальной расчётной цены.

</details>

## Последовательность — защитная полоса срабатывает на скачке

Медиана трёх компонентов уже нейтрализует скачок по одному источнику до срабатывания полосы:

```
block T-1   C2(book mid)=100.0  C1(oracle+EMA)=100.0  C3(perps)=100.0  →  median3 = 100.0

block T:    thin MTF book; adversary lifts the internal mid to 110.0
            C2 = 110.0,  C1 = 100.05,  C3 = 100.05   (external perps + oracle unmoved)
            mark = median3(100.05, 110.0, 100.05) = 100.05
            ↑ the adversarial C2 is the OUTLIER → median discards it; mark ≈ oracle anchor

block T+1:  adversary persists at 110.0; oracle/perps drift up slowly
            C2 = 110.0,  C1 = 100.10,  C3 = 100.10
            mark = median3(100.10, 110.0, 100.10) = 100.10

... mark tracks the (C1, C3) consensus; a single manipulated source never wins the median
```

Защита носит структурный характер: чтобы сдвинуть медиану, злоумышленник должен сдвинуть как минимум **два** из трёх компонентов, а C1 (оракул) + C3 (медиана 5 внешних перп-площадок) — именно те, которые сложно сдвинуть. Опциональная защитная полоса на каждый блок, описанная ниже, является дополнительным ограничением поверх неё.

## FAQ

<details>
<summary>Показать FAQ</summary>

**В: Почему не использовать оракул напрямую?**
О: Чистая привязка к оракулу даёт операторам оракула возможность ликвидировать стакан путём манипуляции с фидом. Медиана трёх источников диверсифицирует поверхность доверия.

**В: Можно ли посмотреть историческое поведение полосы?**
О: История расчётной цены с состоянием полосы будет доступна после запуска WS-канала `mark` (в плане развития) и через архивные ответы индексатора; в текущем читаемом интерфейсе этого ещё нет.

**В: Не приведёт ли полоса к несправедливым ликвидациям?**
О: Полоса замедляет изменение расчётной цены относительно базового актива — поэтому во время реального обвала ваш поддерживающий уровень будет оставаться здоровым примерно на 1 секунду дольше, чем на площадке, использующей цену последней сделки. Обратное тоже верно (расчётная цена медленнее восстанавливается). Суммарный эффект: поведение при ликвидации более детерминировано и сложнее поддаётся манипуляциям.

</details>
