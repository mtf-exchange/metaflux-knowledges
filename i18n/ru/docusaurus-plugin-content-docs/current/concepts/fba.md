# Пакетные аукционы с частым проведением (FBA)

:::info
**Предварительная версия.** Подключение по рынкам через [MIP-3](../mip/mip-3.md); FBA работает не на всех рынках.
:::

## Кратко

FBA заменяет непрерывное сопоставление заявок дискретными аукционными пакетами с интервалом `batch_interval_ms`. Заявки, поступившие в рамках одного пакета, исполняются одновременно по единой расчётной цене. Это устраняет MEV, основанный на задержке: выигрыш от опережения на одну микросекунду отсутствует.

## Непрерывный режим и пакетный

| Свойство | Непрерывный CLOB | FBA |
|----------|-----------------|-----|
| Периодичность сопоставления | При каждом поступлении заявки | Каждые `batch_interval_ms` |
| Ценообразование | По каждой сделке | По каждому пакету (одна расчётная цена) |
| Ценность задержки | Высокая (кто первым пришёл — побеждает при равной цене) | Нулевая внутри пакета |
| Выигрыш от задержки | Достаётся HFT | Возвращается участникам через единую цену |
| Видимость заявок | До торгов (книга заявок) | До пакета (видимая очередь) |

## Механизм

```
пакет t:        принимаем заявки в интервале [t, t + batch_interval_ms)
закрытие t:     замораживаем очередь
                вычисляем расчётную цену p*:
                  p* = цена, при которой |совокупный спрос покупателей| = |совокупный спрос продавцов|
                исполняем все пересекающиеся заявки по p*
                незакрытые заявки переносятся в пакет t+1 (или отменяются согласно TIF)
пакет t+1:      открыт
```

Правила исполнения:
- Все заявки на покупку с ценой ≥ p* исполняются по p*.
- Все заявки на продажу с ценой ≤ p* исполняются по p*.
- Единая расчётная цена p* максимизирует общий исполненный объём (эквивалентно пересечению кривых спроса и предложения).

Цена исполнения **единая** для всех участников пакета — никто не получает худшую цену из-за более позднего поступления заявки.

## Когда применять FBA

| Класс активов | По умолчанию | Причина |
|-------------|---------|-----|
| Крупные бессрочные контракты (BTC, ETH) | Непрерывный CLOB | Ликвидный рынок; выигрыш от задержки невелик относительно спреда |
| Низколиквидные инструменты (MIP-3) | FBA (опционально) | Тонкая книга; токсичность HFT превышает пользу от маркет-мейкинга |
| Спотовые пары | Непрерывный CLOB | Отраслевая норма |
| Индексы / структурированные продукты | FBA | Составное ценообразование требует синхронного клиринга |

Режим сопоставления каждого рынка указан в [`market_info.fba_enabled`](../api/rest/info/perpetuals.md#market_info). Рынки с включённым FBA принимают как `FbaOrder` (с указанием целевого пакета), так и [`submit_order`](../api/rest/exchange.md#submit_order) (обрабатывается как заявка FBA в следующий пакет). См. [каталог действий `/exchange`](../api/rest/exchange.md#action-catalog) — `FbaOrder` сегодня является зарегистрированным, но не отображаемым заглушечным типом.

## Интервал пакета

По умолчанию: 1 секунда (10 блоков при времени блока 100 мс). Устанавливается через управление по каждому рынку в `market_info.fba_batch_interval_ms`. Типичный диапазон: 100 мс – 5 с.

Более короткие интервалы уменьшают время ожидания, но увеличивают вычислительную нагрузку. Значение по умолчанию в 1 секунду балансирует между нейтрализацией HFT и удобством для пользователей.

## Структура заявки

```json
{
  "type": "FbaOrder",
  "params": {
    "asset":     42,
    "side":      "Buy",
    "px":  "10050000000",
    "size":   "100000000",
    "batch_id":  9876,
    "cloid":     "0x..."
  }
}
```

`batch_id` определяет, в какой пакет попадает заявка. Текущий идентификатор пакета доступен в [`market_info`](../api/rest/info/perpetuals.md#market_info) в поле `fba_current_batch_id`. Заявки с `batch_id < current` отклоняются (`{"error":"batch already closed"}`); заявки с `batch_id` > current ставятся в очередь для будущего пакета.

Если `batch_id` не указан, заявка направляется в следующий пакет — сервер самостоятельно выбирает тот, который в данный момент принимает заявки.

## Практический пример

В пакете t содержатся следующие заявки на актив 42:

```
покупки:
  bob:    5 @ 100.10
  alice:  3 @ 100.05
  carol:  2 @ 100.00

продажи:
  dave:   3 @ 99.95
  eve:    4 @ 100.00
  frank:  2 @ 100.05
```

Обходим спрос (накопленный объём при каждой цене ≥ кандидата):

```
покупатели — накопленный объём при цене ≥ p:
  100.10:  5
  100.05:  5+3 = 8
  100.00:  8+2 = 10
  99.95:   10  (нет новых)
```

Обходим предложение (накопленный объём при цене ≤ кандидата):

```
продавцы — накопленный объём при цене ≤ p:
  99.95:   3
  100.00:  3+4 = 7
  100.05:  7+2 = 9
  100.10:  9   (нет новых)
```

Пересечение: при p = 100.00 накопленный спрос = 10, накопленное предложение = 7. При p = 100.05 накопленный спрос = 8, предложение = 9. Пересечение находится между 100.00 и 100.05.

Правило клиринга максимизирует объём:

| p | min(покупки, продажи) |
|---|----------------|
| 99.95  | min(10, 3) = 3 |
| 100.00 | min(10, 7) = 7 |
| 100.05 | min(8, 9)  = 8 |
| 100.10 | min(5, 9)  = 5 |

Максимальный исполненный объём равен 8 при `p* = 100.05`. Итого:

- Все покупки ≥ 100.05 исполняются: bob (5) + alice (3) = 8 BTC куплено по 100.05.
- Все продажи ≤ 100.05 исполняются: dave (3) + eve (4) + frank (2) = 9 BTC предложено. Пропорциональное распределение: 8/9 = 88.9% от каждого → dave 2.67, eve 3.56, frank 1.78.
- Carol (покупка 2 @ 100.00) не исполняется — переносится в t+1 или отменяется согласно TIF.

Все победители исполняются по 100.05. Bob не получает худшую цену за то, что пришёл «раньше» — в FBA понятия «раньше» не существует.

## Справедливость клиринга

Когда предложение превышает спрос при p*, бо́льшая сторона заполняется **пропорционально** — каждый продавец получает одинаковую долю исполнения. Никакого FIFO, никакого ценового приоритета среди стороны с избытком (все уже находятся на уровне p* или лучше).

Это и есть свойство справедливости FBA: по расчётной цене ни один участник не получает более выгодных условий, чем другой.

## Граничные случаи

<details>
<summary>Показать граничные случаи</summary>

- **Пустой пакет.** Нет заявок → клиринговое событие не происходит. Следующий пакет открывается немедленно.
- **Односторонний пакет.** Только покупки (или только продажи). Клиринг не происходит — все заявки переносятся в следующий пакет (Gtc) или отменяются (Ioc).
- **Ничья по расчётной цене.** Если две цены одновременно максимизируют объём, протокол выбирает ту, что ближе к предыдущей расчётной цене (снижает неоднозначность шага маркировки).
- **Рыночные заявки в FBA.** Подаются как IOC по экстремальной цене; участвуют в пакете и исполняются по p*, если пересекаются.
- **Reduce-only в FBA.** Проверяется при закрытии пакета с учётом состояния после исполнения предыдущих заявок в рамках того же пакета. Клирингуется атомарно.

</details>

## Последовательность событий

```
t=0.0s   batch_id = 9876 открыт
t=0.2s   bob:    FbaOrder покупка 5 @ 100.10, пакет 9876
t=0.4s   alice:  FbaOrder покупка 3 @ 100.05, пакет 9876
t=0.5s   carol:  FbaOrder покупка 2 @ 100.00, пакет 9876
t=0.6s   dave:   FbaOrder продажа 3 @ 99.95, пакет 9876
t=0.7s   eve:    FbaOrder продажа 4 @ 100.00, пакет 9876
t=0.8s   frank:  FbaOrder продажа 2 @ 100.05, пакет 9876
t=1.0s   batch_id 9876 закрыт; запускается клиринг
         p* = 100.05; исполняется 8 BTC
         сделки публикуются в WS `trades` с batch_id и "kind":"fba"
t=1.0s   batch_id = 9877 открыт
```

## Запросы

Текущий пул FBA и индикативный клиринг доступны по пути чтения `/info` узла
через [`fba_batch_state`](../api/rest/info.md#fba_batch_state) — полная структура ответа и таблица полей приведены там. Принимает `market_id` (u32). FBA подключается по каждому рынку отдельно, поэтому незарегистрированный рынок возвращает **не 404** — он вернёт 200 с нулевыми полями (`enabled:false`, пустой `orders`, `indicative:null`).

```bash
curl -X POST https://api.devnet.mtf.exchange/info \
  -H 'content-type: application/json' \
  -d '{"type":"fba_batch_state","market_id":42}'
```

```json
{
  "type": "fba_batch_state",
  "data": {
    "market_id":      42,
    "enabled":        true,
    "period_ms":      1000,
    "min_lot":        "1",
    "last_settle_ms": 1735689600000,
    "next_settle_ms": 1735689601000,
    "order_count":    11,
    "bid_count":      5,
    "ask_count":      6,
    "bid_size":       "1000000000",
    "ask_size":       "900000000",
    "orders":         [ /* {oid, owner, side, price, size, stp_group, submitted_at_ms} */ ],
    "indicative":     { "clearing_px": "10050000000", "matched_size": "800000000" }
  }
}
```

Цены и объёмы представлены в виде строк целых чисел с **фиксированной точкой 1e8** (плоскость книги/заявок). `next_settle_ms` **вычисляется** как `last_settle_ms + period_ms`. Блок `indicative` содержит единую цену, максимизирующую объём, и соответствующий исполненный объём, который *мог бы* сложиться на **следующем** пакете при текущем окне — вычисляется только для чтения, ещё не урегулирован — и равен `null`, когда пересечения нет (односторонняя или пустая очередь). Это то, каким был бы p\*, если бы пакет закрылся прямо сейчас; полезно трейдерам при принятии решения о добавлении заявок в пакет.

## Часто задаваемые вопросы

<details>
<summary>Показать FAQ</summary>

**В: Разве FBA не ухудшает ценообразование?**
О: Нет — внутри пакета протокол по-прежнему определяет `p*` на основе заявок участников. Ценообразование происходит с фиксированной периодичностью, а не непрерывно.

**В: Почему пакет 1 с, а не 100 мс?**
О: 100 мс слишком мало, чтобы в каком-либо значимом смысле нейтрализовать задержку — даже в рамках 100 мс более быстрые машины успевают повторно подать заявки. 1 с даёт достаточный буфер, при котором задержка физической сети доминирует над внутрипакетной задержкой, устраняя преимущество HFT.

**В: Могут ли рынки FBA сосуществовать с рынками CLOB?**
О: Да — каждый рынок является либо FBA, либо CLOB независимо от других. Аккаунт может одновременно держать позиции на обоих типах рынков.

**В: Снижает ли FBA затраты на газ / вычисления при сопоставлении?**
О: Приблизительно. Непрерывное сопоставление выполняет O(1) операций на каждое поступление; FBA выполняет O(N log N) при закрытии пакета. Для N заявок в пакете FBA сопоставим по затратам, с преимуществом более предсказуемой стоимости на блок.

</details>
