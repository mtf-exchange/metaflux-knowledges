# Режим хеджирования (двусторонние позиции)

:::info
**Доступно.** Переключатель подключения, явная маршрутизация ордеров по стороне, **независимая маржа на каждое плечо** и **двойная отчётность по позициям** реализованы: счёт может переключиться в режим хеджирования (при отсутствии открытых позиций), направить каждый ордер на конкретное плечо через `position_side`, при этом каждое плечо размещает собственную маржу и отображается как отдельный объект позиции. **Ликвидация по плечам** реализована частично: при принудительном закрытии счёта оба плеча сканируются и оцениваются, а детерминированный порядок закрытия (сначала плечо с наибольшими требованиями к поддерживающей марже) рассчитывается одинаково на всех валидаторах — само исполнение закрытия по плечам относительно книги ордеров ещё внедряется. Режим по умолчанию и рекомендуемый — односторонний (единая чистая позиция на рынок).
:::

## Кратко

По умолчанию счёт держит **одну чистую позицию на рынок** (односторонний режим): покупка при короткой позиции уменьшает, а затем переворачивает ту же позицию. **Режим хеджирования** позволяет держать **длинное и короткое плечо на одном рынке одновременно**, учитывая их раздельно.

Режим хеджирования **подключается на уровне счёта** без **порога по балансу** — его может включить любой счёт. Переключение возможно только при **отсутствии позиций на всех рынках**.

## Односторонний режим vs. хеджирование

| | Односторонний (по умолчанию) | Хеджирование |
|---|---|---|
| Позиций на рынок | 1 чистая (со знаком) | до 2 (длинное плечо + короткое плечо) |
| «Покупка при короткой позиции» | уменьшает, затем переворачивает чистую позицию | уменьшает только **короткое** плечо (либо открывает/увеличивает **длинное** — по выбору) |
| Выбор стороны ордера | определяется из buy/sell | требуется **явный** `position_side` (`long` / `short`) |
| Маржа | единое чистое требование | каждое плечо маржируется независимо |
| Ликвидация | одна цена ликвидации | каждое плечо оценивается по собственной поддерживающей марже; детерминированный порядок закрытия реализован, исполнение закрытия по плечам внедряется |
| Отчётность | один объект чистой позиции | один объект на каждое ненулевое плечо (каждый помечен `position_side`) |

Переключатель, маршрутизация по стороне, независимая маржа на плечо и двойная отчётность по позициям — доступны; выбор плеча при ликвидации реализован, исполнение закрытия по плечам ещё внедряется (см. статусное примечание выше).

## Включение

Смена режима позиций — подписанное действие; оно допустимо только при **отсутствии позиций на всех рынках**.

```json
// enable hedge mode (only legal when flat on ALL markets)
{ "type": "set_position_mode", "params": { "hedge": true } }
```

```json
// back to one-way (also only when flat)
{ "type": "set_position_mode", "params": { "hedge": false } }
```

`hedge` — булево значение: `true` = режим хеджирования (двусторонний), `false` = односторонний (по умолчанию).
Предусловие **отсутствия позиций на всех рынках** — мера безопасности: переключение при наличии открытой позиции отклоняется (без изменений состояния), чтобы чистая позиция никогда не была молчаливо переинтерпретирована как осиротевшее плечо. Установка того же значения режима при отсутствии позиций — успешная холостая операция.

Подробности запроса/ответа — в описании [`set_position_mode`](../api/rest/exchange.md#set_position_mode) в справочнике `/exchange`.

## Размещение ордеров в режиме хеджирования

В режиме хеджирования каждый ордер **должен содержать явный `position_side`** (`long` / `short`). Счёт в одностороннем режиме **не должен** передавать `position_side`; счёт в режиме хеджирования — обязан. Поле находится в теле ордера `submit_order` рядом с `side`, `reduce_only` и т. д.

```json
{
  "type": "submit_order",
  "order": {
    "owner": "0x...aa", "market": 0, "side": "bid",
    "kind": "limit", "size": 100000000, "limit_px": 5000000000,
    "tif": "gtc", "stp_mode": "cancel_oldest",
    "reduce_only": false,
    "position_side": "long"
  }
}
```

| Намерение | `side` | `position_side` | `reduce_only` |
|---|---|---|---|
| Открыть / увеличить длинную | `bid` | `long` | false |
| Уменьшить / закрыть длинную | `ask` | `long` | true |
| Открыть / увеличить короткую | `ask` | `short` | false |
| Уменьшить / закрыть короткую | `bid` | `short` | true |

`position_side` указывается **явно, без автовывода** — иначе покупка, предназначенная для *сокращения короткой позиции*, может ошибочно открыть или увеличить длинную. `reduce_only` проверяется **только относительно указанного плеча**: ордер `reduce_only` на `short`-плечо никогда не затронет `long`-плечо.

В режиме хеджирования **«переворот» невозможен**. Закрытие длинного плеча никогда не открывает короткое — это отдельный ордер против короткого плеча.

## Маржа

Каждое плечо маржируется **независимо** — длинное и короткое плечо каждое размещают собственную начальную и поддерживающую маржу, суммируемую в требование по счёту:

```
required_margin = init_margin(long_leg) + init_margin(short_leg)
```

Это намеренно консервативный подход: длинная + короткая позиция на одном рынке дельта-нейтральна по цене, но каждое плечо всё равно блокирует маржу. (Будущее обновление может предусмотреть кредит неттинга для взаимозачётных плеч в рамках [портфельной маржи](./portfolio-margin.md); до тех пор держите оба плеча только при наличии подлинно раздельных экспозиций — например, разных цен входа, которыми вы намерены управлять независимо.)

Каждое плечо сохраняет собственный [режим маржи](./margin-modes.md) — например, длинное плечо можно держать в изолированном режиме, а короткое — в кросс-режиме.

## Ликвидация

При принудительном закрытии счёта оба плеча сканируются и оцениваются по собственному вкладу в поддерживающую маржу, а детерминированный порядок закрытия — **сначала плечо с наибольшими требованиями к поддерживающей марже** (при равенстве — длинное перед коротким), одинаковый на всех валидаторах — рассчитывается по стандартной лестнице [многоуровневой ликвидации](./tiered-liquidation.md). Само исполнение закрытия по плечам относительно книги ордеров ещё внедряется; когда оно будет завершено, ликвидация одного плеча не затронет другое.

## Отчётность

При включённом режиме хеджирования состояние счёта и запросы позиций через `/info` возвращают **один объект позиции на каждое ненулевое плечо** для рынка с обоими плечами, каждый помечен своим `position_side` (`"long"` / `"short"`). Счёт в одностороннем режиме возвращает единственную *чистую* позицию **без** поля `position_side`, точно как сейчас. Открытый интерес на уровне рынка остаётся единой чистой цифрой. См. [`account_state` positions](../api/rest/info.md#account_state).

## FAQ

<details>
<summary>Показать FAQ</summary>

**В: Обязательно ли использовать режим хеджирования?**
О: Нет. Односторонний (чистый) режим — по умолчанию и ведёт себя точно как прежде; режим хеджирования исключительно добровольный.

**В: Есть ли минимальный баланс для его включения?**
О: Нет порога — любой счёт может переключиться в режим хеджирования при отсутствии позиций.

**В: Почему нельзя переключаться при открытой позиции?**
О: Чтобы существующая чистая позиция не превратилась в неоднозначное, осиротевшее плечо. Закройте позиции, затем переключайтесь.

**В: Длинная + короткая позиции одинакового размера требуют вдвое больше маржи?**
О: Да — плечи маржируются независимо, поэтому взаимозачётные длинная+короткая блокируют маржу обоих плеч. Кредит неттинга для взаимозачётных плеч — это будущее улучшение (в рамках [портфельной маржи](./portfolio-margin.md)).

</details>
