# Авто-делеверидж (ADL)

:::info
**Предварительный просмотр.** T4 срабатывает только тогда, когда страховой пул не может полностью покрыть дефицит T3 — редкая ситуация при штатной работе, но алгоритм спроектирован так, чтобы быть детерминированным в момент срабатывания.
:::

## Кратко

Когда страховой пул не может поглотить остаточный убыток от ликвидации, протокол взыскивает средства с прибыльных контрагентов по тому же инструменту — пропорционально их нереализованному PnL. Механизм распределения MetaFlux использует ранжирование на основе онлайн-обучения, цель которого — минимизировать **избыточный хэркат** (то есть хэркат сверх фактически необходимого для покрытия дефицита).

## Когда срабатывает ADL

[Многоуровневая лестница](./tiered-liquidation.md):

```
T0 жёлтая карточка  →  T1 частичная  →  T2 полная  →  T3 бэкстоп  →  T4 ADL
```

T3 передаёт позицию ликвидируемого счёта мультисиг-бэкстопу и производит выплату из страхового пула. Если `insurance_pool < shortfall`, протокол переходит к T4: оставшийся непокрытый убыток распределяется в виде хэрката позиции среди прибыльных контрагентов.

```
shortfall  =  liquidation_loss - insurance_pool_drawn
if shortfall > 0:
    fire_adl(asset, shortfall)
```

Скорость списания из страхового пула ограничена — см. раздел [многоуровневая ликвидация](./tiered-liquidation.md#t3-backstop--netting-at-mark).

## Как вычисляется ADL

> **Дополнительная литература:** «Autodeleveraging as Online Learning» (arXiv:2602.15182).

MTF **не использует** единую оценку ранжирования. ADL разбивается на две независимые подзадачи: одномерное решение по **степени** хэрката (насколько урезать в текущем раунде), обучаемое в онлайн-режиме, и детерминированное **пропорциональное распределение** (кто платит — на основе ёмкости PnL).

> ⚠️ **Исправление предыдущей документации.** В более ранней версии описывалось единственное ранжирование методом онлайн-обучения по формуле `score = α·pnl% + β·leverage + γ·age`. Это **не** реализованный алгоритм. Реальный контроллер использует скаляр `θ ∈ [0,1]` (степень хэрката) через проецированный OGD и пропорциональное распределение по ёмкости. Формула ранжирования `α/β/γ` была отклонена («2D OGD — взрыв размерности»). Классическая очередь `pnl% × leverage` — это базовый алгоритм HL, который MTF заменяет, а не то, что запускает MTF.

### 1. Степень хэрката — одномерный онлайн-градиентный спуск по θ

В каждом раунде выбирается скаляр `θ_t ∈ [0,1]` — доля дефицита текущего раунда, подлежащая хэркату:

```
B_t          = θ_t · D_t                                  # бюджет раунда
θ_needed_t   = clamp(B̂_needed / D_t, 0, 1)                 # предварительная оценка реальной потребности
grad         = D_t · sign(θ_t − θ_needed_t)
θ_{t+1}      = clamp(θ_t − η · grad, 0, 1)                 # шаг проецированного OGD
```

`D_t` — дефицит раунда, `B̂_needed` — оценка реальной потребности от оценщика цены исполнения.

**Размер шага η** (`AdlController::current_eta`):
- Режим по умолчанию — **Адаптивный** (следствие 1 из статьи): `η* = sqrt( (1 + 2·P_T^θ) / Σ D_t² )`, пересчитывается в каждом раунде на основе накопленной телеметрии (`path_variation`, `cumulative_squared_deficit`).
- В первом раунде (`Σ D_t² == 0`) используется резервное значение `η₀ = 0.01` (`default_eta`), настраиваемое через управление.
- Режим `Fixed(c)` фиксирует `η = c` (аварийный выключатель управления / воспроизводимость).

Контроллер имеет **ограничение динамического сожаления** (утверждение 1):

```
Reg_T^dyn  ≤  sqrt( (1 + 2·P_T^θ) · Σ D_t² )
```

доступно через `analytical_bound()`; `check_bound(slack)` проверяет, что эмпирическое сожаление ≤ `slack · bound` (значение slack по умолчанию — 4) в тестах хаоса.

Все дробные поля (`θ`, `η`, `path_variation`, …) имеют тип `rust_decimal::Decimal`; `sqrt` реализован через целочисленный метод Ньютона для `Decimal` (без `f64`); все аккумуляторы используют `saturating_*` (иначе `Decimal::from(u128 > MAX)` вызвал бы панику и остановил ядро). Состояние хранится в аккумуляторе BOLE, **слот 5**.

### 2. Распределение — детерминированное пропорциональное распределение по ёмкости

При бюджете `B_t` распределение осуществляется среди прибыльных контрагентов `W_t` (каждый с ёмкостью хэрката `u_i` = нереализованный PnL, подлежащий хэркату, тип `u128`):

```
total_u = Σ u_i
x_i     = floor( u_i · B_t / total_u )      capped at u_i        # 128-битное умножение, затем 128-битное деление
```

Целочисленный **остаток деления** `B_t − Σ x_i` перераспределяется поединично в **порядке возрастания AccountId** любому победителю с остаточной ёмкостью (`BTreeMap` итерирует в порядке ключей → побайтово идентично на всех узлах). Если `B_t > total_u`, раунд ограничен ёмкостью и `Σ x_i = total_u`.

Это решение заменяет отклонённые аллокаторы на основе векторного зеркального спуска и ILP (ILP оптимален, но использует недетерминированный решатель — не применим в блокчейне).

**Почему пропорциональное распределение** (реплей HL от 10 октября 2025):

| Алгоритм | Суммарный результат 10 октября (чем меньше, тем лучше) |
|-----------|-----------------------------------------|
| HL production (ROE heuristic) | ~$45M перехвата |
| **MTF pro-rata** | **$3.40M** (~13× лучше) |
| Vector mirror descent | $4.41M |
| Min-max ILP (оптимум, только вне сети) | $106k |

Пропорциональное распределение также даёт **0 %** нарушений монотонности (против 11,4 % у HL) и стабильность ранга ≈ 1,0 (против 0,34 у HL).

### Расчёт цены ADL (сторона чтения)

EVM-прекомпайл `0x0902 adl_pro_rata_price` позволяет Solidity-хелперу *запросить* цену исполнения VWAP для ADL объёма N, проходя очередь в соответствующем приоритете (ADL лонга: наивысшая цена первой; ADL шорта: наименьшая первой) — **все цены в формате с фиксированной точкой 1e8** (`price_e8`, `capacity_e8`). Прекомпайл работает только с пропорциональным распределением; OGD степени хэрката находится в основном состоянии, а не в статическом прекомпайле (степень хэрката — одно решение на раунд; расчёт цены может вызываться много раз в секунду).

## Механика хэрката

Хэркат — это не передача позиции: размер позиции контрагента **уменьшается**, а его нереализованный PnL конвертируется в реализованный убыток. Противоположная позиция ликвидируемого счёта аннулируется на ту же сумму.

Конкретный пример: счёт A открыт в лонг на 1 BTC по цене входа 100, счёт B — в шорт на 1 BTC по цене входа 100, марк-цена = 110.

- A прибыльный (+10 USDC нереализованного PnL).
- B — ликвидируемый счёт; позиция закрывается по марк-цене 110, но на счёте B только 5 USDC собственного капитала. Дефицит — 5 USDC.
- Страховой пул: 0 (исчерпан).
- ADL срабатывает против A:
  - Лонг A сокращается до 0,5 BTC.
  - A фиксирует +5 USDC PnL (хэркатнутая часть).
  - Оставшийся лонг A в 0,5 BTC остаётся по цене входа 100, марк-цена 110, +5 USDC нереализованного PnL.
  - Шорт B полностью закрыт.

A сохраняет нереализованный PnL по оставшейся позиции; A теряет только PnL *закрытой* части.

## Уведомления

События ADL поступают через [WS-канал `userEvents`](../api/ws/subscriptions.md#userevents):

```json
{
  "kind":        "adl",
  "asset":       0,
  "haircut_sz": "50000000",
  "realised_pnl": "5000000",
  "block":       12345
}
```

А также общее уведомление по счёту:

```json
{ "kind": "marginChange", "free": "...", ... }
```

Для автоматических ботов: обрабатывайте события `adl` так же, как принудительное исполнение — ваша позиция изменилась, протокол предоставил цену исполнения (марк-цена на блоке хэрката), после чего обычно следует пересмотр стратегии.

## Прогнозирование подверженности ADL

Для систем мониторинга рисков состояние счёта через [`/info`](../api/rest/info.md) включает поле `adl_rank_estimate`:

```json
"adl_rank_estimate": {
  "asset": 0,
  "percentile": 95,
  "score": 1.2
}
```

`percentile: 95` означает, что вы входите в топ-5% счетов по риску на данном активе. Счета в верхнем дециле подвергаются наибольшей экспозиции при срабатывании ADL.

Это оценка — фактическое ранжирование происходит в момент срабатывания ADL на основе текущего состояния. Для маркет-мейкеров, ведущих крупные книги заявок, основной риск — это концентрация (одна крупная позиция, доминирующая в открытом интересе по активу); диверсификация по активам снижает подверженность ADL.

## Граничные случаи

<details>
<summary>Показать граничные случаи</summary>

- **Несколько дефицитов в одном блоке.** Каждый распределяется независимо против текущего состава контрагентов. Ранги могут меняться между событиями.
- **Пустой набор контрагентов.** Если буквально ни одного прибыльного контрагента по тому же инструменту нет, дефицит социализируется в регистр «непокрытых убытков» страхового пула, подлежащий погашению при следующем пополнении пула. Для ликвидного актива такого не должно происходить; теоретически возможно на рынке с низкой ликвидностью (MIP-3).
- **Контрагент, участвующий в PM.** ADL по-прежнему целится в нереализованный PnL по тому же инструменту — участие в PM не изменяет гранулярность ADL на уровне актива. Сценарный движок PM видит состояние после хэрката начиная со следующего блока.
- **Спот-рынки.** У спота нет нереализованного PnL в смысле перпетуальных контрактов. ADL для спота не определён в V1; спот-позиции исключены из ранжирования ADL.

</details>

## Последовательность — ADL на активе с тонким хвостом

```
block T:   account X liquidates on asset 42 (MIP-3 market), loss = 100 USDC
           insurance pool on asset 42 = 60 USDC
           D_t (deficit) = 40 USDC
           severity controller: θ_t resolves to 1.0 (full deficit needed)
           B_t = θ_t · D_t = 40                  # budget this round
           winners W_t on asset 42 (by haircut capacity u_i):
             A: u_A = 30
             B: u_B = 50    →    total_u = 80
           pro-rata: x_i = floor(u_i · B_t / total_u)
             x_A = floor(30·40/80) = 15
             x_B = floor(50·40/80) = 25          # Σ = 40, no dust
           result:
             A's position haircut by 15 USDC of PnL realised, 15 kept
             B's position haircut by 25 USDC of PnL realised, 25 kept
```

(Распределение основано на **пропорциональной ёмкости**, а не на обходе по рейтингу: каждый победитель отдаёт одинаковую *долю* своей ёмкости — здесь 50 % — что и является свойством min-max справедливости пропорционального распределения. Сравните со старой моделью «ранжировать по очкам, списывать с верхнего уровня первым» — именно это код и не делает.)

## Часто задаваемые вопросы

<details>
<summary>Показать FAQ</summary>

**В: Можно ли отказаться от ADL?**
О: Нет. ADL — это протокольный механизм взаимной социализации убытков; отказ просто перекладывает потери на других. Защитой служит цель минимизации избыточного хэрката.

**В: Почему распределение пропорционально ёмкости PnL, а не по очереди на основе рейтинга?**
О: Пропорциональный хэркат урезает каждого победителя на одинаковую *долю* его хэркатного PnL — встроенная min-max справедливость, отсутствие нарушений монотонности (два счёта с одинаковой ёмкостью получают одинаковый результат) и стабильность ранга ≈ 1,0. На реплее от 10 октября 2025 этот подход оказался примерно в 13× лучше, чем очередь HL по эвристике ROE, и в пределах ~30 % от внецепочечного оптимума ILP, оставаясь при этом полностью детерминированным и пригодным для блокчейна. Онлайн-обучение применяется только к *степени* (сколько всего хэркатить); *кто платит* — это простое пропорциональное распределение.

**В: Учитывает ли ADL режим Strict-Iso?**
О: Да. ADL работает на уровне отдельного актива; позиции в режиме Strict-Iso являются кандидатами в контрагенты тогда и только тогда, когда они держат тот же актив.

**В: Является ли ранжирование детерминированным на всех валидаторах?**
О: Да — все входные данные (ёмкость PnL каждого победителя `u_i`, дефицит раунда `D_t`) считываются из зафиксированного состояния, а состояние контроллера степени хэрката (`θ`, `path_variation`, `Σ D_t²`, `η`) хранится в аккумуляторе BOLE, слот 5, включённом в LtHash, что гарантирует побайтовую идентичность верификации на каждом узле. Пропорциональное распределение использует 128-битное целочисленное умножение и деление с обработкой остатков в порядке возрастания AccountId — никаких вещественных чисел, никакого `HashMap`.

</details>
