# Портфельная маржа

:::info
**Запущено на devnet.** Движок сценариев полностью работоспособен: пользователи подключаются через действие
`UserPortfolioMargin` (с ограничением по собственному капиталу, по умолчанию ≥ 100 К USDC), а
SPAN-образная сетка сценариев (±5/10/20 % по цене × ±20/50 % по волатильности) вычисляет поддерживающую маржу в
реальном времени. Как интерфейс действий, так и движок сценариев развёрнуты и протестированы
на четырёхузловом консенсусном запуске.
:::

## TL;DR

Портфельная маржа (PM) рассматривает весь ваш счёт как единый показатель риска. Захеджированные или коррелированные позиции взаимно зачитываются; протокол взимает маржу исходя из наихудшего сценария на откалиброванной сетке шоков `(цена, волатильность)`. Типичная эффективность капитала для сбалансированного портфеля: **в 2–5 раз** выше классической.

PM является добровольной опцией, ограниченной по минимальному капиталу (по умолчанию ≥ 100 К USDC), и обратимой.

## Классическая маржа vs PM — сравнение

Пример захеджированного портфеля: лонг 1 BTC по $100, шорт 25 ETH по $4 (полностью нейтральный по USD, при условии идеальной корреляции BTC и ETH — что примерно так и есть).

**Классическая маржа:**

```
maint(BTC) = 1 BTC × $100 × 5% = $5
maint(ETH) = 25 ETH × $4 × 5% = $5
total maint = $10
```

**PM:**

```
scenario grid: (BTC ±10%, ETH ±10% × ±50% vol)
worst-case loss (correlated -10% on both): -$10 on BTC + (-$10) profit on short ETH = $0
worst-case loss (decorrelated, BTC -10% / ETH +10%): -$10 + (-$10) = -$20  ← worst
+ concentration penalty (basket is balanced; small)
total maint ≈ $20  (vs classical $10) ?
```

Подождите — это хуже. Пересчитаем: шорт 25 ETH по $4 = $100 шорт-нотионал, противоположный лонгу $100 BTC.

```
BTC -10%, ETH -10%:   long BTC loses $10, short ETH gains $10  → net  0
BTC +10%, ETH +10%:   long BTC gains $10, short ETH loses $10  → net  0
BTC -10%, ETH +10%:   long BTC loses $10, short ETH loses $10  → net -$20
BTC +10%, ETH -10%:   long BTC gains $10, short ETH gains $10  → net +$20
```

Максимальный убыток: $20 — но только в сценарии декорреляции. С учётом вероятностного взвешивания, шок декорреляции редок (30-дневная корреляция BTC/ETH ≈ 0,85). Откалиброванный набор сценариев PM учитывает это реалистично; фактическая поддерживающая маржа обычно составляет ~$5–10, а не наивные $20.

Классическая маржа в $10 просто не учитывает корреляцию. PM — учитывает.

## Как работает PM

> Движок портфельной маржи работает внутри в **центах USD** (плоскость `Decimal` в целых USDC × 100). Число PM **заменяет** классическую сумму поддерживающей маржи по каждому активу, а не добавляется к ней. Также существует read-сайд EVM-прекомпилятор (`portfolio_margin_eval`) для офчейн-котирования.

В рамках PM поддерживающая маржа определяется **SPAN-образным движком сценариев**, который перебирает детерминированную сетку шоков `(price-shock, vol-shock)` по всему портфелю:

```
for each (δp, δσ) in price_shocks × vol_shocks:
    scenario_total = Σ_i ( delta_pnl_i + gamma_pnl_i )
        delta_pnl_i = size_i · mark_i · δp                       # linear
        gamma_pnl_i = 0.5 · |size_i| · mark_i · iv_i · δσ · δp²   # convex (Black-Scholes-flavoured)
worst        = min( scenario_total over the grid )              # most negative
pm_margin    = max(0, −worst) + concentration_penalty
```

Параметры сетки и коэффициенты концентрации соответствуют настройкам движка по умолчанию (`PortfolioMarginEngine::default`):

| Параметр | По умолчанию (код) |
|-----------|----------------|
| Ценовые шоки | **±5 %, ±10 %, ±20 %** (`default_price_shocks` — 6 значений) |
| Шоки волатильности | **±20 %, ±50 %** (`default_vol_shocks` — 4 значения) |
| Размер сетки | **6 × 4 = 24 сценария** |
| Запасная подразумеваемая волатильность | `0.50` (50 % годовых), когда оракул не предоставляет `iv` |
| Порог концентрации | **50 %** от нетто-стоимости (`default_concentration_threshold`) |
| Штраф за концентрацию | **1 000 bps = 10 %** (`DEFAULT_CONCENTRATION_PENALTY_BPS`) |
| Минимальный капитал для подключения | `10_000_000` центов = **100 000 USDC** |

> ⚠️ **Исправления относительно предыдущего текста.** Реализованный движок: (1) включает **гамма/выпуклый член**, определяемый подразумеваемой волатильностью каждой позиции — прежняя документация моделировала сценарии как чисто линейный PnL; (2) использует единую **сетку из 24 сценариев (±5/10/20 × ±20/50)** без таблицы по классам активов — уровневая сетка «BTC/альткоины/длинный хвост» из старой документации отсутствует в коде (сетка является единым общим умолчанием движка, настраиваемым через динамический риск); (3) порог концентрации равен **50 %**, а не 60 %; (4) штраф за концентрацию равен **10 %** (1000 bps), а не 5 %.

Сценарии применяются **одновременно** ко всему портфелю. Взаимозачёт происходит естественным образом: составляющие `delta_pnl_i` захеджированного портфеля компенсируют друг друга в `scenario_total`, поэтому `worst` остаётся небольшим. (Примечание: текущий движок применяет каждый `δp` равномерно ко всем позициям — явная матрица попарных корреляций является задокументированным расширением, которое пока не реализовано как поле движка.)

`concentration_penalty` доначисляет маржу, когда один актив доминирует в портфеле:

```
max_abs = max over positions of |notional_i|        # cents
if max_abs / net_value > 0.50:
    over    = max_abs − 0.50 · net_value
    penalty = trunc( over · 1000 / 10000 )           # 10% of the over-concentrated portion
else:
    penalty = 0
```

(Пропускается при `net_value ≤ 0` — такой счёт обрабатывается путём BOLE с отрицательным собственным капиталом.)

## Подключение

```json
{ "type": "UserPortfolioMargin", "params": { "enabled": true } }
```

Только для мастер-счёта. Отключение выполняется симметрично.

| Ограничение | Значение |
|------------|-------|
| `pm_min_equity` | по умолчанию 100 000 USDC; задаётся через управление |
| Вступает в силу | с блока, следующего за подтверждением |
| Текущие нарушающие позиции | подключение отклоняется, если PM переводит счёт в T1+; сначала сократите позиции |

Отключение возвращает к классической модели с следующего блока. Отключение в состоянии T0+ разрешено (можно в любой момент перейти к более консервативной модели).

## Строгая изоляция

Даже при включённой PM можно пометить отдельные активы как **строго изолированные**. Позиция в режиме строгой изоляции:

- Рассчитывает собственную маржу независимо (классическая модель)
- НЕ участвует в движке сценариев PM
- Ликвидируется независимо — потери ограничены данным активом

Случаи применения:
- Новые / неликвидные активы, для которых матрица корреляций не откалибрована
- Спекулятивный бюджет, изолированный от захеджированного ядра портфеля
- Высокорисковые эксперименты

```json
{
  "type": "UpdateMarginMode",
  "params": { "asset": 7, "mode": "StrictIso" }
}
```

См. [режимы маржи](./margin-modes.md).

## Ликвидация при PM

Счета в режиме PM проходят стандартную [уровневую ликвидацию](./tiered-liquidation.md), однако `maint_margin` определяется значением PM, а не классической суммой.

Одна PM-специфичная особенность: частичное закрытие по T1 может достаточно изменить наихудший сценарий, чтобы оставшаяся позиция оказалась здоровой по PM, но не по классической модели. Это ожидаемое поведение — частичное закрытие рассчитывается исходя из PM в обоих направлениях.

```
before T1: long 1 BTC + short 25 ETH, PM maint = $20, account_value = $18, health = 0.9 → T1
T1 partial: close 50% of both legs
after: long 0.5 + short 12.5 ETH, PM maint = $10, account_value = $13, health = 1.3 → Safe
```

## Риски для оператора

PM повышает эффективность капитала для пользователей — именно поэтому она несёт риск для протокола: ошибка в спецификации сценариев может позволить счёту принять на себя больший риск, чем цепочка способна ликвидировать в штатном режиме. MetaFlux снижает этот риск с помощью:

- `concentration_penalty` (порог 50 % / ставка 10 %) для противодействия манипуляциям с PM через одиночный актив — **реализовано** в движке.
- Минимального порога `min_enroll_account_value_cents` (100 000 USDC) — **реализовано** (`meets_enrollment_floor`; отрицательная нетто-стоимость всегда является основанием для отказа).
- Консерватизма набора сценариев, динамической калибровки при смене режимов волатильности, лимита PM на счёт (`pm_max_account_notional`, проектное значение 100M USDC) и обязательного возврата к классической модели при сбое движка сценариев — **намерение дизайна / пока не реализовано как поля `PortfolioMarginEngine`**; текущий движок содержит только сетку, параметры концентрации и минимальный порог для подключения.

Набор сценариев, величины шоков и коэффициенты концентрации являются параметрами управления (динамический риск). Подпишитесь на обновления параметров, если работаете вблизи маржинальных лимитов.

## Подробный пример — штраф за концентрацию

Штраф сравнивает **наибольший абсолютный нотионал по одному активу** с **нетто-стоимостью** счёта (`денежные средства + Σ размер × маркировка`) при значениях по умолчанию: порог **50 %**, ставка **10 %**.

Счёт: нетто-стоимость `$1000`, крупнейшая позиция `|notional_BTC| = $700`.

```
frac    = 700 / 1000 = 0.70  > 0.50 threshold
over    = 700 − 0.50 × 1000 = 700 − 500 = $200
penalty = trunc( 200 × 1000 / 10000 ) = $20      # 10% of the over-concentrated portion
```

Если проход по сценариям PM вычисляет (например) максимальный убыток `$25`, то `pm_margin = max(0, −worst) + penalty = $25 + $20 = $45`.

Более сбалансированный портфель, в котором ни один актив не превышает 50 % нетто-стоимости, не несёт **никакого** штрафа — `pm_margin` равна только наихудшему сценарному убытку. Штраф препятствует концентрации в одном активе внутри PM; классическая маржа такого различия не делает.

## Запросы

```bash
curl -X POST https://api.devnet.mtf.exchange/info \
  -H 'content-type: application/json' \
  -d '{"type":"account_state","address":"0x<addr>"}'
```

Нативный запрос [`account_state`](../api/rest/info.md#account_state) возвращает
`pm_enabled` (признак активности PM для счёта) вместе с `maint_margin`,
`init_margin`, `health` и `tier`. Когда PM включена, `maint_margin` уже
отражает поддерживающую маржу, рассчитанную через PM:

```json
{
  "pm_enabled":   true,
  "maint_margin": "8",
  "init_margin":  "12",
  "health":       "...",
  "tier":         "Safe"
}
```

> **Планируется.** Сравнение классической маржи и PM, а также детализация наихудшего сценария
> (какая комбинация ценового/волатильностного шока определила значение PM) **пока не выведены**
> в виде отдельных полей в ответе
> [`account_state`](../api/rest/info.md#account_state) — движок PM вычисляет их внутренне,
> но сегодня на поверхность выводится только итоговое значение `maint_margin`. В будущем
> планируется добавить поле с детализацией по сценариям в `account_state`.

## Граничные случаи

<details>
<summary>Показать граничные случаи</summary>

- **Сбой движка PM**: редкая ситуация; протокол переходит к `max(classical_maint, prior_pm_maint)` для данного блока. Ликвидации в этом блоке используют консервативный резервный расчёт.
- **Открытие кросс-активной позиции в режиме волатильного рынка**: новая позиция допускается через движок PM на момент открытия — но движок считывает веса сценариев из зафиксированного состояния, что исключает манипулятивное переключение режимов.
- **Подключение PM в состоянии T0**: разрешено. PM может вывести счёт из T0 (если даёт меньшую маржу) или оставить в T0 (если нет). Автоматического возврата при ухудшении значения PM не происходит.
- **Отключение PM в состоянии T0+**: разрешено. Полезно для возврата к классической модели при неисправности PM на уровне протокола.

</details>

## Часто задаваемые вопросы

<details>
<summary>Показать FAQ</summary>

**В: Могут ли субаккаунты иметь разные настройки PM?**
О: Да. Каждый субаккаунт независим. Мастер-счёт может быть подключён к PM, пока его субаккаунты работают в классическом режиме, и наоборот.

**В: Какова стоимость газа для расчёта PM?**
О: Выше, чем для классической модели (из-за сетки сценариев), но ограничена. Протокол кэширует результаты сценариев на уровне счёта и пересчитывает их только при изменении позиций или обновлении параметров сценариев.

**В: Прозрачна ли PM — можно ли увидеть точное значение маржи до выставления ордера?**
О: Да. `/info clearinghouseState` возвращает как классическую маржу, так и PM. SDK предоставляют эту информацию через вспомогательный метод `getOrderImpact()`.

**В: Получают ли листинги MIP-3 зачёт в PM?**
О: Только если рыночная спецификация листинга включает актив в корреляционную матрицу PM. Многие листинги с длинным хвостом по умолчанию будут в режиме Strict-Iso из соображений безопасности. Проверяйте `market_info.pm_eligible` для каждого рынка.

</details>
