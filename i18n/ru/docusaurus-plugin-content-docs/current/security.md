# Модель безопасности

:::tip
**Стабильно.**
:::

Что протокол гарантирует, что не гарантирует и где риск несёте вы.

## Коротко о главном

- Протокол гарантирует: детерминированную семантику конечного автомата, авторизацию через подписи и on-chain аудируемость каждого действия.
- Протокол НЕ гарантирует: корректность оракула за рамками опубликованного состава, хранение вашего приватного ключа или отсутствие рисков управления.
- Программы bug bounty работают на сторонних платформах; ожидается скоординированное раскрытие уязвимостей.

## Поверхность доверия

### Что принадлежит протоколу

| Уровень | Гарантия протокола |
|-------|--------------------|
| Консенсус | M-of-N соглашение валидаторов; детерминированные переходы состояний; подписанные блоки |
| Конечный автомат | Идентичное выполнение на всех валидаторах; детерминированное время; арифметика только на целых числах |
| Восстановление подписи | EIP-712 поверх msgpack; восстановление secp256k1; карта одобрения агентов |
| Маркировочная цена | Формула состава + полоса допустимых значений, описанные в разделе [маркировочные цены](./concepts/mark-prices.md) |
| Ликвидация | Многоуровневая лестница срабатывает детерминированно против зафиксированного состояния |
| Расчёт комиссий | Таблица уровней + коэффициент сжигания применяются одинаково для каждого исполнения |

Узел, не соблюдающий эти правила, не является валидным валидатором — консенсус его отклоняет.

### Что принадлежит пользователю

| Уровень | Ответственность пользователя |
|-------|--------------------|
| Хранение приватного ключа | Холодное хранилище для мастер-ключа; горячее — для агентов; соблюдение ротации |
| Логика офф-чейн ботов | Какие ордера выставлять, когда пополнять маржу, когда закрывать позиции |
| Управление рисками | Размер позиции относительно бакета / капитала |
| Риск контрагента моста | Выбор кошелька исходной сети и маршрута через мост |

### Где доверие разделено

| Уровень | Допущение доверия |
|-------|------------------|
| Состав оракула | Доверие к оракулу, публикуемому валидаторами, в рамках задокументированного состава |
| MetaBridge | Доверие к валидаторному набору MetaFlux — мультиподпись с весом ⅔ стейка на ВСЕ переводы через мост, включая USDC, — за окном оспаривания вывода (те же ключи, что и у консенсуса; без стороннего сервиса аттестации; см. [мост](./bridge/)) |
| Управление | Изменения параметров контролируются управлением; доверие к тому, что управление действует в интересах протокола |

Принцип: доверие минимизировано, а не устранено. Там, где разделённое доверие неизбежно (оракулы, сервисы аттестации), поверхность доверия задокументирована и ограничена.

## Модель угроз

### Вне области видимости протокола

- Пользователь подписал ордер, о котором впоследствии пожалел.
- Горячий ключ пользователя украден, и злоумышленник подписывает сделки (именно поэтому агенты лишены права вывода средств).
- Пользователь не пополнил маржу и был ликвидирован по задокументированной многоуровневой лестнице.
- Пользователь принял котировку RFQ по невыгодной цене.
- Пользователь внёс средства в хранилище, которое потеряло деньги.
- Параметр, установленный управлением, изменился в допустимых пределах и повлиял на позицию пользователя.

Это не проблемы безопасности. Это операционные риски, которые несут пользователи.

### В области видимости (сообщайте об этом)

- Подделка подписи / принятие недействительных подписей.
- Недетерминированное выполнение конечного автомата (два валидатора расходятся во мнении о зафиксированном состоянии).
- Повторное использование действительных подписей в разных сетях (обход изоляции домена chainId).
- Эскалация привилегий (агент получает право вывода; немастер запускает действие только для мастера).
- Потеря средств за пределами задокументированных механик ликвидации / ADL / комиссий.
- Уязвимости интеграции моста (мультиподпись MetaBridge / проверка кворума ⅔, повтор message-id, обход окна оспаривания).
- Обход аутентификации WS (подписка на приватные каналы без авторизации).
- DoS, препятствующий выполнению допустимых действий в пределах задокументированных ограничений по частоте запросов.
- Задокументированные инварианты, которые не выполняются (например, обход монотонности nonce).

## Политика раскрытия

По уязвимостям безопасности:

1. **Не открывайте** публичный issue на GitHub.
2. Отправьте письмо на `security@mtf.exchange` (PGP-ключ на сайте до запуска) с указанием:
   - Описания уязвимости
   - Шагов воспроизведения
   - Вашей оценки последствий
   - Ваших контактных данных для дальнейшей связи
3. Ожидайте ответа с подтверждением получения в течение 48 часов.
4. Срок скоординированного раскрытия: 90 дней с момента подтверждения или раньше, если патч выпущен и развёрнут.

Программа bug bounty с многоуровневыми вознаграждениями работает на сторонней платформе; подробности будут опубликованы до запуска.

## On-chain аудируемость

Каждое действие навсегда остаётся on-chain. Инструменты криминалистического анализа позволяют восстановить:

- Полную историю действий по адресу (подписант, action_hash, блок фиксации).
- Полную историю ликвидаций (аккаунт, уровень, маркировочная цена, реализованный убыток).
- Жизненный цикл одобрения агентов (мастер, агент, события одобрения / истечения / повторного одобрения).
- Траекторию NAV хранилища и таблицу депозиторов.

Обозреватели предоставляют эти данные; индексаторы поставляют их в удобном для запросов формате.

## Детерминированное выполнение

Конечный автомат является чистой функцией:

```
state_{t+1} = apply(state_t, ordered_actions_in_block)
```

Валидаторы, расходящиеся во мнении о `state_{t+1}`, являются несоответствующими. Источники недетерминизма (числа с плавающей точкой, итерация по неупорядоченным отображениям, системное время) запрещены в пути консенсуса. Аудиты явно направлены на проверку этого свойства.

Если ваш бот вычисляет будущее значение состояния (например, ожидаемую PM-маржу после ордера), он может вычислить его идентично тому, что вычислила бы сеть, при тех же входных данных. Всё необходимое для этого содержится в спецификации протокола.

## Рекомендации по операционной безопасности

Для институциональных / продуктивных пользователей:

| Рекомендация | Зачем |
|----------------|-----|
| Мультиподпись для мастер-аккаунта | Компрометация одного ключа — единственная точка отказа |
| Один агент на хост / стратегию | Радиус поражения при компрометации ограничен |
| Короткие сроки действия агентов (≤ 30 дней) | Принудительный ритм ротации |
| HSM / аппаратный кошелёк для мастера и мастера субаккаунта | Поверхность подписи в холодном хранилище |
| Ограничивайте исходящий трафик собственного бота | Предотвращение бесконтрольных циклов, исчерпывающих бюджет аккаунта |
| Отдельный агент мониторинга рисков | Пополнение маржи независимо от торговой логики |
| Дублирование узлов для WS-потоков | Устойчивость к задержкам и переподключению |
| Подписка на оповещения о статусе | Инциденты на стороне оператора влияют на задержку / доступность |
| Аудит согласования action_hash → commit | Обнаружение тихих пропусков |
| Тестирование миграций при изменениях на тестнете за 60 дней вперёд | Избегайте сюрпризов в день запуска mainnet |

## Что если с сетью что-то пошло не так

Остановки консенсуса (например, раздел сети, препятствующий кворуму) операционно редки, но возможны. Во время остановки:

- `/info` продолжает отдавать данные из последнего зафиксированного состояния.
- `/exchange` отклоняет запросы с ответом `503 chain_unavailable`.
- WS поддерживает соединение; новые сообщения не поступают до возобновления работы.
- Ликвидации приостанавливаются — маркировочная цена остаётся на последнем значении; во время остановки переходов между уровнями не происходит.

После возобновления сеть воспроизводит состояние с последнего зафиксированного блока. Никаких потерь состояния не происходит. Время продвигается на основе времени блока, определяемого консенсусом, а не системных часов; платежи финансирования накапливаются и выполняются при возобновлении.

Если узел обнаруживает остановку консенсуса, переключитесь на другой узел / шлюз (набор валидаторов распределён). Архитектура протокола предполагает, что ≥ 2/3 валидаторов честны и онлайн; кратковременные остановки в пределах этого порога отказов ожидаемы.

## Часто задаваемые вопросы

<details>
<summary>Показать FAQ</summary>

**В: Формально ли верифицирован консенсус?**
О: Модель консенсуса формально специфицирована; формальная верификация (TLA+ / Stateright) охватывает инварианты безопасности и живучести. Производственная реализация аудируется на соответствие спецификации.

**В: Могут ли оракулы быть подвергнуты слэшингу?**
О: Данные оракула подписываются набором валидаторов. Валидатор, систематически публикующий явно неверные данные оракула (выходящие за пределы допустимых диапазонов), подлежит слэшингу согласно правилам [стейкинга](./concepts/staking.md).

**В: Каков максимальный убыток пользователя при известной ошибке протокола?**
О: Зависит от ошибки. Архитектура ограничивает радиус поражения — изоляция субаккаунтов, блокировка вывода для агентов, лимиты моста по коридорам, страховой пул — однако глубокая ошибка конечного автомата теоретически может привести к опустошению аккаунтов. Именно поэтому важны раскрытие уязвимостей и непрерывный аудит.

**В: Может ли протокол откатить состояние?**
О: Не в одностороннем порядке. Откат требует скоординированного решения набора валидаторов и рассматривается как хард-форк. Стандартная политика: никаких откатов при индивидуальных потерях пользователей; откат только при общепротокольных ошибках, нарушающих корректность консенсуса. Точный порог определяется управлением.

</details>
