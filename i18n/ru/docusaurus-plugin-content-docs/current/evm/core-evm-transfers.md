# Переводы Core ↔ EVM

:::tip
**Работает в devnet.** Действия по переводу средств из EVM в Core (`SpotSend`, `SendAsset`,
`UsdClassTransfer`, `VaultTransfer` через CoreWriter) и материализация кредитов из Core в EVM
функционируют и протестированы. [Мост](../bridge/) (кросс-чейн хранение) активен.
:::

Стоимость перемещается между **Core** (клиринговая система L1 / книга спот-балансов) и стороной
**EVM** в двух направлениях. Оба направления детерминированы и привязаны к конкретному аккаунту.

## EVM → Core (через CoreWriter)

Контракт перемещает стоимость в Core, отправляя действие L1 через
[CoreWriter](interacting-with-core.md#writing-to-core--corewriter) (`0x3333…3333`).
Действующий аккаунт — это вызывающий контракт (`msg.sender`):

| Действие | Эффект |
|--------|--------|
| `SpotSend` | Перевод спот-токена на другой аккаунт в Core |
| `SendAsset` | Универсальный перевод активов (классы perp / spot / vault) |
| `UsdClassTransfer` | Перемещение USDC между счетами классов perp и spot |
| `VaultTransfer` | Внесение средств в хранилище или вывод из него |

На эти операции распространяется правило атомарности CoreWriter: вызов тратит газ и генерирует
событие `RawAction`; любой последующий сбой на стороне L1 происходит **молча** (без отката EVM).

## Core → EVM (системные псевдо-транзакции)

Когда эффект блока begin-block на стороне L1 должен отразиться на стороне EVM — например, спот-перевод,
получатель которого является адресом на стороне EVM, или входящий минт моста — он ставится в очередь
и материализуется как **детерминированная системная псевдо-транзакция в следующем блоке EVM**:

| Операция | Источник | Масштаб суммы |
|----|--------|--------------|
| `SpotCredit` | спот-баланс L1, зачисленный 20-байтовому EVM-получателю | фиксированная точка `1e8` |
| `BridgeMint` | входящий минт [MetaBridge](../bridge/) (например, USDC) | `1e6` (нативный USDC) |

Порядок и пропускная способность:

- Ставятся в очередь по **раундам L1**, обрабатываются в порядке возрастания номера раунда, FIFO внутри раунда —
  таким образом, два валидатора материализуют одни и те же операции в одном и том же порядке (детерминизм).
- За каждую операцию взимается **системный газ**, который списывается из **эластичного
  системного лимита газа на блок** (масштабируется вместе с бюджетом газа блока); оставшиеся операции
  переносятся на следующий блок. Кредиты Core→EVM поступают в течение нескольких блоков, а не мгновенно
  в том же блоке, где были инициированы.

## Кросс-чейн (отдельный механизм)

`CrossChainSend` (действие CoreWriter 19) **не** перемещает стоимость в локальный EVM —
оно ставит в очередь вывод средств в [хранилищный мост MetaBridge](../bridge/), который
производит выплату в целевом блокчейне (Base / Solana) по совместной подписи ⅔ валидаторов
после окончания периода оспаривания.
