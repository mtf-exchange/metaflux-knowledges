# Тайминги взаимодействий

:::tip
**Активно на devnet.** Ритм блоков и тайминги взаимодействий — задержки действий CoreWriter и материализация зачислений Core→EVM — функционируют в соответствии с описанием. Ритм и лимиты могут быть скорректированы до запуска.
:::

Сколько времени занимает каждое взаимодействие EVM↔Core — чтобы бот мог корректно рассчитывать окна подтверждения.

## Ритм блоков

Один унифицированный блок на каждый раунд консенсуса с ритмом **менее одной секунды** — отдельного медленного потока не существует, поэтому торговля, переводы, вызовы CoreWriter, чтение через прекомпилы и развёртывание контрактов подтверждаются в одном раунде. Значение `block.timestamp` формируется консенсусом (см. [Модель исполнения](execution-model.md)).

## EVM → Core (CoreWriter)

1. Контракт вызывает `sendRawAction`; вызов сжигает газ и немедленно генерирует событие `RawAction`.
2. L1 обрабатывает действие после короткой **задержки действия** (оно ставится в очередь, а не применяется мгновенно), после чего вносит изменения в состояние Core.
3. **EVM-подтверждения нет** — контракт должен отслеживать результат на стороне Core (например, через API или последующее чтение через прекомпил), а не по возвращаемому значению `sendRawAction`.

Следствие для архитектуры: считайте действие CoreWriter вызовом по принципу **«отправить и подтвердить позже»**, а не синхронным вызовом.

## Core → EVM (зачисления)

Зачисление Core→EVM (`SpotCredit` / `BridgeMint`) материализуется как системная псевдо-транзакция в **последующем** блоке, упорядоченном по раунду L1 и ограниченном эластичным системным газовым слотом на блок (см. [Переводы Core ↔ EVM](core-evm-transfers.md)). Зачисление **не** отображается в том же блоке, который его инициировал; ожидайте его через небольшое количество блоков.

## Чтение через прекомпилы

Чтение через прекомпил с помощью `staticcall` выполняется в пределах текущего блока. На данный момент прекомпилы для чтения являются **безсостоятельными вспомогательными функциями котирования** (они вычисляют результат на основе входных данных, переданных вызывающей стороной); **чтение с поддержкой живого состояния Core** (прямой запрос к позициям / книге заявок самой цепочки) станет доступно позже — при этом результат чтения будет отражать состояние Core на момент блока вызова.
