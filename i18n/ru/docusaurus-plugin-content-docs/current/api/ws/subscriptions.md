# Каналы подписки WS

:::info
**Статус.** `l2_book`, `bbo`, `trades`, `active_asset_ctx`, `all_mids`, `fills`, `user_events`, `candles`, `order_updates`, `notifications`, `ledger_updates`, `active_asset_data`, `user_fundings`, `user_twap_slice_fills`, `user_twap_history`, `account_state`, `spot_state` и `web_data2` работают в реальном режиме и передают реально подтверждённые данные — фрейм публикуется только при фактическом изменении состояния канала с момента последнего коммита. Всё остальное из раздела [Дорожная карта](#roadmap--not-yet-available) пока не реализовано. Жизненный цикл подключения и формат фреймов описаны в [WS README](./index.md). Каналы для конкретного рынка (`l2_book`, `bbo`, `trades`, `active_asset_ctx`) требуют параметра `coin`; `candles` требует `coin` **и** `interval`; каналы для конкретного аккаунта (`fills`, `user_events`) требуют `user` (0x-адрес); `active_asset_data` требует **одновременно** `user` и `coin`; `all_mids` не требует ни того, ни другого.
:::

:::info
**Имена каналов в формате snake_case (нативный MTF).** Это нативная точка входа узла `/ws`, поэтому имена каналов в протоколе используют snake_case (`l2_book`, `user_events`, …). Клиенты, желающие использовать имена каналов в стиле HL camelCase (`l2Book`, `userEvents`, `userFills`, `candle`, …), подключаются к шлюзу через **`/hl/ws`** (режим совместимости с HL), который транслирует их в нативные snake_case-каналы. Согласно маршрутизации единого шлюза: `<net>-gateway.mtf.exchange/ws` = нативный snake_case, `/hl/ws` = HL camelCase.
:::

Протокол фреймов аналогичен протоколу HL; **имена каналов используют нативный MTF snake_case**. Подписка осуществляется следующим образом:

```json
{ "method": "subscribe", "subscription": { "type": "<channel>", "coin": "<coin>" } }
```

В ответ вы получаете подтверждение (`subscriptionResponse`), начальный снимок состояния (`is_snapshot: true`), а затем изменения в реальном времени в виде пушей `{"channel":...,"data":...}` (`is_snapshot: false`). Пуш отправляется только при фактическом изменении состояния канала с момента последнего коммита; если канал не изменился, ничего не отправляется. Параметр `coin` **обязателен** для каналов по конкретному рынку (`l2_book`, `bbo`); порядок канонизации описан в разделе [Параметр Coin](./index.md#coin-parameter) (числовой идентификатор актива или символ → ключ asset-id).

## Статус каналов — сводка

| Канал | Статус | Ключ | Источник данных |
|---------|--------|:-------:|-------------|
| `l2_book` | **активен** | `coin` (обязателен) | подтверждённый стакан, при изменении |
| `bbo` | **активен** | `coin` (обязателен) | подтверждённый стакан, при изменении |
| `trades` | **активен** | `coin` (обязателен) | сделки подтверждённого блока, при появлении новых |
| `active_asset_ctx` | **активен** | `coin` (обязателен) | маркировочная/оракульная цена, финансирование, открытый интерес по рынку, при изменении |
| `all_mids` | **активен** | нет | маркировочная цена по рынкам, при изменении |
| `fills` | **активен** | `user`/`address` (обязателен) | сделки подтверждённого блока по аккаунту |
| `user_events` | **активен** | `user`/`address` (обязателен) | сделки подтверждённого блока по аккаунту (дополнительные типы событий будут добавлены) |
| `candles` | **активен** | `coin` + `interval` (оба обязательны) | сделки подтверждённых блоков, агрегированные в OHLCV-бары, при изменении |
| `order_updates` | **активен** | `user`/`address` (обязателен) | жизненный цикл ордеров по аккаунту (выставление / исполнение / отмена / отклонение), при изменении |
| `notifications` | **активен** | `user`/`address` (обязателен) | уведомления о марже и ликвидации по аккаунту, при изменении |
| `ledger_updates` | **активен** | `user`/`address` (обязателен) | движение средств по аккаунту (депозит / вывод / перевод), при изменении |
| `active_asset_data` | **активен** | `user` **и** `coin` (оба обязательны) | контекст кредитного плеча / режима маржи / максимального размера сделки по паре (аккаунт, монета), при изменении |
| `user_fundings` | **активен** | `user`/`address` (обязателен) | реализованные выплаты финансирования по аккаунту, при изменении |
| `user_twap_slice_fills` | **активен** | `user`/`address` (обязателен) | частичные исполнения TWAP по аккаунту (`{fill, twapId}`), при изменении |
| `user_twap_history` | **активен** | `user`/`address` (обязателен) | жизненный цикл TWAP по аккаунту (`{time, state, status}`: активирован / завершён / прерван), при изменении |
| `account_state` | **активен** | `user`/`address` (обязателен) | состояние клирингового дома PERP по аккаунту — скаляры маржи, позиции, балансы — при изменении |
| `spot_state` | **активен** | `user`/`address` (обязателен) | состояние клирингового дома SPOT по аккаунту — балансы по токенам — при изменении |
| `web_data2` | **активен** | `user`/`address` (обязателен) | составной UI-снимок по аккаунту — клиринговый дом + спот-балансы + открытые ордера + капитал хранилища + статус биржи — при изменении |

Подписка на любой другой `type` возвращает `{"channel":"error","data":{"error":"unknown channel: <name>"}}`.

## Активные каналы

### `l2_book`

Агрегированный стакан заявок L2 для одного рынка. **Требует `coin`.**

```json
{ "method": "subscribe", "subscription": { "type": "l2_book", "coin": "BTC" } }
```

Начальный снимок и каждый последующий пуш имеют следующую структуру:

```json
{
  "channel": "l2_book",
  "data": {
    "coin": "BTC",
    "levels": [
      [ { "px": "10050000000", "sz": "12", "n": 2 }, { "px": "10049000000", "sz": "3", "n": 1 } ],
      [ { "px": "10051000000", "sz": "4", "n": 1 }, { "px": "10052000000", "sz": "6", "n": 1 } ]
    ],
    "time": 1735689600000
  }
}
```

- `levels` — массив `[bids, asks]`. Биды упорядочены от лучшего (наивысшего); аски — от лучшего (наименьшего).
- Каждый уровень представлен как `{ px, sz, n }`: `px` / `sz` — сырые значения в формате фиксированной точки в виде десятичных **строк** (масштабирование тика по активу применяется на уровне шлюза), `n` — количество рестинг-ордеров по данной цене.
- Каждая сторона ограничена **20 агрегированными уровнями**.
- `time` — `last_trade_ms` стакана (определяется консенсусом); `0` до первой сделки.

Каждый пуш — это **полный снимок топ-20 уровней**, а не частичный дифф. Обёртка фрейма содержит булево поле `is_snapshot` — `true` для первоначального снимка при подписке, `false` для последующих пушей при изменениях — однако **тело всегда содержит полный топ-20 стакана**, поэтому это поле носит информационный характер: заменяя локальный стакан при каждом фрейме, вы остаётесь в консистентном состоянии.

Частота: по событию изменения — фрейм отправляется только при фактическом изменении стакана с момента последнего коммита; коммит, не затронувший этот стакан, ничего не публикует. Если монета не соответствует ни одному известному рынку, вы всё равно получаете подтверждение, но тело снимка — пустой стакан (`"levels": [[], []]`, `"time": 0`) — и дальнейших пушей не будет.

### `bbo`

Лучший бид/офер на вершине стакана для одного рынка. Облегчённая версия `l2_book`. **Требует `coin`.**

```json
{ "method": "subscribe", "subscription": { "type": "bbo", "coin": "BTC" } }
```

```json
{
  "channel": "bbo",
  "data": {
    "coin": "BTC",
    "time": 1735689600000,
    "bbo": [
      { "px": "10050000000", "sz": "12", "n": 2 },
      { "px": "10051000000", "sz": "4", "n": 1 }
    ]
  }
}
```

- `bbo` — массив `[best_bid, best_ask]`. Каждый элемент — это уровень `{ px, sz, n }` или `null`, если соответствующая сторона пуста.
- `time` — `last_trade_ms`, аналогично `l2_book`.

Частота: по событию изменения — фрейм отправляется только при фактическом изменении вершины стакана с момента последнего коммита; если стакан не изменился, в этом коммите ничего не публикуется.

### `trades`

Публичная лента сделок для одного рынка — по одной записи на каждое исполнение, публикуется в коммитах, в которых рынок действительно торговал. `px`/`sz` — целочисленные строки в **плоскости 1e8**; `side` — сторона тейкера (`"B"` покупка / `"A"` продажа); `time` — временна́я метка консенсусного блока (мс); `tid` — детерминированный идентификатор сделки; `users` — массив `[taker, maker]` (тейкер первым, как агрессор).

```json
{ "method": "subscribe", "subscription": { "type": "trades", "coin": "BTC" } }
```

```json
{ "channel": "trades", "data": { "coin": "BTC", "side": "B", "px": "6700000000000", "sz": "10000000", "time": 1735689600123, "tid": 1234567890, "users": ["0x..taker", "0x..maker"] } }
```

### `active_asset_ctx`

Контекст конкретного рынка — маркировочная / оракульная цена, ставка финансирования и открытый интерес — публикуется при изменении. **Требует `coin`.** Тело содержит те же поля и единицы измерения, что и REST-запрос [`market_info`](../rest/info.md#market_info): `mark_px` / `oracle_px` — в **целых USDC**, выровненные по тику рынка (усечение до ценового тика), а блок `funding` идентичен блоку `market_info.funding`. Используется тот же построитель записей, что и для REST-запроса, поэтому WS-пуш ctx никогда не расходится с `market_info`.

```json
{ "method": "subscribe", "subscription": { "type": "active_asset_ctx", "coin": "BTC" } }
```

```json
{
  "channel": "active_asset_ctx",
  "data": {
    "coin": "BTC",
    "mark_px": "66735.25",
    "oracle_px": "66700",
    "funding": {
      "rate_per_hr": "0",
      "cap_per_hr": "400",
      "interval_ms": 3600000,
      "next_payment_ts": 0
    },
    "open_interest": "5000000000"
  }
}
```

- `mark_px` / `oracle_px` — в целых USDC, выровненные по тику (`"0"` если не задано). Та же плоскость, что и в `market_info`, а НЕ плоскость 1e8 стакана.
- `funding` — `{rate_per_hr, cap_per_hr, interval_ms, next_payment_ts}`, идентично блоку `market_info.funding` для REST (`null` для неизвестного рынка — см. ниже). `rate_per_hr` — последний почасовой образец ставки финансирования (до применения кэпа), `cap_per_hr` — ограничение ставки по данному рынку; оба значения — **строки в базисных пунктах**, усечённые к нулю (например, `"400"` = 0,04/ч); `interval_ms` — периодичность финансирования (`3600000` = 1 ч); `next_payment_ts` — эпоха в мс, `0` до первого образца финансирования.
- `open_interest` — текущий открытый интерес, строка в формате фиксированной точки (`"0"` при отсутствии стакана).

Частота: по событию изменения — фрейм отправляется только при фактическом изменении контекста данного рынка с момента последнего коммита; если контекст не изменился, в этом коммите ничего не публикуется.

Если монета не соответствует ни одному известному рынку, вы всё равно получаете подтверждение, однако снимок — **честно-пустой**: нулевые цены / открытый интерес и блок `funding` равен `null`; дальнейших пушей не будет (клиент, десериализующий фиксированную структуру ctx, не получит ошибку):

```json
{ "channel": "active_asset_ctx", "data": { "coin": "DOGE", "mark_px": "0", "oracle_px": "0", "funding": null, "open_interest": "0" } }
```

### `all_mids`

Глобальная карта мид-цен — маркировочная цена по каждому рынку, публикуется при изменении мидов. Ключ — монета; значения — маркировочные цены в целых USDC с выравниванием по тику, как возвращает REST-запрос [`markets`](../rest/info.md#markets). Параметр `coin` не требуется.

```json
{ "method": "subscribe", "subscription": { "type": "all_mids" } }
```

```json
{ "channel": "all_mids", "data": { "mids": { "BTC": "66703.35", "ETH": "1856.49", "SOL": "73.95", "MTF": "5" } } }
```

### `fills` <a id="fills"></a>

Поток исполнений по аккаунту. Требует `user` (0x-адрес; принимается также `address`) — параметр `coin` НЕ нужен. Каждое исполненное совпадение доставляет запись **обеим** сторонам — каждой со своей точки зрения, с одинаковым набором полей `{coin, side, px, sz, time, oid, cloid, tid, crossed}`:

- запись **тейкера** — собственный `oid` тейкера, его `cloid` (или `null`), сторона тейкера, `crossed: true`;
- запись **мейкера** — собственный `oid` мейкера, `cloid: null` (cloid рестинг-стороны не фиксируется), **противоположная** сторона, `crossed: false`.

Оба плеча одного совпадения имеют одинаковый `tid` (то же значение, что публикуется в ленте `trades`). `px`/`sz` — строки в плоскости 1e8. Записи исполнений по аккаунту **не содержат массива `users`** — адреса контрагентов доступны только в публичной ленте [`trades`](#trades), но не в канале уровня аккаунта.

```json
{ "method": "subscribe", "subscription": { "type": "fills", "user": "0x<address>" } }
```

Начальный снимок — пустой массив `[]`; каждый пуш — массив с одной записью об исполнении:

```json
{ "channel": "fills", "data": [ { "coin": "BTC", "side": "B", "px": "6700000000000", "sz": "10000000", "time": 1735689600123, "oid": 42, "cloid": "0xab..", "tid": 1234567890, "crossed": true } ] }
```

### `user_events` <a id="userevents"></a>

Лента событий по аккаунту. Требует `user` (0x-адрес) — параметр `coin` НЕ нужен. На сегодня содержит события `fills`; типы событий ликвидации и финансирования будут добавлены как соседние ключи.

```json
{ "channel": "user_events", "data": { "fills": [ { "coin": "BTC", "side": "B", "px": "6700000000000", "sz": "10000000", "time": 1735689600123, "oid": 42, "cloid": "0xab..", "tid": 1234567890, "crossed": true } ] } }
```

Нативное имя канала — `user_events` (snake_case); на шлюзе через `/hl/ws` (режим совместимости с HL) эквивалентное имя — HL-вариант `userEvents`.

:::warning
`user_events` содержит данные уровня аккаунта, однако в настоящее время **аутентификация отсутствует** — любое подключение может подписаться на ленту любого адреса. Не воспринимайте этот канал как приватный до реализации механизма auth-at-subscribe; для аутентифицированных чтений и записей используйте `post` с подписанным действием.
:::

### `candles`

Скользящие OHLCV-бары для одного рынка при заданном размере бара. **Требуются оба параметра — `coin` и `interval`** — они вместе образуют ключ маршрутизации, поэтому подписки на `1m` и `5m` для одного рынка — это независимые подписки, каждая со своим снимком и пушами.

```json
{ "method": "subscribe", "subscription": { "type": "candles", "coin": "BTC", "interval": "1m" } }
```

- `interval` ∈ `1m` / `5m` / `15m` / `1h` / `4h` / `1d`. Отсутствующий или нераспознанный `interval` нормализуется до **`1m`** (в подтверждении отражается фактически используемый интервал).
- В подтверждении возвращается значение `interval`, чтобы клиент мог сопоставить пары `(coin, interval)`.

**Начальный снимок** — **массив** последних баров (закрытые + текущий открытый), от старых к новым — `[]` до первой сделки на рынке:

```json
{ "channel": "candles", "data": [
  { "t": 1735689600000, "T": 1735689659999, "s": "BTC", "i": "1m", "o": "67000.00", "c": "67002.50", "h": "67005.00", "l": "66990.00", "v": "12.5", "q": "837843.75", "n": 8 }
] }
```

Каждый **пуш** — это **один объект бара** (не массив) — текущий открытый бар для данной пары `(coin, interval)`, переиздаваемый при каждом подтверждённом блоке, сделки которого относятся к этому рынку:

```json
{ "channel": "candles", "data": { "t": 1735689600000, "T": 1735689659999, "s": "BTC", "i": "1m", "o": "67000.00", "c": "67002.50", "h": "67005.00", "l": "66990.00", "v": "12.5", "q": "837843.75", "n": 8 } }
```

- `t` / `T` — эпоха открытия / закрытия бара в мс (определяется консенсусом); бар охватывает `[t, T]`, и сделка переходит в новый бар, когда временна́я метка её блока пересекает `T`.
- `s` — монета / символ рынка; `i` — токен временно́го интервала.
- `o` / `c` / `h` / `l` — цены открытия / закрытия / максимума / минимума, строки в **десятичных USDC** (в человекочитаемых долларах, например `"67002.50"`).
- `v` — объём в базовом активе, накопленный в баре (размер в монетах). `q` — объём в котировочной валюте (USD) = `Σ цена × размер` по всем сделкам бара. `n` — количество исполнений в баре.

Серия **непрерывная**: интервал без сделок формирует плоский бар с переносом предыдущей цены закрытия (`o = h = l = c = предыдущее закрытие`, `v = q = 0`, `n = 0`). До первой сделки на рынке бары не публикуются — серия начинается с бара первой сделки.

Хранилище содержит до **1000 баров** на серию `(coin, interval)`; холодные серии (без подписчиков) вытесняются, поэтому неотслеживаемые рынки/интервалы не потребляют ресурсов. На шлюзе через `/hl/ws` (режим совместимости с HL) эквивалентный канал называется HL-вариантом `candle` (единственное число).

### `order_updates`

Жизненный цикл ордеров по аккаунту. Требует `user` (0x-адрес). Каждый пуш — массив записей об обновлениях ордеров по данному аккаунту из только что подтверждённого блока; начальный снимок — `[]`.

```json
{ "method": "subscribe", "subscription": { "type": "order_updates", "user": "0x<address>" } }
```

```json
{ "channel": "order_updates", "data": [ {
  "order": { "coin": "BTC", "side": "B", "limit_px": "100", "sz": "600", "orig_sz": "1000",
             "oid": 42, "cloid": "0x..", "tif": "GTC", "reduce_only": false },
  "status": "open", "filled_sz": null, "avg_px": null, "reason": null, "time": 1735689600123 } ] }
```

- `status` ∈ `open` (рестинг; `sz` — остаток в стакане после коммита) / `filled` (тейкер несёт кумулятивные `filled_sz` + `avg_px`; плечо мейкера отражает `filled_sz` по каждому совпадению, при этом `status` остаётся `open` пока есть оставшийся объём) / `canceled` / `rejected` (+`reason`, `oid` равен null) / `cancel_rejected` (+`reason`).
- `limit_px` / `sz` / `orig_sz` / `avg_px` — десятичные строки в плоскости 1e8; `time` — консенсусное время в мс; неизвестные поля равны `null`.
- **Не** публикуется сегодня: `modify` / `batchModify` / `scheduleCancel` / `cancelAllOrders` / переходы TWAP и отмены, инициированные движком (BOLE T0) — наблюдение за диспатчем для них представляет собой непрозрачный ok/err без полезной нагрузки по ордерам.

### `notifications`

Уведомления о марже и ликвидации по аккаунту, формируемые путём сравнения последовательных подтверждённых состояний. Требует `user`. По одному массиву-фрейму на затронутый коммит; начальный снимок — `[]`.

```json
{ "method": "subscribe", "subscription": { "type": "notifications", "user": "0x<address>" } }
```

```json
{ "channel": "notifications", "data": [
  { "kind": "yellow_card", "tier": "yellow_card", "message": "...", "time": 1735689600123 },
  { "kind": "forced_close_tier", "tier": "partial_market_50", "message": "...", "time": 1735689600123 },
  { "kind": "tier_cleared", "tier": null, "message": "...", "time": 1735689600123 },
  { "kind": "forced_close", "coin": "BTC", "side": "long", "closed_sz": "600", "message": "...", "time": 1735689600123 },
  { "kind": "backstop_residual", "coin": "BTC", "side": "long", "lots": "120", "message": "...", "time": 1735689600123 },
  { "kind": "backstop_residual_cleared", "coin": "BTC", "side": "long", "message": "...", "time": 1735689600123 } ] }
```

- `kind` — машинный тег; `message` — человекочитаемый текст. `tier` ∈ `yellow_card` / `partial_market_50` / `full_market` / `backstop_takeover` (или `null` при сбросе).
- `yellow_card` — однoblochное предупреждение о марже (контракт T0 [многоуровневой ликвидации](../../concepts/tiered-liquidation.md)); `forced_close` срабатывает при фактическом исполнении ликвидации против аккаунта.

### `ledger_updates`

Движение средств по аккаунту с указанием **причины** (считывается из payload подтверждённого блока — запись появляется только при применении действия). Требует `user`; начальный снимок — `[]`.

```json
{ "method": "subscribe", "subscription": { "type": "ledger_updates", "user": "0x<address>" } }
```

```json
{ "channel": "ledger_updates", "data": [ { "kind": "usd_send", "destination": "0x..", "amount": "25.5", "time": 1735689600123 } ] }
```

- `kind` ∈ `usd_send` / `usd_receive`, `spot_send` / `spot_receive` (+`token`), `asset_send` / `asset_receive` (+`asset`, `to_perp`), `withdraw` (`via`: `cctp` | `metabridge`), `deposit` (`amount` может быть `null` для входящего CCTP-кредита), `system_credit`, `sub_account_transfer`, `sub_account_spot_transfer`, `vault_transfer`. Перевод формирует по одной записи для каждой стороны (отправитель + получатель).
- Суммы — строки с целыми токенами в десятичной форме, за исключением `withdraw` через MetaBridge, который содержит `amount_units` (сырые базовые единицы). Суммы входящих bridge-кредитов и действия, задержанные CoreWriter (отправляемые в более позднем блоке), пока не атрибутированы.

### `active_asset_data`

Торговый контекст по паре (аккаунт, монета) — кредитное плечо, режим маржи и текущий потолок максимального размера сделки для одного аккаунта на одном рынке. Требует **одновременно** `user` (0x) и `coin`. Начальный снимок — актуальный контекст (нулевые настройки по умолчанию, если у аккаунта нет позиции), а не пустой массив; пуш переиздаёт его только при изменении контекста.

```json
{ "method": "subscribe", "subscription": { "type": "active_asset_data", "user": "0x<address>", "coin": "BTC" } }
```

```json
{ "channel": "active_asset_data", "data": {
  "address": "0x<addr>", "asset_id": 0, "leverage": 7, "margin_mode": "isolated",
  "max_trade_size": "5000000000", "has_position": true } }
```

- `margin_mode` ∈ `cross` / `isolated` / `strict_iso`; `max_trade_size` — потолок размера, ограниченный открытым интересом (строка в сырых лотах); поля идентичны REST-запросу [`active_asset_data`](../rest/info.md).

На шлюзе через `/hl/ws` (режим совместимости с HL) эквивалентный канал называется HL-вариантом `activeAssetData`, а фрейм транслируется в HL-структуру camelCase:

```json
{ "channel": "activeAssetData", "data": {
  "user": "0x<address>", "coin": "BTC", "leverage": 7,
  "maxTradeSzs": ["5.0", "5.0"], "availableToTrade": ["35000.00", "35000.00"] } }
```

- `user` — 0x-адрес аккаунта; `coin` — символ рынка.
- `maxTradeSzs` — `[buy, sell]`: максимальный торгуемый **размер** для каждой стороны (базовые единицы), в виде десятичных строк.
- `availableToTrade` — `[buy, sell]`: доступный **USD-нотионал** для торговли на каждой стороне, в виде десятичных строк.

### `account_state`

Состояние клирингового дома **PERP** по аккаунту — сводка маржи, открытые позиции и балансы — публикуется при изменении. Требует `user` (0x-адрес; принимается также `address`) — параметр `coin` НЕ нужен. Тело формируется тем же построителем записей, что и сфокусированный REST-запрос по аккаунту, поэтому WS-пуш никогда не расходится с этим запросом. Начальный снимок — актуальное состояние (нули для аккаунта без средств), а не пустой массив.

```json
{ "method": "subscribe", "subscription": { "type": "account_state", "user": "0x<address>" } }
```

```json
{
  "channel": "account_state",
  "data": {
    "address": "0x<addr>",
    "account_value": "10000", "free_collateral": "8500", "maint_margin": "300",
    "init_margin": "1500", "health": "0.97", "tier": 0,
    "mode": "cross", "pm_enabled": false,
    "positions": [
      { "asset": 0, "size": "600", "entry": "62000", "upnl": "441",
        "isolated": false, "lev": 7, "side": "long" }
    ],
    "balances": { "usdc": "10000", "spot": { "MTF": { "total": "12.5", "hold": "0" } } }
  }
}
```

- Скаляры маржи (`account_value` / `free_collateral` / `maint_margin` / `init_margin` / `health`) — десятичные строки в **целых USDC**, идентичные `MarginScalars` REST-запроса по аккаунту; `tier` — индекс уровня ликвидации, `mode` — режим аккаунта по умолчанию, `pm_enabled` — включена ли портфельная маржа.
- `positions[]` — по одной записи на каждую открытую перп-позицию: `asset` (числовой id), `size` (знаковая строка в плоскости 1e8), `entry` / `upnl` (в целых USDC), `isolated`, `lev` и `side` (`long` / `short`, присутствует в режиме хеджирования).
- `balances` — `{usdc, spot}`: `usdc` — залоговая котировочная валюта (в целых USDC); `spot` — маппинг токен → `{total, hold}`.

Частота: по событию изменения — фрейм отправляется только при фактическом изменении состояния аккаунта с момента последнего коммита; если аккаунт не изменился, в этом коммите ничего не публикуется.

:::warning
`account_state` содержит данные уровня аккаунта, однако в настоящее время **аутентификация отсутствует** — любое подключение может подписаться на любой адрес. Не воспринимайте этот канал как приватный до реализации механизма auth-at-subscribe.
:::

### `spot_state`

Состояние клирингового дома **SPOT** по аккаунту — спот-балансы по токенам для одного аккаунта — публикуется при изменении. Требует `user`. Начальный снимок — актуальный набор балансов (`[]` для аккаунта без спот-активов).

```json
{ "method": "subscribe", "subscription": { "type": "spot_state", "user": "0x<address>" } }
```

```json
{
  "channel": "spot_state",
  "data": {
    "address": "0x<addr>",
    "balances": [
      { "asset": 1, "name": "USDC", "total": "2500", "hold": "100" },
      { "asset": 2, "name": "MTF", "total": "12.5", "hold": "0" }
    ]
  }
}
```

- `balances[]` — по одной записи на каждый удерживаемый спот-токен: `asset` (числовой id), `name` (символ токена), `total` (строка в целых токенах), `hold` (сумма, зарезервированная рестинг-ордерами на спот). Идентично REST-запросу спот-балансов.

Частота: по событию изменения — фрейм отправляется только при фактическом изменении спот-балансов с момента последнего коммита; если аккаунт не изменился, в этом коммите ничего не публикуется.

### `web_data2`

**Составной** снимок «всего для фронтенда» по аккаунту — сводка клирингового дома перп, спот-балансы, открытые ордера, капитал хранилищ и глобальный статус биржи — всё в одном фрейме, публикуемом при изменении. Требует `user` (0x-адрес; принимается также `address`) — параметр `coin` НЕ нужен. Тело побайтово идентично составному ответу REST-запроса [`web_data2`](../rest/info.md#web_data2) (используются те же подчитатели), поэтому WS-пуш никогда не расходится с этим запросом. Начальный снимок — актуальный составной снимок (нулевые настройки по умолчанию при отсутствии средств / позиций / ордеров), а не пустой массив.

```json
{ "method": "subscribe", "subscription": { "type": "web_data2", "user": "0x<address>" } }
```

```json
{
  "channel": "web_data2",
  "data": {
    "address": "0x<addr>",
    "clearinghouse": {
      "account_value": "10000",
      "margin_used": "300",
      "positions": [
        { "asset": 0, "size": "600", "entry_ntl": "2500", "mode": "cross", "lev": 10 }
      ]
    },
    "spot_balances": [
      { "asset": 2, "name": "MTF", "total": "12.5", "hold": "0" }
    ],
    "open_orders": [
      { "oid": 42, "market_id": 0, "side": "bid", "px": "6700000000000", "size": "10000000",
        "tif": "gtc", "cloid": "0xab..", "trigger": null, "inserted_at_ms": 1735689600123 }
    ],
    "vault_equities": [
      { "vault_id": 1, "vault_address": "0x<vault>", "shares": "1000", "equity": "1050" }
    ],
    "exchange_status": {
      "spot_disabled": false,
      "post_only_until_time_ms": 0,
      "post_only_until_height": 0,
      "scheduled_freeze_height": null,
      "mip3_enabled": true,
      "frozen": false,
      "replay_complete": true
    }
  }
}
```

Составной снимок содержит именно эти разделы (каждый формируется соответствующим подчитателем, поэтому структуры никогда не расходятся с автономными запросами):

- `address` — каноническое строчное 0x-адрес, на который ключируется фрейм.
- `clearinghouse` — сводка перп-аккаунта: `account_value` (кросс-стоимость аккаунта, десятичная строка в целых USDC), `margin_used` (Σ поддерживающей маржи по активам, в целых USDC) и `positions[]`. Каждая строка позиции — `{asset, size, entry_ntl, mode, lev}`: `asset` — числовой id рынка, `size` — знаковая строка размера в плоскости 1e8 (по одной строке на ненулевое плечо; аккаунт в режиме хеджирования отражает оба плеча), `entry_ntl` — в целых USDC, `mode` ∈ `cross` / `isolated` / `strict_iso`, `lev` — максимальное кредитное плечо позиции. Позиции с нулевым размером опускаются.
- `spot_balances` — массив `balances` из [`spot_state`](#spot_state) / REST `spot_clearinghouse_state`: по одной записи на каждый удерживаемый спот-токен, `{asset, name, total, hold}`.
- `open_orders` — массив `orders` из REST `frontend_open_orders`: по одной записи на каждый рестинг-ордер **и** каждый запаркованный TP/SL / стоп-триггер, `{oid, market_id, side, px, size, tif, cloid, trigger, inserted_at_ms}`. `side` ∈ `bid` / `ask`; `px` / `size` — десятичные строки в плоскости 1e8; `tif` ∈ `alo` / `ioc` / `gtc` (или `trigger` для запаркованного офбук-стопа); `cloid` — клиентский идентификатор ордера или `null`; `trigger` — `null` для обычного книжного ордера, иначе `{trigger_px, trigger_above}` (запаркованные стопы также содержат `is_parked: true`).
- `vault_equities` — массив `equities` из REST `user_vault_equities`: по одной записи на каждое хранилище, в котором у аккаунта есть доли, `{vault_id, vault_address, shares, equity}` (`equity` — в целых USDC, `shares` — строка с целым числом). Пустой массив, если аккаунт не подписан ни на одно хранилище.
- `exchange_status` — глобальные скаляры торгового статуса (то же тело, что и REST `exchange_status`): `{spot_disabled, post_only_until_time_ms, post_only_until_height, scheduled_freeze_height, mip3_enabled, frozen, replay_complete}`. Этот блок идентичен для всех подписчиков в рамках одного коммита.

Частота: по событию изменения — фрейм отправляется только при фактическом изменении составного снимка с момента последнего коммита; коммит, не затронувший составной снимок аккаунта, ничего не публикует.

:::warning
`web_data2` содержит данные уровня аккаунта, однако в настоящее время **аутентификация отсутствует** — любое подключение может подписаться на любой адрес. Не воспринимайте этот канал как приватный до реализации механизма auth-at-subscribe; для аутентифицированных чтений используйте `post` с подписанным действием.
:::

## `post` — запрос/ответ через WS

Не является каналом подписки, но позволяет выполнять разовые запросы на чтение и подписанные операции записи через тот же сокет. `request` — тот же конверт `{type, payload}`, что и для REST-маршрутов; запрос диспатчится через идентичные обработчики (`POST /info`, `POST /exchange`). Полные структуры запроса/ответа и правила подписания описаны в разделе [`post` в WS README](./index.md#post-requestresponse-over-ws).

```json
{ "method": "post", "id": 1, "request": { "type": "info", "payload": { "type": "l2_book", "coin": "BTC" } } }
```

Это поддерживаемый путь для аутентифицированного чтения и отправки подписанных действий через WS сегодня.

## Дорожная карта — пока недоступно

Следующие каналы упоминались в более ранних черновиках, но **не реализованы** на WS-поверхности узла. Они не являются распознаваемыми именами каналов; попытка подписки возвращает ошибку `unknown channel`. Перечислены здесь, чтобы интеграторы не вводились в заблуждение устаревшими заглушками SDK.

- **Публичные рыночные данные:** `meta` (метаданные вселенной), `mark` (маркировочная/оракульная цена), `fundingTicks` (обновления ставки финансирования).
- **Для пользователя (потребует аутентификации):** `vaultEvents`, `rfqEvents`.

Также не реализовано сегодня:

- **Diff-based `l2_book`** (частичные `updates`-фреймы) — текущий `l2_book` всегда отправляет полные тела топ-20. Фрейм содержит флаг `is_snapshot` (`true` для начального снимка, `false` для пушей при изменениях), но каждое тело — полный снимок; частичных `updates`-диффов нет.
- **`seq` / `resume` / токены возобновления** — каждая (пере)подписка начинается со свежего снимка.
- **Конверт auth-at-subscribe** для приватных каналов — используйте `post` с подписанным действием для аутентифицированных операций.

## Порядок и доставка

- **Для каждой подписки** фреймы поступают в порядке коммитов (фрейм публикуется только в тех коммитах, где состояние отслеживаемого канала изменилось). `seq` отсутствует; порядок определяется порядком поступления на единственном сокете.
- **Между подписками** гарантии порядка нет — чередование произвольное. Демультиплексируйте по `channel` и полю `coin` внутри `data`.
- Доставка — **не более одного раза на изменение** и **без буферизации для возобновления**: подписка, отставшая более чем на 256 фреймов, прерывается с фреймом ошибки `lagged` (см. [Обратное давление и отставание](./index.md#backpressure--lag)). Для восстановления переподпишитесь — вы получите свежий снимок.
