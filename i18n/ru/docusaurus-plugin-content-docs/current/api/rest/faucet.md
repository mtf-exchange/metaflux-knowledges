# `POST /faucet` — тестовые средства для devnet/testnet

:::warning
**Только для Devnet / testnet.** Кран бесплатно создаёт залоговые и спот-токены из воздуха. Он **принципиально недоступен в основной сети** (chain id `8964`): маршрут там не регистрируется вовсе. Никогда не полагайтесь на него в продакшн-сценариях.
:::

## Кратко

Один запрос `POST /faucet` выдаёт **3000 USDC** в качестве кросс-залога **и 10 MTF** в виде спот-токенов (token id `104`) на произвольный адрес. **Один раз на адрес — навсегда.** Ответ — `"queued"` — средства зачисляются примерно через ~1 блок (они вводятся как системные действия валидатора, а не фиксируются синхронно). Маршрут доступен как `POST /faucet` на основном шлюзе, рядом со стандартными путями `/info` + `/exchange`.

## URL

```
POST  https://<net>-gateway.mtf.exchange/faucet
```

При самостоятельном запуске ноды тот же маршрут `/faucet` доступен напрямую по адресу
`http://localhost:8080`.

| Где | Смонтирован? |
|-----|--------------|
| Devnet (`31337`) / testnet (`114514`), кран включён | да |
| Mainnet (`8964`) | **нет** — маршрут не регистрируется; случайный запрос получит `403` от защитного обработчика |
| Кран отключён в конфигурации ноды | нет |

Маршрут добавляется в основной API-роутер только тогда, когда конфигурация ноды имеет кран **включённым И не находится в основной сети**. Маршрут имеет собственное состояние обработчика и структурно недостижим из дерева обработчиков `/exchange`.

## Запрос

```json
{ "address": "0x00000000000000000000000000000000000ca11e" }
```

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| `address` | `0x`-hex адрес длиной 20 байт | да | Получатель. Принимается 40 или 42 символа (префикс `0x` необязателен). Нулевой адрес отклоняется. |
| `amount` | uint64 (целые USDC) | нет | Опциональная сумма USDC; **ограничение только вниз** — значение выше настроенного максимума (3000) будет обрезано до 3000, а не увеличено. Значение `0` отклоняется. Количество MTF (10) фиксировано и не зависит от этого параметра. |

```bash
curl -s -X POST https://devnet-gateway.mtf.exchange/faucet \
  -H 'content-type: application/json' \
  -d '{"address":"0x00000000000000000000000000000000000ca11e"}'
```

## Ответ

### `200 OK` — в очереди

```json
{
  "address": "0x00000000000000000000000000000000000ca11e",
  "usdc":    3000,
  "mtf":     10,
  "status":  "queued"
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `address` | `0x`-hex строка | Адрес получателя в нормализованном нижнем регистре |
| `usdc` | uint64 | Выданное количество USDC (целое, с учётом возможного ограничения вниз) |
| `mtf` | uint64 | Выданное количество спот-токенов MTF (целое, фиксировано на уровне 10) |
| `status` | `"queued"` | Средства **поставлены в очередь для следующего блока** и ещё не зафиксированы |

`"queued"` буквально означает: выдача оформляется в виде двух системных действий, вводимых валидатором
(`SystemUserModify{AdjustCrossAccountValue}` для USDC + `SystemSpotSend` для MTF),
которые добавляются в начало следующего предлагаемого блока. Для проверки баланса опросите [`account_state`](./info.md#account_state)
(или [`spot_clearinghouse_state`](./info/spot.md#spot_clearinghouse_state)) примерно через ~1 блок:

```json
// account_state после фиксации зачисления:
{ "account_value": "3000", "balances": { "usdc": "3000", "spot": { "MTF": "10" } }, ... }
```

### Ошибки

| HTTP | Тело ответа | Причина |
|------|-------------|---------|
| 400 | `{"error":"invalid address: <detail>"}` | `address` не является корректным `0x`-hex значением (например, неверная длина) |
| 400 | `{"error":"zero address not allowed"}` | Получателем указан нулевой адрес |
| 400 | `{"error":"amount must be positive"}` | Явно указан `amount` равный `0` |
| 429 | `{"error":"address already funded"}` | Этот адрес уже получал средства (**один раз навсегда**, постоянно в течение жизни ноды) |
| 429 | `{"error":"rate limit: this IP requested too recently"}` | С этого IP-адреса уже был запрос в рамках текущего интервала ограничения (по умолчанию 1/мин/IP) |
| 403 | `{"error":"faucet disabled on this network"}` | Защитный обработчик (в норме недостижимо — в основной сети маршрут не регистрируется) |
| 503 | `{"error":"faucet backlog full; retry shortly"}` | Очередь введения переполнена (временное давление на систему; повторите запрос) |

```json
// повторный запрос для того же адреса:
{ "error": "address already funded" }   // HTTP 429
```

## Ограничения

- **Один раз на адрес — навсегда.** Отслеживается в памяти (сбрасывается при перезапуске ноды;
  devnet является эфемерным). Повторный запрос для того же адреса — даже с другого IP, даже значительно позже — вернёт `429 address already funded`. *Отклонённый* запрос **не** расходует однократный слот.
- **Ограничение по IP.** По умолчанию: 1 запрос / минута / IP-адрес источника. Разные адреса
  с одного IP в рамках окна получат `429 rate limit`.
- **Ограничение USDC.** Опциональный параметр `amount` ограничивает сумму только вниз; получить более
  3000 USDC (настроенный максимум) невозможно.

## Почему кран НЕ работает через `/exchange`

Два зачисления крана — это **системные / привилегированные действия**
(`SystemUserModify`, `SystemSpotSend`) — чеканка залога и спот-токенов из воздуха. Они находятся в диапазоне системных идентификаторов действий и **никогда** не входят в список допустимых пользовательских действий `/exchange`. Кран помещает их в **отдельную очередь введения, доступную только валидатору** (не в публичный мемпул); среда выполнения сбрасывает её в payload блока точно так же, как ленту оракула, используя адрес собственного валидатора в качестве отправителя, чтобы проверка `require_system_authority` пропустила их.
Кода пути из публичного пользовательского мемпула в эту очередь не существует. См.
[never expose system actions on /exchange](./exchange.md#non-bridged-actions).

## Граница детерминизма

Всё в HTTP-слое крана недетерминировано (IP-троттлинг по системным часам,
локальный набор уже получавших средства адресов). В консенсус передаются **только** адрес получателя и суммы в двух системных действиях, которые проходят через неизменённые детерминированные обработчики. Состояние локального ограничения по IP и набора использованных адресов никогда не хешируется в AppHash.
