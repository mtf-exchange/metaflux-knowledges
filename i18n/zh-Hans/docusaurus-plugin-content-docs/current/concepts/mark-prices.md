# 标记价格

:::tip
**稳定版。**
:::

## 摘要

**标记价格**是协议针对每种资产在保证金计算、清算、资金费率和触发条件评估中所使用的权威价格。它由三个分量取中位数得出——预言机锚定价（加上基差 EMA）、内部报价中间价、外部永续合约中位价——并持续滚动更新。它**不是**最后成交价。

## 为什么标记价格 ≠ 最后成交价

用最后成交价做保证金基准存在可操控风险：攻击者只需在操控价位完成一笔小额交易，就能将其他用户推入清算。标记价格是经过平滑处理的多源合成价，难以被单方操控。

## 计算构成

> 已实现的标记价格采用下述**三分量中位数**（`mark_source: "MedianOfOraclesAndMid"`），而非简单的 `median(mid, oracle, ema_mid)`。

### 计算方式

```
mark = median( present components of {C1, C2, C3} )
        # 3 present → median3   2 present → midpoint   1 present → that value
        # a lone C2 (no external anchor) yields no update — see below
```

三个分量定义如下：

| 分量 | 定义 |
|-----------|------------|
| **C1**（预言机锚定） | `C1 = oracle + EMA(quote_mid − oracle)` — 预言机价格加上永续合约**基差**的固定衰减 EMA（在重算频率下半衰期约为 **150 秒**） |
| **C2**（内部盘口） | `mid(best_bid, best_ask)` — 仅取最优买/卖价的中间价。需要双边报价且盘口无交叉，否则为 `None`。刻意**排除最后成交价**（将最后成交价折回会使重算产生自我引用，并可能在一侧挂单稀疏时冻结盘口）。 |
| **C3**（外部永续合约） | `median(external perp mids)` — 取自**5 家永续合约交易所**（Binance、OKX、Bybit、Gate、MEXC）的中间价中位数；至少需要 **≥ 2** 家数据可用，否则为 `None` |

外层中位数对单一异常值具有鲁棒性。**缺失的分量直接丢弃**——有两个分量时取中点，有一个时直接使用该值。若内部盘口和外部永续合约数据均不可用，则 `mark = C1 = oracle + EMA(basis)`，优雅地退化为现货预言机锚定，而非冻结价格。

**仅有 C2 时拒绝更新。** 若唯一存在的分量是内部报价中间价（无预言机、无外部永续合约），标记价格将维持不变，而不是让攻击者可控的单一挂单价差决定清算/资金费率价格。允许仅有 C1（外部预言机）或仅有 C3（本身已是 ≥ 2 家交易所的中位数）的情形。

- **C1 EMA** 使用确定性安全的 `DeterministicEma`（定点数 `(num, denom)`，固定衰减系数 `0.9548`，在约 10 秒的重算频率下对应约 150 秒半衰期）。仅当同时存在可用双边报价和预言机价格时，才将 `quote_mid − oracle` 折入 EMA，因此空盘口或单边盘口不会污染输入。对于从未有过可用报价的市场，`EMA = 0`，即 `C1 = 纯预言机价格`。
- EMA 是**逐资产**计算的（每个永续合约市场各有一个）。
- 计算出的中位数写入市场的**权威标记价格**（所有消费方读取该值），因此普通成交记录的最后成交价无法覆盖它——对活跃市场而言，中位数价格是权威来源，而非仅在市场平静时生效。

> ⚠️ **与旧文档的差异说明。** 早期文档将 C2 描述为 `median(bid, ask, last_trade)`，将 C3 描述为 7 家交易所的加权中位数。实际结构为：C2 = **双边报价中间价**（仅取最优买/卖价，不含最后成交价），C3 = **5 家外部永续合约交易所的中位数**（至少需 2 家），C1 = 预言机价格加上永续合约基差的固定衰减（`0.9548`）EMA。缺失分量直接从中位数计算中剔除（不存在 `unwrap_or(C1)` 的回退），仅有 C2 时不触发更新。

### 两套价格数值平面（阅读任何数值前请先理解这一点）

MTF 使用**两套不同的数值平面**来表示价格——这是造成量纲混淆的首要原因：

| 平面 | 类型 | 量纲 | 使用场景 |
|-------|------|-------|---------|
| **盘口 / 订单 / 标记价格平面** | `FixedPrice`（`i128`）/ `price_e8` | **1e8 定点数**（原始整数 = 价格 × 10⁸） | 订单簿、`last_mark_px`、EVM 预编译（`mark_px_e8`、`entry_px_e8`）、`l2_book` 档位 `px`、订单 `limit_px`、`tick_size` |
| **预言机 / 名义价值 / 抵押品平面** | `rust_decimal::Decimal` | **整数 USDC**（1 单位 = 1 USDC） | 预言机聚合器 + 标记价格计算器（C1/C2/C3 均以 `Decimal` 表示）、资金费率 `oracle_px`、PM 场景引擎、保证金/健康度，以及人类可读接口 `market_info`/`markets` 的 `mark_px`/`oracle_px` 字段 |

标记价格计算器和预言机聚合完全在 **`Decimal` 整数 USDC 平面**中运行。结果在写入盘口 / `last_mark_px` 时转换为 **1e8 `FixedPrice` 平面**。然而，人类可读接口 `market_info` / `markets` 返回的 `mark_px` 和 `oracle_px` 已换算回**整数 USDC 平面**（例如 `"67042.335"`，而非原始 1e8 值）——只有订单/盘口的*提交*字段（`l2_book` 档位 px、订单 `limit_px`、`tick_size`）保持 1e8 表示。PnL/资金费率*结算*使用第三套小惯例——USDC `1e6`（`mark_settle` 中的 `accumulated_funding_e6`）。在比较数值大小之前，务必确认公式所在的数值平面。

## 预言机（C1 锚定）

`oracle`——C1 锚定价——是最多来自 10 家外部现货交易所的**加权中位数**（Binance 3 权重，OKX 2，Bybit 2，Coinbase 2，Bitget / Kraken / KuCoin / Gate / MEXC / MetaFlux 现货各 1，合计权重 15），每区块发布一次并由预言机验证者签名。每个交易对的权重可通过治理调整（`SetOracleWeights`，`ActionId 148`）；超过 60 秒未更新或偏离跨交易所中位数超过 5% 的数据源将被剔除；若有效权重不足 50%，该时段保留上一次有效值。

标记价格的 **C3 分量使用独立的数据集**——来自**5 家永续合约交易所**（Binance、OKX、Bybit、Gate、MEXC）的永续合约中间价，而非现货预言机表。完整的数据来源、可靠性规则、逐交易对覆盖及 `oracle_sources` 查询详见**[预言机价格](./oracle-prices.md)**。

[`market_info`](../api/rest/info.md#market_info) 接口暴露 `mark_source`（描述符 `"MedianOfOraclesAndMid"`）以及合成后的 `mark_px` / `oracle_px`；C1/C2/C3 各分量及加权数据源列表目前不作为独立的接口字段输出。

## 标记价格与预言机价格的差异原因

标记价格与预言机价格出现大幅偏差是**正常且符合预期**的。预言机追踪的是**现货**价格；标记价格追踪的是**永续合约**的实际交易价格——而永续合约相对于现货可以长期存在**基差**（溢价或折价）：

- **C2** 是 MetaFlux 永续合约盘口中间价，**C3** 是外部*永续合约*中位价——两者均反映永续合约价格，而非现货价格。
- **C1** 为 `oracle + EMA(quote_mid − oracle)`——基差 EMA 将预言机锚定价也拉向永续合约的持续溢价，因此三个分量均追踪永续合约价格。

因此，当某永续合约相对于现货指数以（例如）30% 折价交易时，`mark ≈ 永续合约价格` 而 `oracle ≈ 现货价格`，两者合理地相差约 30%。正是**[资金费率](./funding-rates.md)**机制负责弥合这一差距——值得注意的是，若预言机本身不可靠，即使标记价格与预言机价格差距悬殊，资金费率也会*被关闭并衰减至 0*（因此大幅价差伴随约 0 资金费率，意味着该市场的预言机不被信任，而非资金费率机制异常）。详见[资金费率关闭机制](./funding-rates.md#gating-when-the-oracle-is-untrusted)。

## 安全区间

> **实现状态：** 三分量中位数（缺失分量直接剔除、仅有 C2 时拒绝更新）已在生产环境实现。本节描述的逐区块价格约束（在中位数之上叠加 `clamp(candidate, prior ± max_step)` 渐进限制）属于**设计规范，尚未在标记价格计算器中以独立 clamp 实现**——目前的主要操控防御机制是结构性中位数。现有的最接近保护措施是在盘口过时时将预言机价格直接复制到 `last_mark_px`，而非渐进收敛。以下数值为预期区间设计；在依赖精确的 clamp 数值前，请以实时值为准。

即使在多源合成机制下，攻击者仍可能同时推动三个来源中的两个。因此，标记价格设计上将对每个区块执行**区间限制**：

```
prior_mark   = mark at block T-1
band_pct     = market parameter, default 0.5% per second
max_step     = prior_mark * band_pct * (block_time_ms / 1000)

candidate    = median(...)
mark_T       = clamp(candidate, prior_mark - max_step, prior_mark + max_step)
```

| 默认参数 | 值 |
|---------|-------|
| `band_pct` | 每秒 0.5% |
| `block_time_ms` | 100 ms |
| `max_step_per_block` | 约 0.05% |

以每 100 ms 约 0.05% 的步长，5% 的价格变动需约 10 秒才能完全反映。真实的快速行情可在少量区块内追上；恶意价格尖刺则被限制在短暂的渐进过程中。

若候选价格在连续 `band_violation_blocks`（默认 50 = 约 5 秒）个区块内持续超出区间，协议将判定发生真实的行情体制转换，并在一个窗口内将区间扩大为原来的 2 倍。这可防止在真实市场剧烈波动时出现永久冻结。

## 标记价格的使用场景

| 使用方 | 原因 |
|----------|-----------|
| 保证金计算（初始 + 维持） | 基准稳定，抗操控 |
| 清算档位评估 | 同上 |
| 资金费率计算（对比预言机） | 资金费率公式要求 |
| 触发订单（止损 / 止盈） | 抵抗单笔成交价格尖刺 |
| 未实现盈亏展示 | 面向用户的稳定数值 |
| 保险基金核算 | 稳定 |

使用**最后成交价**而非标记价格的场景：
- 盘口成交价（实际以盘口价格成交）
- 挂单方/吃单方手续费（按成交价计算，非标记价格）
- 已实现盈亏（按平仓成交价计算）

## 查询接口

```bash
curl -X POST https://devnet-gateway.mtf.exchange/info \
  -H 'content-type: application/json' \
  -d '{"type":"market_info","asset_id":0}'
```

[`market_info`](../api/rest/info.md#market_info) 接口返回的 `mark_px` 和
`oracle_px` 均在**整数 USDC 平面**（例如 `"67042.335"`），并附带
`mark_source` 描述符：

```json
{
  "type": "market_info",
  "data": {
    "asset_id":    0,
    "mark_source": "MedianOfOraclesAndMid",
    "mark_px":     "67042.335",
    "oracle_px":   "67042.335"
  }
}
```

三个内部分量 `C1`/`C2`/`C3`（预言机+EMA 锚定 / 内部盘口中位数 / 外部永续合约加权中位数）以及区间状态（`Ok` / `Banded` /
`Frozen`）保存在标记价格计算器内部，目前**不**作为 `market_info` 的独立接口字段暴露——仅发布合成后的 `mark_px`。`Banded` 表示本区块候选价格被区间约束；`Frozen` 表示所有来源均失效，协议维持上一标记价格。

专用 `mark` WebSocket 频道列于 [WS 路线图](../api/ws/subscriptions.md#roadmap--not-yet-available)（尚未上线流式推送）；目前请通过轮询 [`market_info`](../api/rest/info.md#market_info) 获取 `mark_px`。

## 边界场景

<details>
<summary>展开边界场景</summary>

- **真实行情在 1 秒内波动 5%。** 区间约束在前 10 个区块内限制标记价格追涨速度；之后体制转换检测触发，区间扩大，标记价格更快追平。总计滞后约 1–2 秒，幅度有限但可控。
- **预言机中断，有效权重缺失 > 50%。** 当某一时刻现货权重有效占比低于 50%，预言机时段**不更新**——保留上一有效值。C1 继续使用最后有效预言机价格及其 EMA，标记价格保持锚定。
- **MTF 盘口为空或仅有单边报价。** C2 剔除。标记价格 = `midpoint(C1, C3)`（若外部永续合约也不可用则为 `C1`）——跟踪预言机锚定与外部永续合约的混合价格。资金费率继续使用可用的预言机价格；清算按预言机锚定的标记价格执行。
- **所有外部永续合约数据不可用。** C3 剔除。标记价格 = `midpoint(C1, C2)` — 内部报价中间价对比预言机锚定。
- **无盘口且无外部永续合约数据。** 仅 C1 存在 ⇒ `mark = C1 = oracle + EMA(basis)`。纯预言机锚定。
- **仅有单边/空盘口且无预言机价格的报价。** 仅有 C2 时拒绝更新 ⇒ 不触发更新；标记价格维持最后值，直至外部锚定（预言机或外部永续合约）恢复。
- **C1 EMA 数据过旧。** EMA 一旦至少有过一次内部中间价折入，便始终有定义；在无新中间价输入期间（仅当 C2 存在时才更新），EMA 保持最后值。
- **冻结期间的触发订单。** 触发评估使用标记价格；冻结期间不触发任何订单。挂单保持挂起状态，直至真实标记价格恢复。

</details>

## 时序示意——区间约束在价格尖刺时的介入

三分量中位数在任何区间约束介入前，已可消除单一来源的价格尖刺：

```
block T-1   C2(book mid)=100.0  C1(oracle+EMA)=100.0  C3(perps)=100.0  →  median3 = 100.0

block T:    thin MTF book; adversary lifts the internal mid to 110.0
            C2 = 110.0,  C1 = 100.05,  C3 = 100.05   (external perps + oracle unmoved)
            mark = median3(100.05, 110.0, 100.05) = 100.05
            ↑ the adversarial C2 is the OUTLIER → median discards it; mark ≈ oracle anchor

block T+1:  adversary persists at 110.0; oracle/perps drift up slowly
            C2 = 110.0,  C1 = 100.10,  C3 = 100.10
            mark = median3(100.10, 110.0, 100.10) = 100.10

... mark tracks the (C1, C3) consensus; a single manipulated source never wins the median
```

这一防御机制是结构性的：要移动中位数，攻击者必须同时操控三个分量中的**至少两个**，而 C1（预言机）和 C3（5 家交易所外部永续合约中位数）恰恰是最难被操控的两个分量。下文描述的可选逐区块安全区间是叠加在此之上的额外约束。

## 常见问题

<details>
<summary>展开常见问题</summary>

**Q：为何不直接使用预言机价格？**
A：纯预言机标记价格会赋予预言机运营者通过操控数据源来触发盘口清算的能力。三分量中位数将信任面分散到多个来源。

**Q：我能查看历史上区间约束的触发记录吗？**
A：一旦 `mark` WebSocket 频道上线（路线图中），以及通过归档索引器接口，将可查询含区间状态的标记价格历史；目前该数据尚未在实时接口中暴露。

**Q：区间约束会导致不公平的清算吗？**
A：区间约束会使标记价格的响应速度慢于底层资产——因此在真实暴跌行情中，与使用最后成交价的交易所相比，您的维持保证金健康度可以多保持约 1 秒。反之亦然（标记价格的回撤也较慢）。综合来看，清算行为更具确定性，也更难被武器化利用。

</details>
