# 自动减仓（ADL）

:::info
**预览功能。** T4 仅在保险池无法完全覆盖 T3 亏损缺口时才会触发——正常运行中极少发生，但一旦触发，机制是完全确定性的。
:::

## 简要说明

当保险池无法吸收清算后的剩余亏损时，协议会按比例从同一标的的盈利对手方扣回，比例基于其未实现盈亏。MetaFlux 的分配机制采用在线学习排名，目标是将**超额扣减**（即超出实际亏损缺口的扣减量）降至最低。

## ADL 的触发时机

[分级清算阶梯](./tiered-liquidation.md)：

```
T0 黄牌警告  →  T1 部分清算  →  T2 全额清算  →  T3 兜底  →  T4 ADL
```

T3 阶段会将待清仓账户的持仓转交给兜底多签账户，并从保险池支付损失。若 `insurance_pool < shortfall`，协议进入 T4：剩余的未兜底亏损以持仓扣减的方式分摊给盈利对手方。

```
shortfall  =  liquidation_loss - insurance_pool_drawn
if shortfall > 0:
    fire_adl(asset, shortfall)
```

保险池的提款本身有速率限制——详见[分级清算](./tiered-liquidation.md#t3-backstop--netting-at-mark)。

## ADL 的计算方式

> **延伸阅读：** "Autodeleveraging as Online Learning"（arXiv:2602.15182）。

MTF **不**使用单一排名分数。ADL 拆分为两个独立子问题：一个在线学习的一维**严重程度**决策（本轮扣减多少），以及一个确定性的**按比例分配**（扣减谁，依据盈亏容量）。

> ⚠️ **与旧版文档的差异说明。** 旧版文档描述了一个单一的在线学习*排名*公式：`score = α·pnl% + β·leverage + γ·age`。这**并非**实际实现的算法。真实控制器是通过投影 OGD 求得严重程度标量 `θ ∈ [0,1]`，再结合容量按比例分配。`α/β/γ` 排名公式已被否决（"二维 OGD——维度爆炸"）。经典的 `pnl% × leverage` 队列是 MTF 所替代的 HL 基准方案，而非 MTF 本身的实现。

### 1. 严重程度——对 θ 的一维在线梯度下降

每轮选取标量 `θ_t ∈ [0,1]`，即本轮亏损缺口中需要扣减的比例：

```
B_t          = θ_t · D_t                                  # 本轮预算
θ_needed_t   = clamp(B̂_needed / D_t, 0, 1)                 # 对实际所需量的事前估计
grad         = D_t · sign(θ_t − θ_needed_t)
θ_{t+1}      = clamp(θ_t − η · grad, 0, 1)                 # 投影 OGD 步骤
```

`D_t` = 本轮亏损缺口，`B̂_needed` = 执行价格估算器对真实所需量的预测。

**步长 η**（`AdlController::current_eta`）：
- 默认为**自适应**模式（论文推论 1）：`η* = sqrt( (1 + 2·P_T^θ) / Σ D_t² )`，每轮根据运行时遥测数据（`path_variation`、`cumulative_squared_deficit`）重新计算。
- 首轮（`Σ D_t² == 0`）回退至治理可调参数 `η₀ = 0.01`（`default_eta`）。
- `Fixed(c)` 模式将 `η` 固定为 `c`（治理熔断开关 / 可复现性用途）。

控制器具有**动态遗憾上界**（命题 1）：

```
Reg_T^dyn  ≤  sqrt( (1 + 2·P_T^θ) · Σ D_t² )
```

通过 `analytical_bound()` 对外暴露；`check_bound(slack)` 在混沌测试中断言经验遗憾 ≤ `slack · bound`（默认 slack 为 4）。

所有小数字段（`θ`、`η`、`path_variation` 等）均使用 `rust_decimal::Decimal`；`sqrt` 为整数牛顿法实现的 `Decimal` 开方（不使用 `f64`）；所有累加器使用 `saturating_*`（否则 `Decimal::from(u128 > MAX)` 会 panic 并导致内核停机）。状态持久化于 BOLE 累加器**槽位 5**。

### 2. 分配——确定性容量按比例分配

给定预算 `B_t`，在盈利对手方集合 `W_t` 中分配（每个对手方的扣减容量 `u_i` = 可扣减的未实现盈亏，类型为 `u128`）：

```
total_u = Σ u_i
x_i     = floor( u_i · B_t / total_u )      capped at u_i        # 128 位乘法后 128 位除法
```

整数除法产生的**余量** `B_t − Σ x_i` 按**账户 ID 升序**逐单位分配给仍有剩余容量的获益方（`BTreeMap` 按键顺序迭代，确保各节点计算结果完全一致）。若 `B_t > total_u`，本轮受容量上限约束，`Σ x_i = total_u`。

该方案替代了已否决的向量镜像梯度下降和整数线性规划（ILP）分配器（ILP 虽最优，但求解器不确定性，无法上链）。

**选择按比例分配的原因**（基于 HL 2025 年 10 月 10 日回放数据）：

| 算法 | 10 月 10 日总目标值（越低越好） |
|-----------|-----------------------------------------|
| HL 生产版本（ROE 启发式） | ~$45M 超额扣减 |
| **MTF 按比例分配** | **$3.40M**（约好 13 倍） |
| 向量镜像梯度下降 | $4.41M |
| 最小最大 ILP（最优，仅限链下） | $106k |

按比例分配的单调性违反率为 **0%**（HL 为 11.4%），排名稳定性 ≈ 1.0（HL 为 0.34）。

### ADL 价格报价（读取侧）

EVM 预编译 `0x0902 adl_pro_rata_price` 允许 Solidity 辅助合约*报价*：对规模为 N 的 ADL，按照各方向优先级遍历队列（多头 ADL：价格从高到低；空头 ADL：价格从低到高），计算出成交的 VWAP 均价——**所有价格均基于 1e8 定点数平面**（`price_e8`、`capacity_e8`）。该预编译仅处理比例分配；严重程度 OGD 存在于核心状态中，而非无状态预编译（严重程度每轮只做一次决策，价格报价则每秒调用多次）。

## "扣减"在机制上的含义

扣减不是持仓转移——对手方的持仓规模**缩小**，其未实现盈亏转化为已实现亏损。待清仓账户的反向持仓同等数量蒸发。

具体举例：假设账户 A 以 100 做多 1 BTC，账户 B 以 100 做空 1 BTC，当前标记价格 = 110。

- A 盈利（未实现盈亏 +10 USDC）。
- B 为待清仓账户；持仓以标记价格 110 结算，但 B 仅有 5 USDC 净资产。亏损缺口 5 USDC。
- 保险池：0（已耗尽）。
- ADL 触发，作用于 A：
  - A 的多头减至 0.5 BTC。
  - A 实现 +5 USDC 盈亏（被扣减的那部分）。
  - A 剩余 0.5 BTC 多头的入场价仍为 100，标记价 110，未实现盈亏 +5 USDC。
  - B 的空头完全平仓。

A 的剩余持仓上的未实现盈亏得以保留；A 仅损失*已平仓*部分的盈亏。

## 通知

ADL 事件通过 [`userEvents` WS 频道](../api/ws/subscriptions.md#userevents)推送：

```json
{
  "kind":        "adl",
  "asset":       0,
  "haircut_sz": "50000000",
  "realised_pnl": "5000000",
  "block":       12345
}
```

同时推送账户级通知：

```json
{ "kind": "marginChange", "free": "...", ... }
```

对于自动化交易机器人，请将 `adl` 事件视同强制成交处理——您的持仓已变化，协议以扣减发生时的标记价格作为成交价，通常需要重新评估策略。

## 预测 ADL 风险敞口

风险监控系统可通过 [`/info`](../api/rest/info.md) 接口获取账户状态，其中包含 `adl_rank_estimate` 字段：

```json
"adl_rank_estimate": {
  "asset": 0,
  "percentile": 95,
  "score": 1.2
}
```

`percentile: 95` 表示您在该标的上处于风险最高的前 5% 账户之列。位于前 10% 的账户在 ADL 触发时面临最高风险敞口。

这只是估算——实际排名在 ADL 触发时刻依据当时的最新状态计算。对于持有大型账簿的做市商而言，主要风险来自集中度（单一大仓主导某标的未平仓量）；将持仓分散到多个标的可有效降低 ADL 风险敞口。

## 边界情况

<details>
<summary>展开边界情况</summary>

- **同一区块内出现多次亏损缺口。** 每次均独立对当时的对手方集合进行分配。两次事件之间排名可能发生变化。
- **对手方集合为空。** 若同一标的上确实不存在任何盈利对手方，该亏损缺口将计入保险池的"未覆盖亏损"登记，待下次池子补充时偿付。流动性好的标的理论上不会出现此情况；长尾 MIP-3 市场理论上可能发生。
- **已加入组合保证金的对手方。** ADL 仍针对同一标的的未实现盈亏——加入组合保证金不改变 ADL 的逐标的粒度。组合保证金情景引擎在下一区块看到扣减后的状态。
- **现货市场。** 现货没有永续合约意义上的未实现盈亏。V1 版本未定义现货 ADL；现货持仓不参与 ADL 排名。

</details>

## 流程——长尾标的上的 ADL

```
block T:   account X liquidates on asset 42 (MIP-3 market), loss = 100 USDC
           insurance pool on asset 42 = 60 USDC
           D_t (deficit) = 40 USDC
           severity controller: θ_t resolves to 1.0 (full deficit needed)
           B_t = θ_t · D_t = 40                  # budget this round
           winners W_t on asset 42 (by haircut capacity u_i):
             A: u_A = 30
             B: u_B = 50    →    total_u = 80
           pro-rata: x_i = floor(u_i · B_t / total_u)
             x_A = floor(30·40/80) = 15
             x_B = floor(50·40/80) = 25          # Σ = 40, no dust
           result:
             A's position haircut by 15 USDC of PnL realised, 15 kept
             B's position haircut by 25 USDC of PnL realised, 25 kept
```

（分配方式为**容量按比例**，而非按分数排名逐级清空：每位获益方放弃相同*比例*的容量——本例为 50%——这正是按比例分配所具备的最小最大公平性。与旧的"按分数排名、从顶层开始逐步清空"模型形成对比——代码实际运行的并非那种方式。）

## 常见问题

<details>
<summary>展开常见问题</summary>

**问：能否退出 ADL？**
答：不能。ADL 是协议级别的亏损分摊机制；退出只会将亏损转嫁给其他人。"最小化超额扣减"目标本身就是对参与者的保护。

**问：为何按盈亏容量进行按比例分配，而不是按分数排名队列？**
答：按比例分配对每位获益方扣减其可扣减盈亏的相同*比例*——具备内置的最小最大公平性，无单调性违反（两个容量相同的账户命运一致），排名稳定性 ≈ 1.0。在 2025 年 10 月 10 日的回放数据中，该方案比 HL 的 ROE 启发式队列优约 13 倍，与链下 ILP 最优解相差约 30%，同时完全确定性且可上链。*严重程度*（总共扣减多少）是在线学习的部分；*由谁承担*则完全采用按比例分配。

**问：ADL 是否遵守严格隔离模式（Strict-Iso）？**
答：是的。ADL 在设计上以标的为粒度；严格隔离模式的持仓仅当持有相同标的时，才会成为 ADL 的对手方候选。

**问：各验证节点的排名结果是否确定性一致？**
答：是的——所有输入（每位获益方的盈亏容量 `u_i`、本轮亏损缺口 `D_t`）均从已提交状态读取，严重程度控制器的状态（`θ`、`path_variation`、`Σ D_t²`、`η`）存储于 BOLE 累加器槽位 5，并折叠进 LtHash，确保每个节点逐字节验证一致。按比例分配使用 128 位整数乘除法，余量处理采用账户 ID 升序——无浮点数，无 `HashMap`。

</details>
