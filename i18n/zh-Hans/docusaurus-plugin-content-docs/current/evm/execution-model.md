# 执行模型

:::tip
**已在 devnet 上线。** 统一区块模型——每个共识轮产生一个区块，并行冲突分层执行——已投入运行且经过测试。节奏和 gas 值在上线前可能仍会调整。[bridge](../bridge/) 已上线。
:::

MetaFlux EVM 在每个共识轮产生**一个统一区块** —— 没有分开的"小"和"大"区块大小。在每个区块内，执行被划分为**并行冲突分层**，因此吞吐量随核心数扩展，**每个交易类别——包括合约部署——都在同一个亚秒级轮次内确认**。

## 一个区块，并行分层

- **每个共识轮产生一个区块**，节奏为亚秒级。没有 60 秒的重块车道：合约部署或一笔大额结算会在下一个区块中立即出现，与交易流一起 —— 而不是等待分钟级的延迟。
- 区块的交易根据其读/写访问集被分组为**分层**。独立的交易**并发**执行；冲突的交易依次执行。总吞吐量是 `每条车道预算 × 并行宽度`，因此它随可用核心数扩展，而不是固定的单区块 gas 层级。
- 分区**是可选的** —— 它只决定什么在并行运行。提交的结果（状态 + 状态根）由与普通按顺序回放相同的确定性执行产生，因此无论核心数或线程调度如何，它在**每个诚实节点上都相同**。

## 区块形成

`ASSEMBLE → PARTITION → EXECUTE → COMMIT`：

1. **Assemble** —— Core → EVM 信用额（现货发送给 EVM 端接收者，bridge 铸币）首先被放置，然后是规范共识顺序中的用户交易。
2. **Partition** —— 交易根据每个节点相同地重新计算的内容衍生规则被分组为冲突分层。
3. **Execute** —— 分层在 Block-STM 执行器下并行运行。被误估的访问集由读集重新验证和重新运行捕获，因此正确性从不依赖分区 —— 只有速度依赖。
4. **Commit** —— 最终化的写入按交易索引顺序提交；状态根从提交的状态中获取。

## Gas 和费用

- **总区块 gas 限制**（反 DoS 上限）加上**单笔交易 gas 上限**。单笔交易上限吸收了旧的重块角色：部署 / `CREATE` **每个区块**都获得高上限；普通交易被限制为较低值。
- 单个 **EIP-1559** 基础费用市场；基础费用被**销毁**。区块 gas 预算是**弹性的** —— 它在持续负载下扩大，在空闲时收缩 —— 具有硬**最小底线**，因此最坏情况容量不会低于固定基准。
- 节奏由共识轮控制；`block.timestamp` 从共识推导（确定性 —— 无墙钟）。

## 抗 MEV 交易（可选，按市场）

市场微观结构是一流关注点，因此 MEV 抗性是**区块构建**的属性，而不是费用市场的事后想法 —— 它按市场可选：

- **频繁批次拍卖 (FBA)** 模式下的市场将其一个轮次的订单意图收集为**一个原子批次**，以**单一统一价格**清算。批次内没有优先级可前置运行，三明治攻击无意义（每个人都得到一个价格），延迟竞速不会移动价格。
- 订单意图可能被**阈值加密** —— 它们的内容对区块提议者隐藏，直到排序已承诺。
- 拍卖无法覆盖的交易获得**可验证公平排序**（从区块自己的父哈希 + 编号衍生的种子），消除提议者排序自由裁量权。
- 市场默认为**连续模式**（与标准 EVM 期望大致兼容），每个市场可选，因此推出是增量且可逆的。

清算在 MetaFlux Core 匹配引擎上运行；EVM 区块**同步**其交易意图流到该拍卖 —— 只有**一个**清算路径，不是在 EVM 内的重复。

## 确认层级

- **最终**（共识）确认每轮 —— 唯一进入提交状态的层级。
- 可选的**软确认**可能暴露用于延迟敏感的 UX；它**不是**共识的一部分。风险承担行动 —— bridge 铸币、提取 —— 仅依赖**最终**确认。
