# 执行模型

:::tip
**已在 Devnet 上线。** 统一区块模型——每个共识轮次产生一个区块，并采用并行冲突分层执行——已经上线并通过测试。具体的出块节奏和 Gas 参数在正式上线前可能还会微调。[跨链桥](../bridge/)已上线。
:::

MetaFlux EVM 在**每个共识轮次产生一个统一区块**——没有"小区块"和"大区块"之分。在每个区块内部，执行被划分为**并行冲突分层**，因此吞吐量可随 CPU 核心数线性扩展，**所有交易类型——包括合约部署——均在同一亚秒级轮次内完成确认**。

## 一个区块，并行分层

- **每个共识轮次一个区块**，出块节奏为亚秒级。系统没有 60 秒重型区块通道：合约部署或大额结算会在下一个区块中与正常交易流一并处理，无需等待一分钟。
- 区块内的交易根据读/写访问集被划分为**分层（strata）**。无冲突的交易**并发执行**；存在冲突的交易按顺序执行。整体吞吐量为 `单通道预算 × 并行宽度`，可随可用核心数扩展，而非受固定的单区块 Gas 上限约束。
- 该分层方案是**建议性的**——它仅决定哪些交易可以并行运行。最终提交的结果（状态 + 状态根）与顺序回放的确定性执行结果完全一致，因此**每个诚实节点得到的结果完全相同**，与核心数或线程调度无关。

## 区块构建流程

`ASSEMBLE → PARTITION → EXECUTE → COMMIT`：

1. **Assemble** — 优先放入 Core→EVM 入账交易（现货转账至 EVM 侧接收方、跨链桥铸币），随后按共识规范顺序排入用户交易。
2. **Partition** — 根据内容派生规则将交易划分为冲突分层，每个节点以相同方式独立重算该规则。
3. **Execute** — 各分层在 Block-STM 执行器下并行运行。若访问集估算有误，读集重验证机制会捕获并重新执行，因此正确性从不依赖分层结果——仅速度受影响。
4. **Commit** — 最终写入按交易索引顺序提交；状态根从已提交状态上计算得出。

## Gas 与费用

- **单区块聚合 Gas 上限**（防 DoS 上限）加**单笔交易 Gas 上限**。单笔上限承担了原重型区块的角色：合约部署 / `CREATE` 在**每个区块**均可获得较高上限；普通交易则上限较低。
- 采用统一的 **EIP-1559** 基础费用市场；基础费用会被**销毁**。区块 Gas 预算具有**弹性**——在持续高负载时扩容，空闲时收缩——并设有硬性**最低下限**，确保最坏情况下的容量不低于固定基准。
- 出块节奏由共识轮次驱动；`block.timestamp` 来源于共识（确定性——不依赖系统时钟）。

## 抗 MEV 交易（按市场可选）

市场微观结构是首要关注点，因此抗 MEV 能力是**区块构建**层面的属性，而非费用市场的事后补丁——且按市场独立启用：

- 处于**频繁批量拍卖（FBA）**模式的市场会将一个轮次内的订单意图汇集成**一个原子批次**，以**单一统一价格**清算。批次内不存在优先级可供抢跑，夹单攻击毫无意义（所有人获得同一价格），延迟竞速也不会影响成交价格。
- 订单意图可采用**门限加密**——其内容在排序已经确定之后才对区块提议者公开。
- 拍卖无法覆盖的交易将获得**可验证的公平排序**（种子由区块自身的父哈希 + 区块号派生），消除提议者对交易排序的自由裁量权。
- 市场默认采用**连续撮合模式**（与标准 EVM 预期广泛兼容），可按市场独立选择加入 FBA 模式，因此推出过程是渐进且可逆的。

清算在 MetaFlux Core 撮合引擎上运行；EVM 区块将其交易意图流**同步**到该拍卖——只有**一条**清算路径，EVM 内部不存在重复流程。

## 确认层级

- **最终确认**（共识级）在每个轮次产生——这是唯一进入已提交状态的层级。
- 可选的**软确认**可用于对延迟敏感的用户体验场景；它**不**属于共识的一部分。涉及资金风险的操作——跨链桥铸币、提款——仅依赖**最终确认**。
