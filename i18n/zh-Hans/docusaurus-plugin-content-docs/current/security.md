# 安全模型

:::tip
**内容稳定。**
:::

协议做出哪些保证、哪些不在承诺范围内，以及哪些风险由你自行承担。

## 摘要

- 协议保证：确定性状态机语义、签名绑定的授权机制、每一笔操作均可在链上审计。
- 协议**不**保证：oracle 组合方式之外的价格正确性、你的私钥安全，以及治理风险的缺失。
- 漏洞赏金计划在第三方平台运行；期望以协调披露方式报告漏洞。

## 信任边界

### 协议负责的部分

| 层级 | 协议保证 |
|-------|--------------------|
| 共识 | M-of-N 验证者达成一致；确定性状态转换；区块带签名 |
| 状态机 | 所有验证者执行结果完全一致；确定性时间；仅使用整数运算 |
| 签名恢复 | 基于 msgpack 的 EIP-712；secp256k1 恢复；代理授权映射 |
| 标记价格 | 组合公式 + 合理区间检查，详见 [标记价格](./concepts/mark-prices.md) |
| 清算 | 分层阶梯对已提交状态确定性触发 |
| 手续费计算 | 手续费层级表 + 销毁比例按每次成交统一应用 |

不遵守上述规则的节点将被视为无效验证者，共识层会拒绝其参与。

### 用户负责的部分

| 层级 | 用户责任 |
|-------|--------------------|
| 私钥保管 | 主账户使用冷存储；代理账户使用热存储；定期轮换 |
| 链下机器人逻辑 | 下什么单、何时补充保证金、何时平仓 |
| 风险管理 | 仓位规模相对于资金桶 / 净值的配置 |
| 跨链桥对手方风险 | 源链钱包与跨链桥路由的选择 |

### 共同承担的部分

| 层级 | 信任假设 |
|-------|------------------|
| Oracle 组合 | 在已记录的组合范围内信任验证者发布的 oracle |
| MetaBridge | 信任 MetaFlux 验证者集体对**所有**跨链转账（含 USDC）进行 ⅔ 权益加权联合签名，并设有提款争议窗口期（与共识使用同一套密钥；无第三方证明服务；详见 [跨链桥](./bridge/)） |
| 治理 | 参数变更受治理控制；信任治理将以协议利益为导向行事 |

核心原则：将信任最小化，而非完全消除。凡是不可避免地需要共享信任之处（oracle、证明服务），均已明确记录并划定边界。

## 威胁模型

### 不在协议处理范围内

- 用户签署了一笔事后后悔的订单。
- 用户热钱包私钥被盗，窃贼用其签署交易（这正是代理账户不具备提款权限的原因）。
- 用户未能及时补充保证金，按文档规定的分层阶梯被清算。
- 用户在 RFQ 报价中以劣势价格成交。
- 用户存入某个亏损的资金库。
- 治理设定的参数在其许可范围内调整，影响了用户仓位。

以上情况不属于安全问题，而是用户自行承担的运营风险。

### 在范围内（请上报以下问题）

- 签名伪造 / 接受无效签名。
- 状态机执行非确定性（两个验证者对已提交状态存在分歧）。
- 有效签名跨网络重放（chainId 域隔离绕过）。
- 权限提升（代理获得提款权限；非主账户触发仅主账户可执行的操作）。
- 在文档规定的清算 / ADL / 手续费机制之外发生资金损失。
- 跨链桥集成缺陷（MetaBridge 联合签名 / ⅔ 法定人数校验、消息 ID 重放、争议窗口期绕过）。
- WebSocket 认证绕过（未经认证订阅私有频道）。
- 阻止合法操作在文档规定速率限制内被接受的 DoS 攻击。
- 已记录的不变量未能成立（例如 nonce 单调性绕过）。

## 披露政策

发现安全漏洞时：

1. **不要**在 GitHub 上公开提 Issue。
2. 发送邮件至 `security@mtf.exchange`（PGP 公钥将在上线前公布），内容包括：
   - 漏洞描述
   - 复现步骤
   - 你对影响范围的评估
   - 后续跟进的联系方式
3. 预计 48 小时内收到确认回复。
4. 协调披露时间线：自确认之日起 90 天，或在修复并部署后提前披露。

分层奖励的漏洞赏金计划在第三方平台运行，详情将于上线前公布。

## 链上可审计性

每一笔操作都永久记录在链上。取证工具可重建以下信息：

- 每个地址的完整操作历史（签名者、action_hash、提交区块）。
- 完整清算历史（账户、层级、标记价格、已实现亏损）。
- 代理授权生命周期（主账户、代理账户、授权 / 过期 / 重新授权事件）。
- 资金库净值曲线与存款人明细表。

区块链浏览器会展示这些数据；索引器以可查询形式提供数据。

## 确定性执行

状态机是一个纯函数：

```
state_{t+1} = apply(state_t, ordered_actions_in_block)
```

对 `state_{t+1}` 存在分歧的验证者即为不合规节点。浮点运算、无序 map 迭代、系统时间等非确定性来源在共识路径中均被禁止。审计工作会显式针对这一特性展开。

如果你的机器人需要预测未来状态值（例如下单后的预期 PM 保证金），在输入相同的前提下，它的计算结果与链上完全一致。wire 规范包含了你所需的一切信息。

## 运维安全建议

面向机构 / 生产环境用户：

| 建议 | 原因 |
|----------------|-----|
| 主账户使用多签 | 单密钥被攻击 = 单点故障 |
| 每台主机 / 每个策略使用独立代理 | 限制被攻击后的波及范围 |
| 缩短代理有效期（≤ 30 天） | 强制定期轮换 |
| 主账户和子账户主账户使用 HSM / 硬件钱包 | 冷存储签名面 |
| 对机器人的出站请求做速率限制 | 防止失控循环耗尽单账户配额 |
| 维护独立的风险监控代理 | 保证金补充与交易逻辑相互独立 |
| 双节点接入 WebSocket 数据流 | 提升延迟稳定性与断线重连韧性 |
| 订阅状态告警 | 运营侧事故会影响你的延迟和可用性 |
| 审计 action_hash → 提交的对账记录 | 捕获静默丢弃 |
| 在测试网提前 60 天测试破坏性变更迁移 | 避免主网上线时出现意外 |

## 链出问题怎么办

共识停止（例如因网络分区导致无法达到法定人数）在运营层面较为罕见，但确实可能发生。停止期间：

- `/info` 继续提供最后一个已提交状态的数据。
- `/exchange` 返回 `503 chain_unavailable`。
- WebSocket 保持心跳连接；恢复前不推送新数据。
- 清算暂停——标记价格停留在最后一个值；停止期间不发生层级迁移。

恢复后，链从最后一个已提交区块开始重放，不会丢失任何状态。时间推进基于共识派生的区块时间，而非挂钟时间；资金费率支付在队列中积累，于恢复后执行。

如果某个节点出现共识停止，请切换至其他节点 / 网关（验证者集合为分布式部署）。协议设计假设 ≥ 2/3 的验证者诚实且在线；低于该阈值的短暂停止属于预期情况。

## 常见问题

<details>
<summary>展开常见问题</summary>

**Q：共识机制是否经过形式化验证？**
A：共识模型已经过形式化规范；形式化验证（TLA+ / Stateright）覆盖安全性与活性不变量。生产实现已针对规范完成审计。

**Q：Oracle 提供者是否可被罚没？**
A：Oracle 数据由验证者集合签名。若某验证者反复发布明显错误的 oracle 数据（多次超出合理区间），可根据[质押](./concepts/staking.md)规则对其进行罚没。

**Q：在已知协议漏洞的情况下，用户最坏可能损失多少？**
A：取决于漏洞本身。架构层面通过子账户隔离、代理提款拦截、单通道跨链桥限额、保险池等机制限制了波及范围，但深层状态机漏洞在理论上可能导致账户资产被耗尽。这正是及时披露和持续审计至关重要的原因。

**Q：协议能回滚状态吗？**
A：不能单方面回滚。回滚需要验证者集合协调决策，并视为硬分叉处理。标准政策：不为个别用户损失回滚；仅在影响共识正确性的全协议级漏洞时才考虑回滚。具体门槛由治理决定。

</details>
