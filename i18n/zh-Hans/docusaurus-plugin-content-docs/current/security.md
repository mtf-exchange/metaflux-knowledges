# 安全模型

:::tip
**稳定。**
:::

协议保证的内容、不保证的内容以及风险由谁承担。

## 简要概览

- 协议保证：确定性状态机语义、签名绑定的授权、每项操作的链上可审计性。
- 协议不保证：预言机的正确性（超出已发布的组合之外）、您的私钥存储安全性或不存在治理风险。
- 漏洞赏金在第三方平台运营；预期进行协调公开。

## 信任表面

### 协议所有者

| 层级 | 协议保证 |
|-------|--------------------|
| 共识 | M-of-N 验证器协议；确定性状态转换；已签署的区块 |
| 状态机 | 验证器间相同的执行；确定性时间；仅整数算术 |
| 签名恢复 | msgpack 上的 EIP-712；secp256k1 恢复；代理批准映射 |
| 标记价格 | 组合公式 + 在 [标记价格](./concepts/mark-prices.md) 中记录的完整性带 |
| 清算 | 分级梯度确定性针对已提交状态触发 |
| 费用数学 | 等级表 + 燃烧比例按相同方式应用于每次成交 |

不遵循这些规则的节点不是有效验证器；共识会拒绝它。

### 用户所有者

| 层级 | 用户责任 |
|-------|--------------------|
| 私钥存储 | 主密钥冷存储；代理热存储；轮换卫生 |
| 链下机器人逻辑 | 下单的内容、何时补充、何时平仓 |
| 风险管理 | 相对于桶 / 权益的头寸大小 |
| 跨链桥接对手方风险 | 源链钱包与跨链路由的选择 |

### 信任共享的位置

| 层级 | 信任假设 |
|-------|------------------|
| 预言机组合 | 信任验证器发布的预言机在已记录的组合内 |
| MetaBridge | 信任 MetaFlux 验证器集的 ⅔ 股权加权共同签署 ALL 跨链转账 — 包括 USDC — 在提现异议窗口后面（相同的密钥作为共识；无第三方认证服务；参见 [跨链桥接](./bridge/)) |
| 治理 | 参数变更受治理控制；信任治理会按照协议利益行动 |

原则：信任最小化，而非消除。 无法避免的共享信任（预言机、认证服务）的信任表面是记录和界定的。

## 威胁模型

### 超出协议范围

- 用户签署一项他们后悔的订单。
- 用户的热密钥被盗，小偷签署交易（这就是为什么代理没有提现权限）。
- 用户未能补充保证金并根据已记录的分级梯度被清算。
- 用户以不利价格接受 RFQ 报价。
- 用户存入损失资金的金库。
- 治理设置的参数在其范围内变更并影响用户的头寸。

这些不是安全问题。 它们是用户承担的操作风险。

### 范围内（报告这些）

- 签名伪造 / 接受无效签名。
- 非确定性状态机执行（两个验证器对已提交状态不同意）。
- 有效签名在网络间的重播（chainId 域隔离绕过）。
- 权限提升（代理获得提现权限；非主账户触发主账户专用操作）。
- 已记录的清算 / ADL / 费用机制之外的资金损失。
- 跨链桥接集成缺陷（MetaBridge 共同签署 / ⅔-法定人数验证、消息 ID 重播、异议窗口绕过）。
- WS 认证绕过（在没有认证的情况下订阅私人渠道）。
- DoS 阻止有效操作在已记录速率限制下被接纳。
- 不成立的已记录不变量（例如 nonce 单调性绕过）。

## 公开披露政策

对于安全漏洞：

1. **不**打开公开 GitHub 问题。
2. 向 `security@mtf.exchange` 发送电子邮件（网站上的 PGP 密钥预发布）：
   - 漏洞的描述
   - 复现步骤
   - 您对影响的评估
   - 您的后续联系方式
3. 预期在 48 小时内收到确认收到的响应。
4. 协调公开时间表：确认后 90 天，或如果已修补 + 部署则更早。

漏洞赏金计划在第三方平台运营具有分级奖励；详情预发布公布。

## 链上可审计性

每项操作都永久在链上。 取证工具可以重建：

- 每个地址的完整操作历史（签署者、action_hash、提交区块）。
- 完整的清算历史（账户、层级、标记、已实现损失）。
- 代理批准生命周期（主账户、代理、批准 / 过期 / 重新批准事件）。
- 金库 NAV 轨迹和存款人表。

浏览器公开这些；索引器以可查询的形式提供数据。

## 确定性执行

状态机是纯函数：

```
state_{t+1} = apply(state_t, ordered_actions_in_block)
```

在 `state_{t+1}` 上不同意的验证器不符合。 非确定性的来源（浮点、无序映射迭代、系统时间）在共识路径中被禁止。 审计明确针对此属性。

如果您的机器人计算未来状态值（例如 订单后的预期 PM 保证金），给定相同的输入，它可以计算得与链计算的完全相同。 线路规范捕获您需要的一切。

## 操作安全建议

对于机构 / 生产用户：

| 建议 | 原因 |
|----------------|-----|
| 对主账户进行多签 | 单密钥泄露 = 单点故障 |
| 每个主机 / 策略一个代理 | 泄露影响范围受限 |
| 紧凑的代理过期（≤ 30 天） | 强制轮换节奏 |
| HSM / 硬件钱包用于主账户和子账户主账户 | 冷存储签署表面 |
| 速率限制您自己的机器人出站 | 防止失控循环耗尽每个账户预算 |
| 维护单独的风险监视代理 | 保证金补充独立于交易逻辑 |
| 为 WS 提要运行双节点 | 延迟 / 重新连接恢复力 |
| 订阅状态警报 | 运营商端事件影响您的延迟 / 可用性 |
| 审计您的 action_hash → 提交协调 | 捕获无声丢弃 |
| 在主网前 60 天针对测试网测试破坏性变更迁移 | 避免主网当天的惊喜 |

## 如果链出错了

共识停止（例如 阻止法定人数的分区）在操作上很少但可能。 在停止期间：

- `/info` 继续从最后提交的状态提供。
- `/exchange` 拒绝并提示 `503 chain_unavailable`。
- WS 保持活力；恢复前无新的推送。
- 清算停止 — 标记保持在最后值；停止期间无层级转换。

恢复后，链从最后提交的区块重播。 不会丢失任何状态。 时间根据共识派生的区块时间而不是壁时钟推进；资金支付排队并在恢复时执行。

如果节点看到共识停止，切换到另一个节点 / 网关（验证器集是分布式的）。 协议的设计假设 ≥ 2/3 的验证器是诚实的且在线的；瞬间停止 < 该阈值的中断是预期的。

## 常见问题

<details>
<summary>显示常见问题</summary>

**问：共识是正式验证吗？**
答：共识模型是正式指定的；正式验证（TLA+ / Stateright）涵盖安全性 + 活性不变量。 生产实现根据规格进行审计。

**问：预言机是否可以被削减？**
答：预言机数据由验证器集签署。 验证器发布明显错误的预言机数据（在完整性带外反复发布）在 [质押](./concepts/staking.md) 规则下是可削减的。

**问：用户在已知协议错误的情况下的最坏损失是什么？**
答：取决于错误。 该架构限制了影响范围 — 子账户隔离、代理提现阻止、每条走廊跨链桥接上限、保险池 — 但深度状态机错误原则上可能会耗尽账户。 这就是为什么公开披露很重要并且审计是持续的。

**问：协议可以回滚状态吗？**
答：不是单边的。 回滚需要协调的验证器集决定并被视为硬分叉。 标准政策：决不为个别用户损失回滚；仅回滚损害共识正确性的协议范围错误。 确切的阈值是治理。

</details>
