# المزادات الدفعية المتكررة (FBA)

:::info
**معاينة.** الاشتراك يتم على مستوى كل سوق عبر [MIP-3](../mip/mip-3.md)؛ ليست جميع الأسواق تعمل بنظام FBA.
:::

## ملخص سريع

يستبدل نظام FBA المطابقة المستمرة بدفعات مزاد منفصلة تتكرر كل `batch_interval_ms`. تُصفَّى الأوامر المُصطفَّة ضمن الدفعة في آنٍ واحد بسعر تسوية موحَّد واحد. هذا يُبطل مُستخرَج قيمة الحفازة (MEV) القائم على الكمون الزمني: لا فائدة من التقدم بميكروثانية واحدة.

## المطابقة المستمرة مقابل الدفعية

| الخاصية | CLOB المستمر | FBA |
|---------|-------------|-----|
| دورة المطابقة | عند وصول كل أمر | كل `batch_interval_ms` |
| اكتشاف السعر | لكل صفقة | لكل دفعة (سعر تسوية واحد) |
| قيمة الكمون الزمني | عالية (الأول في الوصول يكسب التعادلات عند السعر) | صفر داخل الدفعة |
| الفائض من الكمون | يُحصِّله المتداولون عالي التردد (HFT) | يُعاد للمشاركين عبر السعر الموحَّد |
| رؤية الأوامر للعموم | قبل الصفقة (دفتر الأوامر النشط) | قبل الدفعة (قائمة انتظار مرئية) |

## آلية العمل

```
batch t:        accept orders during [t, t + batch_interval_ms)
batch close t:  freeze the queue
                compute clearing price p*:
                  p* = price at which |aggregated buy demand| = |aggregated sell demand|
                fill all crossing orders at p*
                roll any non-crossing orders into batch t+1 (or cancel per TIF)
batch t+1:      open
```

قواعد التسوية:
- تُملأ جميع أوامر الشراء بسعر ≥ p* عند p*.
- تُملأ جميع أوامر البيع بسعر ≤ p* عند p*.
- يُعظِّم سعر التسوية الوحيد p* إجمالي الحجم المصفَّى (ما يُعادل السير على منحنيَي العرض والطلب حتى نقطة التقاطع).

سعر التنفيذ **موحَّد** لجميع المشاركين في الدفعة — لا أحد يُملأ بسعر أسوأ بسبب وصوله متأخراً.

## متى يُستخدَم FBA

| فئة الأصل | الافتراضي | السبب |
|-----------|-----------|-------|
| العقود الدائمة الرئيسية (BTC، ETH) | CLOB المستمر | سيولة عالية؛ ميزة الكمون ضئيلة نسبياً للفارق بين العرض والطلب |
| الإدراجات ذات الذيل الطويل (MIP-3) | FBA اختياري | دفتر ضحل؛ ضرر HFT يفوق توفير السيولة |
| أزواج السبوت | CLOB المستمر | الممارسة المعتادة |
| المنتجات المركَّبة / المهيكَلة | FBA | التسعير المركَّب يحتاج تسوية متزامنة |

وضع المطابقة لكل سوق موجود في [`market_info.fba_enabled`](../api/rest/info.md#market_info). الأسواق التي تعمل بـ FBA تقبل كلاً من `FbaOrder` (الدفعة المستهدفة) و[`submit_order`](../api/rest/exchange.md#submit_order) (يُعامَل كأمر FBA للدفعة التالية). انظر [كتالوج إجراءات `/exchange`](../api/rest/exchange.md#action-catalog) — `FbaOrder` هو نقيصة معروفة غير مُعيَّنة حالياً.

## فترة الدفعة

الافتراضي: ثانية واحدة (10 كتل بزمن كتلة 100 مللي ثانية). تُضبَط بالحوكمة لكل سوق في `market_info.fba_batch_interval_ms`. النطاق الاعتيادي: 100 مللي ثانية – 5 ثوانٍ.

الفترات الأقصر تُقلِّص وقت الانتظار لكنها تزيد التكلفة الحسابية. يُحقق الافتراضي ثانية واحدة التوازن بين تحييد HFT وتجربة المستخدم.

## شكل الأمر

```json
{
  "type": "FbaOrder",
  "params": {
    "asset":     42,
    "side":      "Buy",
    "px":  "10050000000",
    "size":   "100000000",
    "batch_id":  9876,
    "cloid":     "0x..."
  }
}
```

يُحدِّد `batch_id` الدفعة التي ينضم إليها الأمر. معرف الدفعة الحالية موجود في [`market_info`](../api/rest/info.md#market_info) تحت `fba_current_batch_id`. الأوامر ذات `batch_id < current` تُرفَض (`{"error":"batch already closed"}`); الأوامر ذات `batch_id` > current تُصطَف لتلك الدفعة المستقبلية.

احذف `batch_id` لاستهداف الدفعة التالية — يختار الخادم الدفعة التي تقبل الأوامر حالياً.

## مثال تطبيقي

تحتوي الدفعة t على الأوامر التالية للأصل 42:

```
buys:
  bob:    5 @ 100.10
  alice:  3 @ 100.05
  carol:  2 @ 100.00

sells:
  dave:   3 @ 99.95
  eve:    4 @ 100.00
  frank:  2 @ 100.05
```

المشي على الطلب (الحجم التراكمي عند كل سعر ≥ المرشَّح):

```
buy-side  cumulative at price ≥ p:
  100.10:  5
  100.05:  5+3 = 8
  100.00:  8+2 = 10
  99.95:   10  (none here)
```

المشي على العرض (الحجم التراكمي عند سعر ≤ المرشَّح):

```
sell-side cumulative at price ≤ p:
  99.95:   3
  100.00:  3+4 = 7
  100.05:  7+2 = 9
  100.10:  9   (none here)
```

نقطة التقاطع: عند p = 100.00، الحجم التراكمي للشراء = 10، والحجم التراكمي للبيع = 7. عند p = 100.05، الحجم التراكمي للشراء = 8، وللبيع = 9. التقاطع بين 100.00 و100.05.

قاعدة التسوية تُعظِّم الحجم:

| p | min(buy, sell) |
|---|----------------|
| 99.95  | min(10, 3) = 3 |
| 100.00 | min(10, 7) = 7 |
| 100.05 | min(8, 9)  = 8 |
| 100.10 | min(5, 9)  = 5 |

أقصى حجم مصفَّى هو 8 عند `p* = 100.05`. إذن:

- تُملأ جميع أوامر الشراء ≥ 100.05: bob (5) + alice (3) = 8 BTC مشتراة بـ 100.05.
- تُملأ جميع أوامر البيع ≤ 100.05: dave (3) + eve (4) + frank (2) = 9 BTC معروضة. توزيع تناسبي: 8/9 = 88.9% من كل منهم ← dave 2.67، eve 3.56، frank 1.78.
- Carol (شراء 2 @ 100.00) لا تُملأ — تنتقل إلى t+1 أو تنتهي صلاحيتها وفق TIF.

جميع الرابحين يُملأون بـ 100.05. Bob لا يحصل على سعر أسوأ لكونه "أسبق" — لا وجود للأسبقية في FBA.

## العدالة في التسوية

حين يفوق العرض الطلب عند p*، يُملأ الجانب الأكبر بصورة **تناسبية** — كل بائعٍ يحصل على نفس الكسر. لا FIFO ولا أولوية سعرية بين الجانب الزائد (الجميع عند p* أو أفضل منه بالفعل).

هذه هي خاصية عدالة FBA: عند سعر التسوية، لا يحصل أي مشارك على صفقة أفضل من غيره.

## الحالات الطرفية

<details>
<summary>عرض الحالات الطرفية</summary>

- **دفعة فارغة.** لا أوامر → لا حدث تسوية. تبدأ الدفعة التالية فوراً.
- **دفعة أحادية الجانب.** شراء فقط أو بيع فقط. لا تسوية — جميع الأوامر تنتقل للدفعة التالية (Gtc) أو تُلغى (Ioc).
- **تعادل في سعر التسوية.** حين يُعظِّم سعران الحجمَ معاً، يختار البروتوكول السعر الأقرب للعلامة السابقة (للحد من غموض خطوة العلامة).
- **أوامر السوق في FBA.** تُرسَل كـ IOC بسعر متطرف؛ تشارك في الدفعة وتُملأ عند p* إن كانت متقاطعة.
- **Reduce-only في FBA.** يُتحقَّق منها عند إغلاق الدفعة، مقارنةً بحالة ما بعد التنفيذ للتعبئات السابقة ضمن الدفعة ذاتها. يُسوَّى ذرياً.

</details>

## التسلسل الزمني

```
t=0.0s   batch_id = 9876 opens
t=0.2s   bob:    FbaOrder buy 5 @ 100.10, batch 9876
t=0.4s   alice:  FbaOrder buy 3 @ 100.05, batch 9876
t=0.5s   carol:  FbaOrder buy 2 @ 100.00, batch 9876
t=0.6s   dave:   FbaOrder sell 3 @ 99.95, batch 9876
t=0.7s   eve:    FbaOrder sell 4 @ 100.00, batch 9876
t=0.8s   frank:  FbaOrder sell 2 @ 100.05, batch 9876
t=1.0s   batch_id 9876 closes; clearing fires
         p* = 100.05; 8 BTC clears
         fills published on `trades` WS with batch_id and "kind":"fba"
t=1.0s   batch_id = 9877 opens
```

## الاستعلام

تُعرَض حوضة FBA الحية والتسوية الاستشارية على مسار القراءة `/info` للعقدة عبر [`fba_batch_state`](../api/rest/info.md#fba_batch_state) — راجع ذلك الإدخال للاطلاع على شكل الاستجابة الكاملة وجدول الحقول. يقبل `market_id` (u32). FBA اشتراك اختياري لكل سوق، لذا فإن السوق غير المسجَّل **ليس 404** — يُعيد 200 بحقول مُصفَّرة (`enabled:false`، `orders` فارغة، `indicative:null`).

```bash
curl -X POST https://devnet-gateway.mtf.exchange/info \
  -H 'content-type: application/json' \
  -d '{"type":"fba_batch_state","market_id":42}'
```

```json
{
  "type": "fba_batch_state",
  "data": {
    "market_id":      42,
    "enabled":        true,
    "period_ms":      1000,
    "min_lot":        "1",
    "last_settle_ms": 1735689600000,
    "next_settle_ms": 1735689601000,
    "order_count":    11,
    "bid_count":      5,
    "ask_count":      6,
    "bid_size":       "1000000000",
    "ask_size":       "900000000",
    "orders":         [ /* {oid, owner, side, price, size, stp_group, submitted_at_ms} */ ],
    "indicative":     { "clearing_px": "10050000000", "matched_size": "800000000" }
  }
}
```

الأسعار والأحجام هي سلاسل أعداد صحيحة بنقطة ثابتة **1e8** خام (مستوى الدفتر/الأوامر). `next_settle_ms` هو قيمة **مُشتقَّة** كـ `last_settle_ms + period_ms`. كتلة `indicative` هي السعر الموحَّد المُعظِّم للحجم والحجم المطابق الذي ستُسوِّيه الدفعة **التالية** لو أُغلقت النافذة الحالية — تُحسَب للقراءة فقط ولم تُسوَّى بعد — وتكون `null` حين لا يوجد تقاطع (نافذة أحادية الجانب أو فارغة). هذا ما سيكون عليه p\* لو أُغلقت الدفعة الآن، وهو مفيد للمتداولين لتقرير ما إذا كانوا سيضيفون للدفعة.

## الأسئلة الشائعة

<details>
<summary>عرض الأسئلة الشائعة</summary>

**س: ألا يتخلى FBA عن اكتشاف السعر؟**
ج: لا — داخل الدفعة يظل البروتوكول يكتشف `p*` من أوامر المشاركين. الاكتشاف يحدث بدورية ثابتة بدلاً من الاستمرارية.

**س: لماذا دفعة بثانية واحدة بدلاً من 100 مللي ثانية؟**
ج: 100 مللي ثانية ضيقة جداً لتحييد الكمون بأي معنى جوهري — حتى في 100 مللي ثانية، الأجهزة الأسرع تستطيع إعادة الإرسال. ثانية واحدة تمنح هامشاً كافياً بحيث يهيمن كمون الشبكة الفيزيائية على كمون ما داخل الدفعة، مما يُزيل ميزة HFT.

**س: هل يمكن أن تتعايش أسواق FBA مع أسواق CLOB؟**
ج: نعم — كل سوق إما FBA أو CLOB باستقلالية. يمكن للحساب الواحد الاحتفاظ بمراكز في كليهما في آنٍ واحد.

**س: هل يُقلل FBA من تكلفة الغاز/الحوسبة للمطابقة؟**
ج: تقريباً. المطابقة المستمرة تؤدي O(1) عملاً لكل وصول؛ بينما يؤدي FBA O(N log N) عند إغلاق الدفعة. لـ N أمر لكل دفعة، FBA مقارِب في الكلفة مع ميزة كونه أكثر قابلية للتوقع من حيث تكلفة كل كتلة.

</details>
