# قنوات الاشتراك عبر WS

:::info
**الحالة.** القنوات `l2_book` و`bbo` و`trades` و`active_asset_ctx` و`all_mids` و`fills` و`user_events` و`candles` و`order_updates` و`notifications` و`ledger_updates` و`active_asset_data` و`user_fundings` و`user_twap_slice_fills` و`user_twap_history` و`account_state` و`spot_state` و`web_data2` جميعها مفعّلة وتدفع بيانات حقيقية مُثبَّتة — مدفوعةً بالتغيير؛ إذ لا تُصدر القناة إطاراً إلا عند تغيُّر حالتها فعلياً منذ آخر إيداع. أما ما عدا ذلك مما هو مُدرج في [خارطة الطريق](#roadmap--not-yet-available) فغير مُوصَّل بعد. يمكن الاطلاع على دورة حياة الاتصال وتنسيق الإطارات في [WS README](./index.md). تتطلب القنوات المرتبطة بالسوق (`l2_book` و`bbo` و`trades` و`active_asset_ctx`) تحديد `coin`؛ وتتطلب `candles` تحديد `coin` **و**`interval` معاً؛ وتتطلب القنوات المرتبطة بالحساب (`fills` و`user_events`) تحديد `user` (عنوان 0x)؛ وتتطلب `active_asset_data` تحديد `user` **و**`coin` معاً؛ في حين لا تحتاج `all_mids` إلى أيٍّ منهما.
:::

:::info
**أسماء القنوات بصيغة snake_case (النمط الأصلي لـ MTF).** هذه هي الواجهة الأصلية للعقدة عبر `/ws`، لذا تأتي أسماء القنوات بصيغة snake_case (مثل `l2_book` و`user_events` و…). أما العملاء الراغبون في استخدام أسماء القنوات بصيغة camelCase الخاصة بـ HL (كـ `l2Book` و`userEvents` و`userFills` و`candle` و…)، فيتصلون بـ **`/hl/ws`** في البوابة (وضع التوافق مع HL)، الذي يُترجم هذه الأسماء داخلياً إلى قنوات snake_case الأصلية. وفق توجيه البوابة الموحدة: `<net>-gateway.mtf.exchange/ws` = snake_case الأصلية، و`/hl/ws` = camelCase الخاصة بـ HL.
:::

يعكس بروتوكول الإطارات ما تستخدمه HL، فيما **تكون أسماء القنوات بصيغة snake_case الأصلية لـ MTF**. يتم الاشتراك عبر:

```json
{ "method": "subscribe", "subscription": { "type": "<channel>", "coin": "<coin>" } }
```

ثم تستقبل رسالة تأكيد (`subscriptionResponse`)، ثم لقطةً ابتدائية (`is_snapshot: true`)، يتبعها دفق مباشر مدفوع بالتغيير `{"channel":...,"data":...}` (`is_snapshot: false`). لا يُرسَل الإطار إلا عند تغيُّر حالة تلك القناة فعلياً منذ آخر إيداع؛ فالقناة التي لم تتغير لا تُصدر شيئاً. يُعدّ `coin` **إلزامياً** للقنوات المرتبطة بالسوق (`l2_book` و`bbo`)؛ راجع [معامل Coin](./index.md#coin-parameter) لمعرفة كيفية توحيد قيمته (معرّف الأصل الرقمي أو الرمز ← مفتاح معرّف الأصل).

## نظرة عامة على حالة القنوات

| القناة | الحالة | المفتاح | المصدر المباشر |
|---------|--------|:-------:|-------------|
| `l2_book` | **مفعّلة** | `coin` (إلزامي) | دفتر الأوامر المُثبَّت، عند التغيير |
| `bbo` | **مفعّلة** | `coin` (إلزامي) | دفتر الأوامر المُثبَّت، عند التغيير |
| `trades` | **مفعّلة** | `coin` (إلزامي) | تنفيذات الكتلة المُثبَّتة، عند وجود تنفيذات جديدة |
| `active_asset_ctx` | **مفعّلة** | `coin` (إلزامي) | سعر Mark/Oracle وتمويل والفائدة المفتوحة لكل سوق، عند التغيير |
| `all_mids` | **مفعّلة** | لا شيء | سعر Mark لكل سوق، عند التغيير |
| `fills` | **مفعّلة** | `user`/`address` (إلزامي) | تنفيذات الكتلة المُثبَّتة لذلك الحساب |
| `user_events` | **مفعّلة** | `user`/`address` (إلزامي) | تنفيذات الكتلة المُثبَّتة لذلك الحساب (مزيد من أنواع الأحداث قادم) |
| `candles` | **مفعّلة** | `coin` + `interval` (كلاهما إلزامي) | تنفيذات الكتلة المُثبَّتة مُدمَجة في أشرطة OHLCV، عند التغيير |
| `order_updates` | **مفعّلة** | `user`/`address` (إلزامي) | دورة حياة الأوامر لكل حساب (وضع / تنفيذ / إلغاء / رفض)، عند التغيير |
| `notifications` | **مفعّلة** | `user`/`address` (إلزامي) | إشعارات الهامش والتصفية لكل حساب، عند التغيير |
| `ledger_updates` | **مفعّلة** | `user`/`address` (إلزامي) | حركة الأموال لكل حساب (إيداع / سحب / تحويل)، عند التغيير |
| `active_asset_data` | **مفعّلة** | `user` **و**`coin` (كلاهما إلزامي) | سياق الرافعة المالية ووضع الهامش والحجم الأقصى للصفقة لكل (مستخدم، عملة)، عند التغيير |
| `user_fundings` | **مفعّلة** | `user`/`address` (إلزامي) | مدفوعات التمويل المحققة لكل حساب، عند التغيير |
| `user_twap_slice_fills` | **مفعّلة** | `user`/`address` (إلزامي) | تنفيذات شرائح TWAP لكل حساب (`{fill, twapId}`)، عند التغيير |
| `user_twap_history` | **مفعّلة** | `user`/`address` (إلزامي) | دورة حياة TWAP لكل حساب (`{time, state, status}`: تفعيل / اكتمال / إنهاء)، عند التغيير |
| `account_state` | **مفعّلة** | `user`/`address` (إلزامي) | حالة غرفة المقاصة للعقود الدائمة لكل حساب — مقاييس الهامش والمراكز والأرصدة — عند التغيير |
| `spot_state` | **مفعّلة** | `user`/`address` (إلزامي) | حالة غرفة المقاصة الفورية لكل حساب — أرصدة الرموز — عند التغيير |
| `web_data2` | **مفعّلة** | `user`/`address` (إلزامي) | لقطة مركّبة لواجهة المستخدم لكل حساب — غرفة المقاصة + أرصدة السوق الفوري + الأوامر المفتوحة + أحقاق الخزينة + حالة البورصة — عند التغيير |

الاشتراك في أي `type` آخر يُعيد `{"channel":"error","data":{"error":"unknown channel: <name>"}}`.

## القنوات المفعّلة

### `l2_book`

دفتر الأوامر المُجمَّع من المستوى الثاني لسوق واحدة. **يتطلب `coin`.**

```json
{ "method": "subscribe", "subscription": { "type": "l2_book", "coin": "BTC" } }
```

تتشارك اللقطة الابتدائية وكل إطار لاحق هذا الشكل:

```json
{
  "channel": "l2_book",
  "data": {
    "coin": "BTC",
    "levels": [
      [ { "px": "10050000000", "sz": "12", "n": 2 }, { "px": "10049000000", "sz": "3", "n": 1 } ],
      [ { "px": "10051000000", "sz": "4", "n": 1 }, { "px": "10052000000", "sz": "6", "n": 1 } ]
    ],
    "time": 1735689600000
  }
}
```

- `levels` هو `[bids, asks]`. تُرتَّب أوامر الشراء من الأفضل (الأعلى) إلى الأدنى؛ وأوامر البيع من الأفضل (الأدنى) إلى الأعلى.
- كل مستوى هو `{ px, sz, n }`: حيث `px` / `sz` هما مقادير نقطة ثابتة خام بصيغة سلاسل **عشرية** (يُطبَّق ضبط مقياس التدرج لكل أصل لاحقاً في البوابة)، و`n` هو عدد الأوامر الراسية عند ذلك السعر.
- يُحدَّد كل جانب بحد أقصى **20 مستوى مُجمَّع**.
- `time` هو `last_trade_ms` للدفتر (مُشتَق من الإجماع)؛ القيمة `0` حتى يحدث أول تداول في الدفتر.

كل إطار هو **لقطة كاملة لأفضل 20 مستوى**، وليس فارقاً جزئياً. يحمل غلاف الإطار قيمة منطقية `is_snapshot` — `true` للقطة الأولية عند الاشتراك، و`false` للإطارات اللاحقة المدفوعة بالتغيير — غير أن **المحتوى يمثّل دفتر الـ 20 مستوى الكاملة في الحالتين**، لذا فالحقل إعلامي فحسب: استبدل نسخة دفتر الأوامر المحلية لديك في كل إطار للبقاء محدَّثاً.

التكرار: مدفوع بالتغيير — يُرسَل الإطار فقط عند تغيُّر الدفتر فعلياً منذ آخر إيداع؛ الإيداع الذي لا يمسّ هذا الدفتر لا يُصدر شيئاً. إذا لم تتوافق العملة مع أي سوق معروف، ستستقبل رسالة التأكيد مع لقطة دفتر فارغة (`"levels": [[], []]`، `"time": 0`) دون أي إطارات لاحقة.

### `bbo`

أفضل سعر شراء/بيع لسوق واحدة — نسخة مختصرة من `l2_book`. **يتطلب `coin`.**

```json
{ "method": "subscribe", "subscription": { "type": "bbo", "coin": "BTC" } }
```

```json
{
  "channel": "bbo",
  "data": {
    "coin": "BTC",
    "time": 1735689600000,
    "bbo": [
      { "px": "10050000000", "sz": "12", "n": 2 },
      { "px": "10051000000", "sz": "4", "n": 1 }
    ]
  }
}
```

- `bbo` هو `[best_bid, best_ask]`. كل إدخال هو مستوى `{ px, sz, n }`، أو `null` إذا كان ذلك الجانب فارغاً.
- `time` هو `last_trade_ms`، بنفس طريقة `l2_book`.

التكرار: مدفوع بالتغيير — يُرسَل الإطار فقط عند تغيُّر أفضل سعر شراء/بيع فعلياً منذ آخر إيداع؛ الدفتر غير المتغير لا يُصدر شيئاً في هذا الإيداع.

---

### `trades`

شريط التداول العام لسوق واحدة — سجل واحد لكل تنفيذ في تلك السوق، يُصدَر في الكتل التي جرى فيها تداول فعلي. `px`/`sz` هما سلاسل صحيحة بمستوى **1e8** خام؛ `side` هو الجانب الآخذ (`"B"` شراء / `"A"` بيع)؛ `time` هو طابع زمني للكتلة بالإجماع (بالمللي ثانية)؛ `tid` هو معرّف تداول حتمي؛ `users` هو `[taker, maker]` (الآخذ أولاً، وهو الطرف المبادر).

```json
{ "method": "subscribe", "subscription": { "type": "trades", "coin": "BTC" } }
```

```json
{ "channel": "trades", "data": { "coin": "BTC", "side": "B", "px": "6700000000000", "sz": "10000000", "time": 1735689600123, "tid": 1234567890, "users": ["0x..taker", "0x..maker"] } }
```

### `active_asset_ctx`

السياق الخاص بسوق واحدة — سعر Mark / Oracle والتمويل والفائدة المفتوحة — يُدفع عند تغيُّره. **يتطلب `coin`.** يحمل المحتوى الحقول والوحدات ذاتها التي يُعيدها الطلب REST [`market_info`](../rest/info.md#market_info): `mark_px` / `oracle_px` بوحدة **USDC كاملة** مُقرَّبة بمقياس السعر (مقطوعة إلى تدرج السعر للسوق)، وكتلة `funding` تعكس `market_info.funding`. مبنيٌّ على منشئ السجل ذاته المستخدم في قراءة REST، لذلا يختلف إطار WS ctx أبداً عن `market_info`.

```json
{ "method": "subscribe", "subscription": { "type": "active_asset_ctx", "coin": "BTC" } }
```

```json
{
  "channel": "active_asset_ctx",
  "data": {
    "coin": "BTC",
    "mark_px": "66735.25",
    "oracle_px": "66700",
    "funding": {
      "rate_per_hr": "0",
      "cap_per_hr": "400",
      "interval_ms": 3600000,
      "next_payment_ts": 0
    },
    "open_interest": "5000000000"
  }
}
```

- `mark_px` / `oracle_px` — بوحدة USDC كاملة، مُقرَّبة بمقياس السعر (`"0"` إذا لم تُضبط). نفس المستوى المستخدم في `market_info`، لا مستوى الدفتر 1e8.
- `funding` — `{rate_per_hr, cap_per_hr, interval_ms, next_payment_ts}`، مطابق لكتلة `market_info.funding` في REST (يساوي `null` للسوق غير المعروفة — انظر أدناه). `rate_per_hr` هو آخر عينة لمعدل التمويل الساعي (قبل تطبيق الحد الأقصى)، و`cap_per_hr` هو الحد الأقصى لمعدل التمويل لكل سوق، وكلاهما **سلاسل نقاط أساس** مقطوعة باتجاه الصفر (مثال: `"400"` = 0.04/ساعة)؛ `interval_ms` هو دورة التمويل (`3600000` = ساعة واحدة)؛ `next_payment_ts` هو طابع زمني بالمللي ثانية، القيمة `0` حتى تحصل السوق على أول عينة تمويل.
- `open_interest` — الفائدة المفتوحة الحالية، سلسلة نقطة ثابتة (`"0"` إذا لم يكن هناك دفتر أوامر).

التكرار: مدفوع بالتغيير — يُرسَل الإطار فقط عند تغيُّر السياق فعلياً منذ آخر إيداع؛ السياق غير المتغير لا يُصدر شيئاً.

إذا لم تتوافق العملة مع أي سوق معروف، ستستقبل رسالة التأكيد مع لقطة **فارغة صادقة** — أسعار وفائدة مفتوحة معدومة وكتلة `funding` تساوي `null` — دون أي إطارات لاحقة (حتى لا ينكسر المستخدم الذي يُفسّر بنية ctx ثابتة):

```json
{ "channel": "active_asset_ctx", "data": { "coin": "DOGE", "mark_px": "0", "oracle_px": "0", "funding": null, "open_interest": "0" } }
```

### `all_mids`

خريطة Mid العالمية — سعر Mark لكل سوق، يُدفع عند تغيُّر الأسعار. مفهرَسة بالعملة؛ القيم هي سعر Mark بوحدة USDC كاملة مُقرَّبة بمقياس السعر الذي تُبلِّغ عنه قراءة REST [`markets`](../rest/info.md#markets). لا تحتاج إلى معامل `coin`.

```json
{ "method": "subscribe", "subscription": { "type": "all_mids" } }
```

```json
{ "channel": "all_mids", "data": { "mids": { "BTC": "66703.35", "ETH": "1856.49", "SOL": "73.95", "MTF": "5" } } }
```

### `fills` <a id="fills"></a>

دفق تنفيذات لكل حساب. يتطلب `user` (عنوان 0x؛ يُقبل `address` أيضاً) — وليس `coin`. كل تطابق مُنفَّذ يُسلّم سجلاً لكلا الطرفين، كلٌّ من منظوره الخاص، بنفس مجموعة الحقول `{coin, side, px, sz, time, oid, cloid, tid, crossed}`:

- سجل **الآخذ** — `oid` الآخذ الخاص به، و`cloid` الخاص به (أو `null`)، والجانب الخاص بالآخذ، و`crossed: true`؛
- سجل **الصانع** — `oid` الصانع الخاص به، و`cloid: null` (لا يُسجَّل cloid للجانب الراسي)، والجانب **المعاكس**، و`crossed: false`.

يتشارك كلا طرفي التطابق الواحد في نفس `tid` (القيمة ذاتها التي يحملها شريط `trades` العام). `px`/`sz` هما سلاسل مستوى 1e8. لا تحمل سجلات التنفيذ لكل حساب **مصفوفة `users`** — عناوين الأطراف المقابلة لا تظهر إلا في شريط [`trades`](#trades) العام، وليس في التغذية المرتبطة بالحساب.

```json
{ "method": "subscribe", "subscription": { "type": "fills", "user": "0x<address>" } }
```

اللقطة الابتدائية هي مصفوفة فارغة `[]`؛ كل إطار دفع هو مصفوفة تحتوي على سجل تنفيذ واحد:

```json
{ "channel": "fills", "data": [ { "coin": "BTC", "side": "B", "px": "6700000000000", "sz": "10000000", "time": 1735689600123, "oid": 42, "cloid": "0xab..", "tid": 1234567890, "crossed": true } ] }
```

### `user_events` <a id="userevents"></a>

تغذية أحداث لكل حساب. يتطلب `user` (عنوان 0x) — وليس `coin`. حالياً تُصنّف `fills`؛ وستُضاف أنواع أحداث التصفية والتمويل لاحقاً كمفاتيح مشابهة.

```json
{ "channel": "user_events", "data": { "fills": [ { "coin": "BTC", "side": "B", "px": "6700000000000", "sz": "10000000", "time": 1735689600123, "oid": 42, "cloid": "0xab..", "tid": 1234567890, "crossed": true } ] } }
```

اسم القناة الأصلي هو `user_events` (بصيغة snake_case)؛ أما على بوابة `/hl/ws` (وضع التوافق مع HL)، فالاسم المعادل هو `userEvents` الخاص بـ HL.

:::warning
بيانات `user_events` خاصة بكل حساب، لكنها حالياً **بلا مصادقة** — يمكن لأي اتصال الاشتراك في تغذية أي عنوان. لا تعتبرها قناة خاصة حتى يُطبَّق نظام المصادقة عند الاشتراك؛ وللقراءة/الكتابة الموثَّقة، استخدم `post` مع إجراء موقَّع.
:::

### `candles`

أشرطة OHLCV المتجددة لسوق واحدة بحجم شريط واحد. **يتطلب كلاً من `coin` و`interval`** — يُشكّلان معاً مفتاح التوجيه، لذا فاشتراكا `1m` و`5m` على نفس السوق هما اشتراكان مستقلان، لكلٍّ منهما لقطته الابتدائية وإطاراته الخاصة.

```json
{ "method": "subscribe", "subscription": { "type": "candles", "coin": "BTC", "interval": "1m" } }
```

- `interval` ∈ `1m` / `5m` / `15m` / `1h` / `4h` / `1d`. إذا غاب `interval` أو كان غير معروف، يُعيَّن افتراضياً بـ **`1m`** (تُعكس في رسالة التأكيد الفترة الزمنية المستخدمة فعلياً).
- تُعيد رسالة التأكيد قيمة `interval` في الاشتراك حتى يتمكن المستخدم من ربط `(coin, interval)`.

**اللقطة الابتدائية** هي **مصفوفة** من الأشرطة الأخيرة (المغلقة + الشريط المفتوح)، من الأقدم إلى الأحدث — `[]` حتى تبدأ السوق بالتداول:

```json
{ "channel": "candles", "data": [
  { "t": 1735689600000, "T": 1735689659999, "s": "BTC", "i": "1m", "o": "67000.00", "c": "67002.50", "h": "67005.00", "l": "66990.00", "v": "12.5", "q": "837843.75", "n": 8 }
] }
```

كل **إطار دفع** هو **كائن شريط واحد** (وليس المصفوفة) — الشريط المفتوح الحالي لذلك `(coin, interval)`، يُعاد إصداره في كل كتلة مُثبَّتة تقع فيها تنفيذات لهذه السوق:

```json
{ "channel": "candles", "data": { "t": 1735689600000, "T": 1735689659999, "s": "BTC", "i": "1m", "o": "67000.00", "c": "67002.50", "h": "67005.00", "l": "66990.00", "v": "12.5", "q": "837843.75", "n": 8 } }
```

- `t` / `T` — طابع زمني لفتح/إغلاق الشريط بالمللي ثانية (مُشتَق من الإجماع)؛ يغطي الشريط النطاق `[t, T]` وينتقل التنفيذ إلى شريط جديد عندما يتخطى طابعه الزمني قيمة `T`.
- `s` — رمز العملة/السوق؛ `i` — رمز دلو الفترة الزمنية.
- `o` / `c` / `h` / `l` — الفتح / الإغلاق / الأعلى / الأدنى، سلاسل **USDC عشرية** (بالدولار للإنسان، مثال: `"67002.50"`).
- `v` — حجم الأصل الأساسي المُدمَج في الشريط (حجم العملة). `q` — حجم القيمة (بالدولار) = `Σ السعر × الحجم` على تنفيذات الشريط. `n` — عدد التنفيذات في الشريط.

السلسلة **بلا فجوات**: الفترة الزمنية التي لا تشهد تداولات تُصدر شريطاً مسطحاً يحمل الإغلاق السابق (`o = h = l = c = الإغلاق السابق`، `v = q = 0`، `n = 0`). لا يُصدَر أي شريط قبل أول تداول في السوق — تبدأ السلسلة من دلو أول تنفيذ.

يخزّن النظام ما يصل إلى **1000 شريط لكل سلسلة `(coin, interval)`**؛ تُزال السلاسل الباردة (التي لا يشترك فيها أحد)، لذا لا تكلف السوق/الفترة غير المتابَعة شيئاً. على بوابة `/hl/ws` (وضع التوافق مع HL)، الاسم المعادل للقناة هو `candle` (بالمفرد) الخاص بـ HL.

### `order_updates`

دورة حياة الأوامر لكل حساب. يتطلب `user` (عنوان 0x). كل إطار دفع هو مصفوفة من سجلات تحديث الأوامر لذلك الحساب من الكتلة المُثبَّتة للتو؛ اللقطة الابتدائية هي `[]`.

```json
{ "method": "subscribe", "subscription": { "type": "order_updates", "user": "0x<address>" } }
```

```json
{ "channel": "order_updates", "data": [ {
  "order": { "coin": "BTC", "side": "B", "limit_px": "100", "sz": "600", "orig_sz": "1000",
             "oid": 42, "cloid": "0x..", "tif": "GTC", "reduce_only": false },
  "status": "open", "filled_sz": null, "avg_px": null, "reason": null, "time": 1735689600123 } ] }
```

- `status` ∈ `open` (راسٍ؛ `sz` هو المتبقي في الدفتر بعد الإيداع) / `filled` (الآخذ يحمل `filled_sz` التراكمي + `avg_px`؛ وطرف الصانع يُبلِّغ عن `filled_sz` لكل تطابق مع بقاء `status` على `open` طالما يوجد حجم راسٍ) / `canceled` / `rejected` (+`reason`، `oid` يساوي null) / `cancel_rejected` (+`reason`).
- `limit_px` / `sz` / `orig_sz` / `avg_px` هي سلاسل عشرية بمستوى 1e8؛ `time` بالمللي ثانية حسب الإجماع؛ الحقول غير المعروفة تساوي `null`.
- **غير** مُصدَرة حالياً: `modify` / `batchModify` / `scheduleCancel` / `cancelAllOrders` / انتقالات TWAP وإلغاءات المحرك (BOLE T0) — فمراقبة الإيفاد لهذه العمليات تُعيد ok/err مبهماً بلا حمولة لكل أمر.

### `notifications`

إشعارات الهامش والتصفية لكل حساب، مُستخرَجة بمقارنة الحالات المُثبَّتة المتتالية. يتطلب `user`. إطار مصفوفة واحد لكل إيداع متأثر؛ اللقطة الابتدائية `[]`.

```json
{ "method": "subscribe", "subscription": { "type": "notifications", "user": "0x<address>" } }
```

```json
{ "channel": "notifications", "data": [
  { "kind": "yellow_card", "tier": "yellow_card", "message": "...", "time": 1735689600123 },
  { "kind": "forced_close_tier", "tier": "partial_market_50", "message": "...", "time": 1735689600123 },
  { "kind": "tier_cleared", "tier": null, "message": "...", "time": 1735689600123 },
  { "kind": "forced_close", "coin": "BTC", "side": "long", "closed_sz": "600", "message": "...", "time": 1735689600123 },
  { "kind": "backstop_residual", "coin": "BTC", "side": "long", "lots": "120", "message": "...", "time": 1735689600123 },
  { "kind": "backstop_residual_cleared", "coin": "BTC", "side": "long", "message": "...", "time": 1735689600123 } ] }
```

- `kind` هو الوسم الآلي؛ `message` هو النص المقروء للبشر. `tier` ∈ `yellow_card` / `partial_market_50` / `full_market` / `backstop_takeover` (أو `null` عند المسح).
- `yellow_card` هو فترة سماح مدتها كتلة واحدة لتحذير الهامش (عقد T0 في [التصفية المتدرجة](../../concepts/tiered-liquidation.md))؛ يُطلَق `forced_close` عندما تُنفَّذ التصفية فعلياً ضد الحساب.

### `ledger_updates`

حركة الأموال لكل حساب، منسوبةً إلى **سببها** (مقروءة من حمولة الكتلة المُثبَّتة — يظهر السجل فقط عندما يُطبَّق الإجراء). يتطلب `user`؛ اللقطة الابتدائية `[]`.

```json
{ "method": "subscribe", "subscription": { "type": "ledger_updates", "user": "0x<address>" } }
```

```json
{ "channel": "ledger_updates", "data": [ { "kind": "usd_send", "destination": "0x..", "amount": "25.5", "time": 1735689600123 } ] }
```

- `kind` ∈ `usd_send` / `usd_receive`، `spot_send` / `spot_receive` (+`token`)، `asset_send` / `asset_receive` (+`asset`، `to_perp`)، `withdraw` (`via`: `cctp` | `metabridge`)، `deposit` (`amount` قد يكون `null` لرصيد CCTP وارد)، `system_credit`، `sub_account_transfer`، `sub_account_spot_transfer`، `vault_transfer`. يُصدر التحويل سجلاً واحداً لكل طرف (المرسل + المستلم).
- المبالغ هي سلاسل عشرية بوحدة الرمز كاملة، باستثناء `withdraw` عبر MetaBridge الذي يحمل `amount_units` (وحدات أساسية خام). مبالغ رصيد الجسر الوارد وإجراءات CoreWriter المؤجلة (التي تُيفَّد في كتلة لاحقة) لم تُنسَب بعد.

### `active_asset_data`

سياق التداول لكل (مستخدم، عملة) — الرافعة المالية ووضع الهامش والحد الأقصى لحجم الصفقة لحساب واحد في سوق واحدة. يتطلب **كلاً من** `user` (0x) و`coin`. اللقطة الابتدائية هي السياق الحالي المباشر (الإعدادات الافتراضية المعدومة إذا لم يكن للحساب مركز)، وليست مصفوفة فارغة؛ ويُعاد إصدارها فقط عند تغيُّر ذلك السياق.

```json
{ "method": "subscribe", "subscription": { "type": "active_asset_data", "user": "0x<address>", "coin": "BTC" } }
```

```json
{ "channel": "active_asset_data", "data": {
  "address": "0x<addr>", "asset_id": 0, "leverage": 7, "margin_mode": "isolated",
  "max_trade_size": "5000000000", "has_position": true } }
```

- `margin_mode` ∈ `cross` / `isolated` / `strict_iso`؛ `max_trade_size` هو سقف الحجم المُشتَق من حد الفائدة المفتوحة (سلسلة لوت خام)؛ الحقول مطابقة لقراءة REST [`active_asset_data`](../rest/info.md).

على بوابة `/hl/ws` (وضع التوافق مع HL)، الاسم المعادل للقناة هو `activeAssetData` الخاص بـ HL، ويُترجَم الإطار إلى صيغة camelCase الخاصة بـ HL:

```json
{ "channel": "activeAssetData", "data": {
  "user": "0x<address>", "coin": "BTC", "leverage": 7,
  "maxTradeSzs": ["5.0", "5.0"], "availableToTrade": ["35000.00", "35000.00"] } }
```

- `user` — عنوان الحساب 0x؛ `coin` — رمز السوق.
- `maxTradeSzs` — `[buy, sell]`: الحجم الأقصى القابل للتداول على كل جانب (وحدات أساسية)، كسلاسل عشرية.
- `availableToTrade` — `[buy, sell]`: القيمة الاسمية **بالدولار** المتاحة للتداول على كل جانب، كسلاسل عشرية.

### `account_state`

حالة غرفة المقاصة للعقود الدائمة **PERP** لكل حساب — ملخص الهامش والمراكز المفتوحة والأرصدة لحساب واحد — يُدفع عند تغيُّرها. يتطلب `user` (عنوان 0x؛ يُقبل `address` أيضاً) — وليس `coin`. المحتوى مبنيٌّ على منشئ السجل ذاته المستخدم في قراءة REST المركّزة على الحساب، لذلا يختلف إطار WS أبداً عن تلك القراءة. اللقطة الابتدائية هي الحالة المباشرة الحالية (معدومة لحساب بلا أموال)، وليست مصفوفة فارغة.

```json
{ "method": "subscribe", "subscription": { "type": "account_state", "user": "0x<address>" } }
```

```json
{
  "channel": "account_state",
  "data": {
    "address": "0x<addr>",
    "account_value": "10000", "free_collateral": "8500", "maint_margin": "300",
    "init_margin": "1500", "health": "0.97", "tier": 0,
    "mode": "cross", "pm_enabled": false,
    "positions": [
      { "asset": 0, "size": "600", "entry": "62000", "upnl": "441",
        "isolated": false, "lev": 7, "side": "long" }
    ],
    "balances": { "usdc": "10000", "spot": { "MTF": { "total": "12.5", "hold": "0" } } }
  }
}
```

- مقاييس الهامش (`account_value` / `free_collateral` / `maint_margin` / `init_margin` / `health`) هي سلاسل عشرية بوحدة **USDC كاملة**، مطابقة لـ `MarginScalars` في قراءة الحساب REST؛ `tier` هو مؤشر مرتبة التصفية، `mode` هو الوضع الافتراضي للحساب، `pm_enabled` يشير إلى ما إذا كان هامش المحفظة مفعَّلاً.
- `positions[]` — إدخال واحد لكل مركز عقود دائمة مفتوح: `asset` (المعرّف الرقمي)، `size` (سلسلة مستوى 1e8 موقَّعة)، `entry` / `upnl` (بوحدة USDC كاملة)، `isolated`، `lev`، و`side` (`long` / `short`، يظهر في وضع التحوط).
- `balances` — `{usdc, spot}`: `usdc` هو الضمان بالدولار (USDC كاملة)؛ `spot` يربط الرمز بـ `{total, hold}`.

التكرار: مدفوع بالتغيير — يُرسَل الإطار فقط عند تغيُّر حالة الحساب فعلياً منذ آخر إيداع؛ الحساب غير المتغير لا يُصدر شيئاً في هذا الإيداع.

:::warning
بيانات `account_state` خاصة بكل حساب، لكنها حالياً **بلا مصادقة** — يمكن لأي اتصال الاشتراك في أي عنوان. لا تعتبرها خاصة حتى يُطبَّق نظام المصادقة عند الاشتراك.
:::

### `spot_state`

حالة غرفة المقاصة الفورية **SPOT** لكل حساب — أرصدة الرموز الفورية لكل حساب — تُدفع عند تغيُّرها. يتطلب `user`. اللقطة الابتدائية هي مجموعة الأرصدة المباشرة الحالية (`[]` لحساب بلا حيازات فورية).

```json
{ "method": "subscribe", "subscription": { "type": "spot_state", "user": "0x<address>" } }
```

```json
{
  "channel": "spot_state",
  "data": {
    "address": "0x<addr>",
    "balances": [
      { "asset": 1, "name": "USDC", "total": "2500", "hold": "100" },
      { "asset": 2, "name": "MTF", "total": "12.5", "hold": "0" }
    ]
  }
}
```

- `balances[]` — إدخال واحد لكل رمز فوري محتفَظ به: `asset` (المعرّف الرقمي)، `name` (رمز الرمز)، `total` (سلسلة عشرية بوحدة الرمز كاملة)، `hold` (المبلغ المحجوز بواسطة أوامر السوق الفورية الراسية). مطابق لقراءة أرصدة السوق الفورية في REST.

التكرار: مدفوع بالتغيير — يُرسَل الإطار فقط عند تغيُّر الأرصدة الفورية فعلياً منذ آخر إيداع؛ الحساب غير المتغير لا يُصدر شيئاً في هذا الإيداع.

### `web_data2`

لقطة **مركّبة** "لكل شيء يحتاجه الواجهة الأمامية" لكل حساب — ملخص غرفة مقاصة العقود الدائمة وأرصدة السوق الفوري والأوامر المفتوحة وأحقاق الخزينة وحالة البورصة العالمية لحساب واحد، جميعها في إطار واحد، يُدفع عند تغيُّره. يتطلب `user` (عنوان 0x؛ يُقبل `address` أيضاً) — وليس `coin`. المحتوى مطابق بالبايت للقراءة المركّبة التي تُعيدها [`web_data2`](../rest/info.md#web_data2) في REST (إذ تعتمد على نفس القراءات الفرعية)، لذلا يختلف إطار WS أبداً عن تلك القراءة. اللقطة الابتدائية هي اللقطة المركّبة المباشرة (الإعدادات الافتراضية المعدومة عندما لا يملك الحساب أموالاً/مراكز/أوامر)، وليست مصفوفة فارغة.

```json
{ "method": "subscribe", "subscription": { "type": "web_data2", "user": "0x<address>" } }
```

```json
{
  "channel": "web_data2",
  "data": {
    "address": "0x<addr>",
    "clearinghouse": {
      "account_value": "10000",
      "margin_used": "300",
      "positions": [
        { "asset": 0, "size": "600", "entry_ntl": "2500", "mode": "cross", "lev": 10 }
      ]
    },
    "spot_balances": [
      { "asset": 2, "name": "MTF", "total": "12.5", "hold": "0" }
    ],
    "open_orders": [
      { "oid": 42, "market_id": 0, "side": "bid", "px": "6700000000000", "size": "10000000",
        "tif": "gtc", "cloid": "0xab..", "trigger": null, "inserted_at_ms": 1735689600123 }
    ],
    "vault_equities": [
      { "vault_id": 1, "vault_address": "0x<vault>", "shares": "1000", "equity": "1050" }
    ],
    "exchange_status": {
      "spot_disabled": false,
      "post_only_until_time_ms": 0,
      "post_only_until_height": 0,
      "scheduled_freeze_height": null,
      "mip3_enabled": true,
      "frozen": false,
      "replay_complete": true
    }
  }
}
```

تحمل اللقطة المركّبة هذه الأقسام بالضبط (يُركَّب كلٌّ منها من القراءة الفرعية المقابلة، فلا تختلف الأشكال أبداً عن القراءات المستقلة):

- `address` — عنوان 0x الأساسي بالأحرف الصغيرة الذي يُفهرَس عليه الإطار.
- `clearinghouse` — ملخص حساب العقود الدائمة: `account_value` (قيمة الحساب الكلي بوحدة USDC كاملة كسلسلة عشرية)، `margin_used` (Σ هامش الصيانة المستخدم لكل أصل، بوحدة USDC كاملة)، و`positions[]`. كل صف مركز هو `{asset, size, entry_ntl, mode, lev}`: `asset` هو معرّف السوق الرقمي، `size` هو سلسلة الحجم الموقَّعة بمستوى 1e8 (صف لكل طرف غير صفري، لذا يُبلِّغ الحساب في وضع التحوط عن كلا الطرفين)، `entry_ntl` بوحدة USDC كاملة، `mode` ∈ `cross` / `isolated` / `strict_iso`، `lev` هو الرافعة القصوى للمركز. تُحذف الأطراف ذات الحجم الصفري.
- `spot_balances` — مصفوفة `balances` من [`spot_state`](#spot_state) / `spot_clearinghouse_state` في REST: إدخال واحد لكل رمز فوري محتفَظ به، `{asset, name, total, hold}`.
- `open_orders` — مصفوفة `orders` من `frontend_open_orders` في REST: إدخال واحد لكل أمر راسٍ **و**لكل أمر جني أرباح/وقف خسارة موقوف أو مشغِّل توقف، `{oid, market_id, side, px, size, tif, cloid, trigger, inserted_at_ms}`. `side` ∈ `bid` / `ask`؛ `px` / `size` هما سلاسل عشرية بمستوى 1e8؛ `tif` ∈ `alo` / `ioc` / `gtc` (أو `trigger` لوقف خارج الدفتر موقوف)؛ `cloid` هو معرّف أمر العميل أو `null`؛ `trigger` يساوي `null` لأمر الدفتر العادي، وإلا فهو `{trigger_px, trigger_above}` (تحمل الأوامر الموقوفة أيضاً `is_parked: true`).
- `vault_equities` — مصفوفة `equities` من `user_vault_equities` في REST: إدخال واحد لكل خزينة يملك فيها الحساب أسهماً، `{vault_id, vault_address, shares, equity}` (`equity` بوحدة USDC كاملة، `shares` سلسلة صحيحة خام). فارغة عندما لا يتابع الحساب أي خزينة.
- `exchange_status` — مقاييس حالة التداول العالمية (نفس مضمون `exchange_status` في REST): `{spot_disabled, post_only_until_time_ms, post_only_until_height, scheduled_freeze_height, mip3_enabled, frozen, replay_complete}`. هذه الكتلة مطابقة لجميع المشتركين في إيداع معين.

التكرار: مدفوع بالتغيير — يُرسَل الإطار فقط عند تغيُّر اللقطة المركّبة فعلياً منذ آخر إيداع؛ الإيداع الذي لا يمسّ اللقطة المركّبة لهذا الحساب لا يُصدر شيئاً.

:::warning
بيانات `web_data2` خاصة بكل حساب، لكنها حالياً **بلا مصادقة** — يمكن لأي اتصال الاشتراك في أي عنوان. لا تعتبرها خاصة حتى يُطبَّق نظام المصادقة عند الاشتراك؛ وللقراءات الموثَّقة استخدم `post` مع إجراء موقَّع.
:::

## `post` — طلب/استجابة عبر WS

ليست قناة اشتراك، بل الطريقة لإجراء قراءات آنية وكتابات موقَّعة عبر نفس المقبس. `request` هو غلاف `{type, payload}` ذاته المستخدم في مسارات REST؛ يُرسَل عبر المعالجات المطابقة (`POST /info`، `POST /exchange`). راجع [`post` في WS README](./index.md#post-requestresponse-over-ws) للاطلاع على أشكال الطلب/الاستجابة الكاملة وقواعد التوقيع.

```json
{ "method": "post", "id": 1, "request": { "type": "info", "payload": { "type": "l2_book", "coin": "BTC" } } }
```

هذا هو المسار المدعوم للقراءات الموثَّقة وإرسال الإجراءات الموقَّعة عبر WS اليوم.

## خارطة الطريق — غير متاح بعد

ظهرت القنوات التالية في مسودات سابقة لكنها **غير منفَّذة** على واجهة WS للعقدة. فهي ليست أسماء قنوات معروفة؛ الاشتراك فيها يُعيد خطأ `unknown channel`. مُدرَجة هنا حتى لا يُضلَّل المطوّرون بواجهات SDK قديمة.

- **بيانات السوق العامة:** `meta` (بيانات تعريف الكون)، `mark` (سعر Mark/Oracle)، `fundingTicks` (تحديثات معدل التمويل).
- **لكل مستخدم (تتطلب مصادقة):** `vaultEvents`، `rfqEvents`.

أيضاً غير منفَّذ حالياً:

- **`l2_book` القائم على الفوارق** (إطارات `updates` جزئية) — يُرسِل `l2_book` الحالي دائماً أجساماً كاملة للأفضل 20 مستوى. يحمل الإطار علامة `is_snapshot` (`true` في اللقطة الابتدائية، `false` في الإطارات المدفوعة بالتغيير)، غير أن كل جسم هو لقطة كاملة — لا توجد إطارات `updates` للفوارق الجزئية.
- **`seq` / `resume` / رموز الاستئناف** — كل (إعادة) اشتراك يبدأ من لقطة جديدة.
- **غلاف المصادقة عند الاشتراك** للقنوات الخاصة — استخدم `post` مع إجراء موقَّع للعمليات الموثَّقة.

---

## الترتيب والتسليم

- **لكل اشتراك**، تصل الإطارات بترتيب الإيداع (يُصدَر الإطار فقط في الإيداعات التي تغيَّرت فيها حالة القناة المُراقَبة). لا يوجد `seq`؛ الترتيب ضمني في ترتيب الوصول على المقبس الواحد.
- **عبر الاشتراكات**، لا يوجد ضمان للترتيب — التداخل عشوائي. افصل الإطارات بناءً على `channel` + `coin` داخل `data`.
- التسليم **مرة واحدة كحد أقصى لكل تغيير** و**غير مُخزَّن للاستئناف**: يُسقَط الاشتراك الذي يتأخر بأكثر من 256 إطاراً مع إطار خطأ `lagged` (راجع [الضغط الخلفي والتأخر](./index.md#backpressure--lag)). أعِد الاشتراك للاسترداد؛ ستحصل على لقطة جديدة.
